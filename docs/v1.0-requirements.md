# etmem-operator v1.0 需求说明

## 1. 引言

本文件定义 etmem-operator v1.0 的需求目标、范围边界、交付要求与验收口径。v1.0 以已验证通过的 MVP 能力为基础，目标不是重写现有方案，而是对已证明可行的控制面、执行面与交付链路进行工程化和产品化加固。

本文件只回答“为什么做、做什么、做到什么算完成”，不展开实现方案、字段细节或组件协作机制。

## 2. 项目背景

现有 MVP 已在真实环境中验证了关键闭环：通过 Pod/workload 标记触发 Operator 自动创建与收敛策略；Agent 以 DaemonSet 方式在节点侧执行目标匹配、PID 解析与任务下发；NodeState 可回写任务与健康状态；在真实目标环境中可观测到目标进程 `VmSwap > 0` 以及 `Swap used` 增长。

与此同时，已验证事实也表明：`MemAvailable` 仅适合作为主机级辅助观测项，不能作为单次验收的硬门槛；不同进程的 `VmSwap` 增长幅度可能不同，sleep / wrapper 类进程出现 `VmSwap = 0` 也属于可接受现象。

因此，v1.0 的需求重心应放在稳定部署、可升级、可恢复、可观测、可离线交付与可复现验证上，而不是推翻现有架构或扩展与当前目标无关的能力面。

## 3. 建设目标

v1.0 建设目标如下：

1. 将已验证的 MVP 能力固化为可交付、可验证、可维护的正式版本。
2. 明确控制面、节点执行面与数据面之间的职责边界，确保研发、测试与交付口径一致。
3. 形成可用于真实环境部署、升级、回滚与故障恢复的标准化交付方式。
4. 建立清晰的验收标准，使“功能可用”和“数据面真实生效”均有明确判定依据。

v1.0 的建设重点是工程化补强，不是架构重构；是把已验证链路做成正式产品，而不是重新定义整套系统。

## 4. 范围定义

### 4.1 范围内

- 基于 Pod/workload 标记驱动的自动策略管理。
- Operator 自动创建、更新、删除与回收策略对象。
- Agent 以节点侧 DaemonSet 方式执行目标匹配、PID 解析与任务执行。
- NodeState 写回任务状态与节点健康状态。
- 真实目标环境下的数据面验证与结果判定。
- Helm 安装/升级、离线镜像导入、工作负载 patching 与脚本化观测链路。
- 以现有 MVP 为基线的恢复与回滚能力。

### 4.2 范围外

- 推翻当前 MVP 架构并重新设计控制面或执行面。
- 引入与当前 v1.0 目标无关的新控制器、新 CRD 体系或新的大规模能力面。
- 将 `MemAvailable` 提升为单次验收硬门槛。
- 面向未来版本的高级参数全面展开与重构。
- 未经真实环境验证的更大范围平台化、多租户极限模型或长期演进路线。

## 5. 角色与使用场景

### 5.1 角色

| 角色 | 职责 |
|---|---|
| 平台运维/集群管理员 | 负责部署、升级、回滚、离线导入与环境准备。 |
| 业务开发/服务负责人 | 通过工作负载标记启用或关闭 etmem 能力，并关注业务进程效果。 |
| 验证/发布负责人 | 依据统一验收标准检查控制面、节点侧与数据面结果。 |

### 5.2 典型使用场景

1. 业务侧为目标 Pod/workload 增加标记后，系统自动建立策略并开始管理。
2. Pod 重建后，系统可重新纳管并继续维持既有能力。
3. 需要暂停或回滚时，系统可按既定路径停止相关管理并清理状态。
4. 运维侧通过 Helm 或离线交付包完成安装、升级与恢复。
5. 验证侧通过 `VmSwap`、`Swap used`、NodeState 与日志联合判断是否真实生效。

## 6. 功能需求

| 编号 | 功能需求 | 说明 | 验收要点 |
|---|---|---|---|
| FR-1 | 模板标记启用与自动策略管理 | 系统应支持通过模板标记启用 etmem 能力，并基于已启用的 Pod/workload 标记自动创建、更新并收敛策略对象。 | 模板标记或工作负载标记变化后，系统可正确启用能力且策略对象状态与预期一致。 |
| FR-2 | EtmemPolicy 自动生成与收敛 | 系统应自动生成并持续收敛 EtmemPolicy，使其与已启用的工作负载状态保持一致。 | EtmemPolicy 可随工作负载启停、变更与重建自动生成、更新或回收。 |
| FR-3 | 生命周期跟随 | 系统应在 Pod 删除、重建或重新调度后继续保持一致的管理结果，不产生孤儿策略。 | Pod 生命周期变化后，策略可自动回收或重建。 |
| FR-4 | 暂停与回滚 | 系统应提供暂停和回滚路径，使已启用能力能够按既定口径停止并恢复。 | 执行暂停/回滚后，相关管理行为按预期停止。 |
| FR-5 | 节点侧执行与 cgroup v1 启动 | Agent 应以节点侧常驻方式在 cgroup v1 前提下完成 PID 解析、任务启动与下发，并面向已验证环境完成实际执行。 | 节点侧可基于 cgroup v1 正确解析目标 PID 并启动任务。 |
| FR-6 | NodeState 健康与任务暴露 | 系统应通过 NodeState 暴露节点健康与任务状态，并支持对关键运行状态进行统一回写与查询。 | NodeState 可反映节点健康与任务状态。 |
| FR-7 | 新 Pod 生效验证 | 新增 Pod 的生效判定应提供一条标准化的效果验证路径，能够结合控制面、数据面与主机侧观测结果完成统一验收。 | 可基于控制面、数据面与主机侧观测完成判定，无需固定顺序。 |

## 7. 非功能需求

| 编号 | 非功能需求 | 说明 | 验收要点 |
|---|---|---|---|
| NFR-1 | 稳定性 | 系统应在安装、升级、重建与恢复场景下保持稳定，不因常规操作丢失管理关系。 | 常规生命周期操作后，系统可继续工作。 |
| NFR-2 | 可观测性 | 系统应提供足够的观测信息用于判断控制面、节点侧与数据面状态。 | 可通过日志、NodeState、`VmSwap`、`Swap used` 进行判定。 |
| NFR-3 | 可交付性 | 系统应支持标准化交付，满足 Helm 安装/升级与离线镜像导入的要求。 | 按交付包可完成安装、升级与恢复。 |
| NFR-4 | 可恢复性 | 系统应在故障、重建、升级失败或回滚场景下保持可恢复，且恢复后管理关系与状态可重新对齐。 | 经恢复操作后，系统可回到可用状态并继续管理目标工作负载。 |
| NFR-5 | 可复现性 | 系统的安装、升级、离线导入与验收过程应具备可复现性，确保在相同前提条件下得到一致结果。 | 在相同环境与前提条件下，可按同一流程重复获得一致结论。 |
| NFR-6 | 环境适配与约束 | 系统的真实数据面验收建立在已验证环境条件之上；`MemAvailable` 仅作为辅助观测。 | 在满足前提条件时可完成验收，且不以单次 `MemAvailable` 作为硬门槛。 |

## 9. 验收标准

| 编号 | 验收标准 | 判定方式 |
|---|---|---|
| AC-1 | 通过工作负载标记启用后，系统可自动创建并收敛策略对象。 | 检查策略对象状态与工作负载状态。 |
| AC-2 | Pod 重建后，系统可重新纳管并维持预期管理结果。 | 检查重建前后策略与管理状态连续性。 |
| AC-3 | 暂停/回滚路径可用，且可按既定口径停止相关管理行为。 | 执行暂停/回滚并核对结果。 |
| AC-4 | 节点侧执行链路可用，Agent 能完成目标匹配、PID 解析与任务执行。 | 检查节点日志、NodeState 与任务结果。 |
| AC-5 | 在真实目标环境中，可观察到目标进程存在 swap 使用迹象，且宿主机 swap 使用量呈增长趋势。 | 通过进程侧、控制面与主机侧观测共同确认数据面生效。 |
| AC-6 | Helm 安装/升级、离线镜像导入与工作负载 patching 链路可用。 | 按交付流程完成部署与升级验证。 |
| AC-7 | `MemAvailable` 仅作为辅助观测项，不作为单次硬门槛。 | 验收记录中不得将其作为唯一失败依据。 |

## 10. 风险与约束

| 编号 | 风险/约束 | 影响 | 处理原则 |
|---|---|---|---|
| R-1 | 数据面验证依赖真实目标环境 | 若缺少 swap、etmemd、cgroup v1 等前提，数据面结果无法成立。 | 仅在满足前提条件的环境中执行最终验收。 |
| R-2 | `MemAvailable` 受宿主机其他活动影响 | 单次观测可能不稳定，不适合作为硬判定。 | 仅作为辅助观测项，与 `VmSwap`、`Swap used`、NodeState 联合判断。 |
| R-3 | 交付链路需要可重复性 | 若安装、升级、离线导入不可重复，v1.0 无法稳定交付。 | 以可复现、可回滚、可验证为基本要求。 |
| R-4 | 范围膨胀风险 | 若引入过多未来能力，会削弱 v1.0 交付聚焦。 | 严格保持“基于已验证 MVP 的工程化加固”定位。 |

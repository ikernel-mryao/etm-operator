# etmem-operator v1.0 详细设计说明书

**文档版本**：v1.0  
**状态**：正式  
**文档定位**：给出 v1.0 的实现机制、关键数据结构、处理流程与验证方法，供实现、联调、排障与验证阶段使用。  
**适用范围**：基于已在真实目标环境（openEuler/FusionOS + etmemd + cgroup v1 + swap）中验证通过的代码路径展开，不涉及未经验证的功能或未来演进方向。

---

## 已验证事实基线

本文档以下列已在真实目标环境中验证通过的事实为基础展开实现机制：

1. Pod/workload 标签驱动 Operator 自动创建并收敛 EtmemPolicy 已验证可用。
2. Pod 重建后，Agent 可重新纳管并继续管理。
3. `suspend` / rollback 路径已验证可用。
4. Agent 以 DaemonSet 方式运行；基于 cgroup v1（同时支持 systemd 和 cgroupfs 两种驱动）的节点内 PID 解析已验证可用。
5. NodeState 写回与健康状态观测（`socketReachable`、`etmemdReady`、`tasks`、`metrics.totalManagedPods`）已验证。
6. 交付链路已通过 Helm install/upgrade、离线镜像导入、workload patching、脚本/NodeState 观测路径完整跑通。
7. 不同进程的 `VmSwap` 增长幅度可以不同，不代表失败。
8. wrapper/sleep 类进程可能出现 `VmSwap=0`，不能仅据此判定失败。
9. `MemAvailable` 属于主机级净值辅助观测，受页缓存、内核回收等多因素影响，不作为单次验收硬门槛。
10. 当前版本基于 cgroup v1，不支持 cgroup v2。

## 2. CRD 与对象模型

etmem-operator v1.0 定义两个集群范围的自定义资源（CRD），API 组为 `etmem.openeuler.io/v1alpha1`。

### 2.1 EtmemPolicy

**定位**：由 Operator 自动生成和持续维护的策略对象，承载内存分层策略配置（目标 Pod 选择器、引擎参数、挂起标志等）。用户无需手动创建；Operator 在每个命名空间内最多维护一个名为 `etmem-auto` 的自动生成策略。

**资源作用域**：Namespaced

**关键字段说明**：

| 字段路径 | 类型 | 说明 |
|----------|------|------|
| `spec.selector` | LabelSelector | 通过标签选择目标 Pod（与 `workloadRefs` 互斥）|
| `spec.workloadRefs` | []WorkloadRef | 直接引用工作负载（MVP 中跳过，不处理）|
| `spec.nodeSelector` | map[string]string | 限制策略生效的节点范围 |
| `spec.engine.type` | string | 引擎类型，固定为 `slide` |
| `spec.engine.profile` | string | 预设档位：`conservative`/`moderate`/`aggressive`/`extreme` |
| `spec.engine.overrides` | SlideOverrides | 覆盖预设参数的个别字段（可选）|
| `spec.processFilter.names` | []string | 进程名白名单；为空时管理 Pod 内所有进程 |
| `spec.suspend` | bool | 为 `true` 时，Agent 跳过该 Policy 的所有任务 |
| `spec.circuitBreaker` | CircuitBreakerSpec | 熔断阈值配置（Pod 重启次数、节点 PSI 百分比）|
| `status.summary.activeNodes` | int | 当前有活跃任务的节点数（由 NodeStateAggregator 汇总）|
| `status.summary.managedPods` | int | 当前被管理的 Pod 数（汇总值）|

**自动生成标识**：自动生成的 Policy 带有标签 `etmem.openeuler.io/auto-generated: "true"`。Operator 仅收敛带此标签的策略，用户手工创建的策略不会被覆盖。

**引擎档位参数映射**（来自 `internal/engine/profiles.go`，`swap_flag` 统一为 `no` 以避免 FusionOS 内核 idle_pages ioctl 兼容问题）：

| Profile | loop | interval | sleep | sysmem_threshold | swapcache_high_wmark | swapcache_low_wmark | T | max_threads |
|---------|------|----------|-------|------------------|----------------------|---------------------|---|-------------|
| conservative | 3 | 5 | 3 | 85 | 15 | 10 | 3 | 1 |
| moderate | 1 | 1 | 1 | 90 | 10 | 6 | 1 | 1 |
| aggressive | 1 | 1 | 1 | 95 | 5 | 3 | 1 | 2 |
| extreme | 1 | 1 | 1 | 99 | 2 | 1 | 1 | 2 |

说明：`sysmem_threshold` 表示"系统可用内存低于此百分比时触发 swap"。各档位阈值已针对大内存机器（如可用内存占比 80% 以上）调整，保证在实际环境下能触发换出。

### 2.2 EtmemNodeState

**定位**：由 Agent 创建并持续回写的节点状态对象，反映单个节点上 etmemd 通路健康状态与任务执行情况。每个节点一个对象，名称与节点名一致。

**资源作用域**：Cluster（集群范围，无命名空间）

**关键字段说明**：

| 字段路径 | 类型 | 说明 |
|----------|------|------|
| `status.nodeName` | string | 对应节点名称 |
| `status.socketReachable` | bool | etmem CLI 通路是否可达（探测依据见 §7.1）|
| `status.etmemdReady` | bool | etmemd 是否就绪（与 `socketReachable` 同步更新）|
| `status.conditions` | []Condition | 标准 Kubernetes Condition 列表（含 `Ready` 条件）|
| `status.tasks` | []NodeTask | 当前节点上所有活跃任务的列表 |
| `status.metrics.totalManagedPods` | int | 当前节点被管理的唯一 Pod 数量 |
| `status.metrics.totalSwappedBytes` | string | 当前节点已换出的总字节数（可选，辅助观测）|

**NodeTask 字段说明**：

| 字段路径 | 类型 | 说明 |
|----------|------|------|
| `tasks[].policyRef.name` | string | 对应策略名称 |
| `tasks[].policyRef.namespace` | string | 对应策略命名空间 |
| `tasks[].podName` | string | 对应 Pod 名称 |
| `tasks[].podUID` | string | Pod UID（用于 cgroup 路径构建）|
| `tasks[].projectName` | string | etmemd project 名称（格式见 §6.2）|
| `tasks[].state` | string | 任务状态：`running`/`stopped`/`error`/`circuit-broken` |
| `tasks[].processes` | []ManagedProcess | 被管理的进程列表（含 PID 和进程名）|
| `tasks[].lastReconcileTime` | Time | 最后一次 reconcile 时间 |
| `tasks[].error` | string | 如有错误，记录错误信息 |

**`kubectl get etmemnodestate` 默认列**：`Node`、`Ready`（来自 Conditions）、`Tasks`（来自 `metrics.totalManagedPods`）、`Age`。

## 3. Operator 详细设计

### 3.1 部署形式

- **Deployment**：副本数为 1，运行于 `etmem-system` 命名空间。
- **镜像**：由 `values.yaml` 中 `operator.image` 字段指定，当前基线为 `etmem-operator:v0.4.0-pid`。
- **资源限制**：requests: `cpu=50m, memory=64Mi`；limits: `cpu=200m, memory=256Mi`。
- **ServiceAccount**：`etmem-operator`，绑定 ClusterRole，具备 `pods`（get/list/watch）、`etmempolicies`（所有操作）、`etmemnodestates`（get/list/watch）的权限。

### 3.2 控制器：PodReconciler

`PodReconciler`（`internal/controller/pod_controller.go`）监听带有 `etmem.openeuler.io/enable=true` 标签的 Pod 事件，对每个命名空间维护一个自动生成的 EtmemPolicy。

**Reconcile 触发条件**：

- Pod 创建、更新（标签/注解变更）、删除。
- 通过 `etmemLabelPredicate` 过滤事件：Create/Delete 事件要求 Pod 带有 `etmem.openeuler.io/enable=true` 标签；Update 事件要求标签键 `etmem.openeuler.io/enable` **或** 注解键 `etmem.openeuler.io/profile` 发生变化（任一变化均触发 reconcile）。

**Reconcile 逻辑（按命名空间维度执行）**：

```
1. 列出命名空间内所有标签 etmem.openeuler.io/enable=true 的 Pod
2. 过滤掉 Phase 为 Succeeded 或 Failed 的 Pod
3. 统计有效 Pod 数量（enabledCount）
4. 解析最激进 profile（most-aggressive-wins 算法）：
     - 读取每个 Pod 的 annotation etmem.openeuler.io/profile
     - 按优先级 conservative(1) < moderate(2) < aggressive(3) < extreme(4) 取最大值
     - 若无任何 annotation，默认使用 aggressive
5. 如果 enabledCount > 0：
     - 创建或更新名为 etmem-auto 的 EtmemPolicy，设置 profile 字段
     - 仅更新带有 etmem.openeuler.io/auto-generated=true 标签的策略
     - 用户手工创建的同名策略不被覆盖
6. 如果 enabledCount == 0：
     - 删除 etmem-auto（如果存在且带有 auto-generated 标签）
```

**自动生成策略的 spec 结构**：

```yaml
spec:
  selector:
    matchLabels:
      etmem.openeuler.io/enable: "true"
  engine:
    type: slide
    profile: <resolved_profile>
```

**保护机制**：
- Operator 只处理 `spec.selector` 方式的策略；`spec.workloadRefs` 不为空的策略在 Agent 侧会被跳过（MVP 范围外）。
- Operator 不感知节点级执行细节，不直接与 etmemd 通信。

### 3.3 NodeStateAggregator

`NodeStateAggregator`（`internal/controller/nodestate_aggregator.go`）监听 EtmemNodeState 变化，汇总各节点的 `totalManagedPods` 到 EtmemPolicy 的 `status.summary`，供运维人员通过 `kubectl get etmempolicy` 查看聚合状态。

## 4. Agent 详细设计

### 4.1 部署形式（DaemonSet）

Agent 以 DaemonSet 方式部署到所有目标节点，每节点一个 Pod。核心配置如下（来自 `deploy/helm/templates/agent-daemonset.yaml`）：

| 配置项 | 值 | 说明 |
|--------|-----|------|
| `hostPID: true` | true | 访问宿主机 PID 命名空间，读取 `/proc/<pid>/comm` |
| `hostNetwork: true` | true | 使用宿主机网络命名空间，访问 etmemd 抽象 Unix socket |
| `dnsPolicy` | ClusterFirstWithHostNet | 配合 hostNetwork 使用 |
| `capabilities.add` | SYS_PTRACE | 读取其他进程的内存信息 |
| `runAsUser` | 0 | 以 root 运行，访问 cgroup 和 proc 文件系统 |

**hostPath 挂载列表**：

| 挂载名称 | 宿主机路径 | 容器内路径 | 用途 |
|----------|-----------|-----------|------|
| host-proc | `/proc` | `/host/proc` | 读取进程 comm / status（PID 解析）|
| host-cgroup | `/sys/fs/cgroup` | `/host/sys/fs/cgroup` | 读取 cgroup.procs（PID 收集）|
| etmem-bin | `/usr/bin/etmem` | `/usr/bin/etmem` | etmem CLI 二进制（通路调用）|
| host-lib64 | `/lib64` | `/lib64` | etmem CLI 动态链接库 |
| host-usr-lib64 | `/usr/lib64` | `/usr/lib64` | etmem CLI 动态链接库 |
| etmem-config-dir | `/var/run/etmem/configs` | `/var/run/etmem/configs` | etmem 配置文件（宿主机与容器共享）|
| containerd-sock | `/run/containerd/containerd.sock` | `/run/containerd/containerd.sock` | 容器运行时 socket |

**启动参数**（来自 `values.yaml`）：
- `--reconcile-interval`：默认 `30s`，控制 reconcile 循环间隔。
- `--socket-name`：默认 `etmemd_socket`，etmemd 抽象 Unix socket 名称。
- 环境变量 `NODE_NAME`：通过 Downward API 注入，获取当前节点名。

### 4.2 Agent 五步 Reconcile 流程

Agent 采用固定间隔 ticker 驱动的 reconcile 模式，每个节点独立推导任务，不依赖中心化调度（`cmd/agent/main.go` 中的 `agentReconcile` 函数）。

**步骤一：etmemd 通路探测**

```
tr.ProjectShow(ctx, "")
  → 成功：transportHealthy = true（socketReachable=true, etmemdReady=true）
  → 失败：transportHealthy = false（socketReachable=false, etmemdReady=false）
```

通路状态用于 NodeState 健康字段的更新，即使通路不健康，reconcile 仍会继续（但无法启动新任务）。

**步骤二：节点级熔断检查**

读取 `/proc/pressure/memory` 中的 `avg10` 字段，若超过阈值（默认 70%），停止节点所有任务并退出本轮 reconcile。

**步骤三：构建期望任务集合（Desired State）**

```
for each EtmemPolicy:
  跳过 spec.suspend=true 的策略
  跳过 spec.workloadRefs 不为空的策略（MVP 不支持）
  解析 profile 并应用 overrides

  for each Pod（全量列出后客户端侧过滤）:
    调用 MatchPodToPolicy 检查：
      1. Pod 在运行状态（Phase=Running）
      2. Pod 在目标节点（spec.nodeName == NODE_NAME）
      3. Pod 标签符合 Policy.spec.selector
      4. 节点标签符合 Policy.spec.nodeSelector

    检查 Pod 级熔断（容器重启次数 >= 阈值 或 OOMKilled）

    根据 cgroup 驱动构建路径（见 §5）
    调用 ResolvePIDs 收集目标进程
    过滤基础设施进程（pause, sandbox）
    为每个应用进程生成独立 projectName 和配置
```

**步骤四：Diff（停止不再需要的任务）**

将 `TaskManager.RunningTasks()` 与期望集合比较，停止不在期望集合中的任务。

**步骤五：启动新任务**

为期望集合中尚未运行的任务调用 `TaskManager.StartTask()`。

**最终**：调用 `NodeStateWriter.WriteStatus()` 将当前状态写回 EtmemNodeState。

### 4.3 Pod-Policy 匹配规则

函数 `agent.MatchPodToPolicy`（`internal/agent/pod_watcher.go`）按以下顺序检查条件（任一不满足即返回 false）：

1. Policy 的 `spec.suspend` 为 false（suspend=true 时跳过所有 Pod）。
2. Pod 的 `namespace` 与 Policy 的 `namespace` 相同。
3. Pod 的 `spec.nodeName` 等于当前节点名。
4. 若 Policy 设置了 `spec.nodeSelector`，当前节点标签须包含所有指定键值对。
5. 若 Policy 设置了 `spec.selector`，Pod 标签须满足 LabelSelector 语义。

> **注意**：Pod Phase（Running/Pending 等）的过滤由 Operator 侧 PodReconciler 在 reconcile 中处理（过滤掉 Succeeded/Failed 状态），`MatchPodToPolicy` 本身不检查 Pod Phase。

### 4.4 熔断机制（CircuitBreaker）

熔断保护分为两级，避免 etmem 回收行为导致业务受损：

**Pod 级熔断**（`IsPodTripped`）：
- 检查任意容器的 `RestartCount >= podRestartThreshold`（默认 5）。
- 检查任意容器的 `lastTerminationState.Reason == "OOMKilled"`。
- 条件满足时，跳过该 Pod，不启动 etmem 任务。
- **不自动恢复**：容器重启次数只会单调递增，熔断后 Pod 将在整个生命周期内持续被跳过，需通过 `spec.suspend` 或手动清理 Pod 重置。

**节点级熔断**（`IsNodeTripped`）：
- 读取 `/proc/pressure/memory` 的 `avg10` PSI 值。
- 若 `avg10 >= nodePSIThreshold`（默认 70），停止节点所有任务，当前 reconcile 轮次提前退出。
- **可自动恢复**：`IsNodeTripped` 在每轮 reconcile 中重新读取 PSI，当 avg10 降至阈值以下时，下一轮 reconcile 自动恢复正常流程，无需人工介入。

## 5. PID 解析与 cgroup v1 规则

### 5.1 设计前提

当前版本仅支持 **cgroup v1**。Agent 启动时通过探测 `/host/sys/fs/cgroup/memory/kubepods.slice` 是否存在自动识别 cgroup 驱动类型（`cmd/agent/main.go`）：

- 存在 → systemd 驱动，使用 `.slice` 路径格式。
- 不存在 → cgroupfs 驱动，使用标准层级路径格式。

驱动类型在启动时一次性确定，整个运行期间不变。

### 5.2 systemd cgroup 驱动路径规则

systemd 驱动下，Pod UID 中的 `-` 须替换为 `_`，路径使用 `.slice` 后缀（来自 `BuildCgroupRelPathSystemd`）：

| QoS 等级 | cgroup 路径（相对于 `/host/sys/fs/cgroup`）|
|----------|------------------------------------------|
| Guaranteed | `memory/kubepods.slice/kubepods-pod<uid_下划线>.slice` |
| Burstable | `memory/kubepods.slice/kubepods-burstable.slice/kubepods-burstable-pod<uid_下划线>.slice` |
| BestEffort | `memory/kubepods.slice/kubepods-besteffort.slice/kubepods-besteffort-pod<uid_下划线>.slice` |

示例（Pod UID = `abc-123-def`）：
```
Guaranteed:  memory/kubepods.slice/kubepods-podabc_123_def.slice
Burstable:   memory/kubepods.slice/kubepods-burstable.slice/kubepods-burstable-podabc_123_def.slice
BestEffort:  memory/kubepods.slice/kubepods-besteffort.slice/kubepods-besteffort-podabc_123_def.slice
```

完整宿主机路径在上述相对路径前加 `/sys/fs/cgroup/`（验证脚本中使用绝对路径）：

```
/sys/fs/cgroup/memory/kubepods.slice/kubepods-pod<uid_下划线>.slice
/sys/fs/cgroup/memory/kubepods.slice/kubepods-burstable.slice/kubepods-burstable-pod<uid_下划线>.slice
/sys/fs/cgroup/memory/kubepods.slice/kubepods-besteffort.slice/kubepods-besteffort-pod<uid_下划线>.slice
```

### 5.3 cgroupfs 驱动路径规则

cgroupfs 驱动下，Pod UID 原样使用（无需替换 `-`），路径无 `.slice` 后缀（来自 `BuildCgroupRelPath`）：

| QoS 等级 | cgroup 路径（相对于 `/host/sys/fs/cgroup`）|
|----------|------------------------------------------|
| Guaranteed | `memory/kubepods/pod<uid>` |
| Burstable | `memory/kubepods/burstable/pod<uid>` |
| BestEffort | `memory/kubepods/besteffort/pod<uid>` |

完整宿主机路径：

```
/sys/fs/cgroup/memory/kubepods/pod<uid>
/sys/fs/cgroup/memory/kubepods/burstable/pod<uid>
/sys/fs/cgroup/memory/kubepods/besteffort/pod<uid>
```

### 5.4 PID 收集逻辑（ResolvePIDs）

`PIDResolver.ResolvePIDs`（`internal/agent/pid_resolver.go`）执行以下步骤：

```
1. 拼接 Pod cgroup 目录路径：cgroupRoot + "/" + cgroupRelPath
2. 读取目录自身的 cgroup.procs（Pod 级别，通常为空）
3. 遍历直接子目录（容器 scope，如 cri-containerd-*.scope）
   → 读取每个子目录的 cgroup.procs
   → 合并所有 PID（cgroup v1 中 PID 位于容器子 cgroup）
4. 对每个 PID，读取 /host/proc/<pid>/comm 获取进程名
5. 若 processFilter.names 不为空，按名称过滤（精确匹配 comm 字段）
6. 读取 /host/proc/<pid>/status 中的 VmRSS（可选，供 metrics 使用）
7. 返回 ResolvedProcess 列表（含 PID、Name、RSSKB）
```

**进程过滤规则**：
- `ProcessFilter.names` 为空：返回所有进程（全进程模式）。
- `ProcessFilter.names` 非空：仅返回 comm 匹配白名单的进程。
- 基础设施进程（`pause`、`sandbox`）在 `agentReconcile` 中额外过滤，不纳入 etmem 管理。

### 5.5 进程名约束

- 进程名（comm）最长 15 字符（来自 `/proc/<pid>/comm` 的内核限制）。
- etmem project 名称最长 64 字符，格式为 `<namespace>-<podName>-<processName>-p<PID>`（由 `agent.ProjectNameForProcess` 生成）。超过 64 字符时，取前 55 字符加连字符与完整名称的 SHA256 哈希前 8 位，确保唯一性。

## 6. etmemd 交互与任务生命周期

### 6.1 传输层设计（ExecTransport）

Agent 通过 `ExecTransport`（`internal/transport/exec_transport.go`）调用 `/usr/bin/etmem` CLI 与 etmemd 通信。etmem CLI 通过 **抽象 Unix socket**（非文件系统路径）与 etmemd 通信，因此 Agent Pod 必须启用 `hostNetwork: true`。

CLI 调用格式：

```bash
/usr/bin/etmem <subcommand> [args...] -s <socketName>
```

实现的 Transport 接口方法：

| 方法 | 对应 CLI 命令 | 用途 |
|------|------------|------|
| `ObjAdd(configPath)` | `etmem obj add -f <path> -s <socket>` | 向 etmemd 注册配置文件 |
| `ObjDel(configPath)` | `etmem obj del -f <path> -s <socket>` | 注销配置文件 |
| `ProjectStart(name)` | `etmem project start -n <name> -s <socket>` | 启动 project 的扫描任务 |
| `ProjectStop(name)` | `etmem project stop -n <name> -s <socket>` | 停止 project 的扫描任务 |
| `ProjectShow(name)` | `etmem project show [-n <name>] -s <socket>` | 查询 project 状态（也用于通路探测）|

### 6.2 配置文件格式

每个应用进程对应一个独立的 etmem 配置文件（INI 风格），由 `SlideEngine.GenerateConfig` 生成，保存至 `/var/run/etmem/configs/<projectName>.conf`（权限 0600）：

```ini
[project]
name=<projectName>
scan_type=page
loop=<loop>
interval=<interval>
sleep=<sleep>
sysmem_threshold=<sysMemThreshold>
swapcache_high_wmark=<highWmark>
swapcache_low_wmark=<lowWmark>

[engine]
name=slide
project=<projectName>

[task]
project=<projectName>
engine=slide
name=<projectName>_task_0
type=pid
value=<PID>
T=<T>
max_threads=<maxThreads>
swap_flag=<swapFlag>
```

**重要限制**：etmemd 每个配置文件仅支持一个 `[task]` 段，且同名 project 的第二次 `obj add` 会返回"project has been existed"错误。因此，每个应用进程必须对应一个独立的 project 名称和独立的配置文件。

**配置目录**：`/var/run/etmem/configs`，通过 hostPath 在 Agent 容器与宿主机之间共享，etmemd 从宿主机侧读取此目录中的配置文件。

### 6.3 任务启动五步协议（TaskManager.StartTask）

```
1. 创建配置目录（MkdirAll /var/run/etmem/configs）
2. 写入配置文件（os.WriteFile，权限 0600）
3. etmem obj add -f <configPath> -s <socketName>
     → 失败：删除配置文件，返回错误
4. etmem project start -n <projectName> -s <socketName>
     → 失败：执行 obj del 回滚，删除配置文件，返回错误
5. 将 projectName → configPath 记录到运行中任务 map
```

### 6.4 任务停止协议（TaskManager.StopTask）

```
1. 从运行中任务 map 中删除 projectName（幂等，防止并发重入）
2. etmem project stop -n <projectName> -s <socketName>（容忍失败：etmemd 可能已停止）
3. etmem obj del -f <configPath> -s <socketName>（容忍失败）
4. 删除配置文件（os.Remove）
```

StopTask 设计为幂等操作，即使 etmemd 不响应也能正常清理本地状态，防止资源泄漏。

### 6.5 swapcache 工作链路

```
etmem slide 扫描冷页
  → 标记换出 → 进入 swapcache（驻留内存中）
  → swapcache 用量 > swapcache_high_wmark% 时触发刷盘
  → 写入 swap 设备
  → swapcache 用量降至 < swapcache_low_wmark%
  → 物理页框释放 → 进程 VmSwap 增长、VmRSS 减少
```

## 7. NodeState / 状态回写设计

### 7.1 健康字段探测机制

`etmemdReady` 和 `socketReachable` 两个字段通过调用 `tr.ProjectShow(ctx, "")` 动态探测（`cmd/agent/main.go` 中 `agentReconcile` 的第一步）：

- 若调用成功（exit code 0），则 `transportHealthy = true`，两字段均设为 `true`。
- 若调用失败（CLI 执行错误或 socket 不可达），则 `transportHealthy = false`，两字段均设为 `false`。

两字段同步更新（来自 `buildNodeStatus` 函数），没有独立的探测路径，反映的是 etmem CLI → etmemd socket 通路的整体可用性。

### 7.2 NodeStateWriter：create-then-update 写入模式

`NodeStateWriter.WriteStatus`（`internal/agent/nodestate_writer.go`）实现 create-then-update 语义，符合 Kubernetes status 子资源要求：

```
1. 尝试 GET EtmemNodeState（名称 = 节点名）
2. 若 NotFound：
     a. 创建空对象（client.Create）
     b. 通过 client.Status().Update() 写入 status
3. 若已存在：
     a. 直接通过 client.Status().Update() 更新 status
```

status 字段不能通过 `client.Update()` 更新，必须使用 `client.Status().Update()`（Kubernetes status 子资源机制）。

### 7.3 Bootstrap 恢复机制

Agent 重启时（如 DaemonSet 滚动更新或节点重启后 Pod 重调度），通过读取已有 EtmemNodeState 中的 `status.tasks` 恢复运行中任务的 map，避免将现有 etmem 任务误认为新建：

```go
// cmd/agent/main.go 中的 Bootstrap 逻辑
if err := k8sClient.Get(ctx, nsKey, &existingNodeState); err == nil {
    recovered := tm.BootstrapFromNodeState(existingNodeState.Status.Tasks)
    logger.Info("Bootstrapped running tasks from NodeState", "recovered", recovered)
}
```

Bootstrap 仅恢复内存状态（运行中任务 map），不重新向 etmemd 下发配置（etmemd 重启后需等待 reconcile 重新启动）。

### 7.4 NodeState 查询命令

```bash
# 查看所有节点的 NodeState 摘要
kubectl get etmemnodestate

# 查看指定节点的详细状态
kubectl get etmemnodestate <node-name> -o yaml

# 查看健康字段
kubectl get etmemnodestate <node-name> \
  -o jsonpath='{.status.socketReachable} {.status.etmemdReady} {.status.metrics.totalManagedPods}'

# 查看所有任务
kubectl get etmemnodestate -A \
  -o jsonpath='{range .items[*].status.tasks[*]}{.podName} → {.projectName} ({.state}){"\n"}{end}'
```

## 11. 异常场景与恢复逻辑

### 11.1 Pod 重建场景

**触发条件**：Pod 因故障重建、`kubectl rollout restart`、节点故障迁移等。

**恢复逻辑**：

```
1. 旧 Pod 删除
   → 下一轮 reconcile，Diff 检测旧 project 不在期望集合中
   → TaskManager.StopTask 停止旧任务，清理配置文件

2. 新 Pod 创建（带相同 etmem 标签）
   → Operator PodReconciler 检测到新 Pod，确认 EtmemPolicy 存在（或创建）
   → Agent 下一轮 reconcile，MatchPodToPolicy 匹配新 Pod
   → 解析新 Pod UID 对应的 cgroup 路径和 PID
   → 启动新 project，写回 NodeState
```

**关键点**：新旧 Pod 的 UID 不同，cgroup 路径和 PID 均不同。Agent 通过 Diff 机制自动完成旧任务停止和新任务启动，无需人工干预。

**容错**：若新 Pod 的 etmemd 通路不可达（`transportHealthy=false`），新任务启动失败但 reconcile 继续，下一轮重试（间隔 `reconcileInterval`，默认 30s）。

### 11.2 Agent 重启场景

**触发条件**：Agent Pod 重启（DaemonSet 滚动升级、节点 OOM、手动删除 Pod 等）。

**存储说明**：配置文件目录 `/var/run/etmem/configs` 通过 hostPath 卷挂载到 Agent 容器，存储在宿主机磁盘上，**不随 Pod 重建而消失**。配置文件是否存在，取决于上一次 Agent 退出时是否执行了 `StopAll`。

**恢复逻辑（按重启方式分类）**：

```
情形 A：优雅重启（Agent 收到 SIGTERM，如滚动升级或 kubectl delete pod）
  1. Agent 信号处理器调用 StopAll：
     a. 对每个 running map 中的 project 调用 StopTask
     b. StopTask 向 etmemd 发送 ProjectStop/ObjDel，并删除 hostPath 上的配置文件
     c. running map 清空
  2. 新 Agent Pod 启动，执行 Bootstrap：
     a. 读取 EtmemNodeState.status.tasks
     b. 对每个 state=running 的 project 检查 hostPath 配置文件是否存在
     c. 配置文件已被 StopAll 删除 → 无法写入 running map，recovered=0
  3. 第一轮 reconcile：IsRunning 对所有期望任务返回 false
     → 对所有期望任务执行 StartTask 重新注册

情形 B：崩溃式重启（OOM、kill -9、节点宕机等，进程未执行 StopAll）
  1. hostPath 上的配置文件保留（StopAll 未运行）
  2. 新 Agent Pod 启动，执行 Bootstrap：
     a. BootstrapFromNodeState 对 state=running 且配置文件存在的 project
        写入 running map（recovered=<n>）
  3. 第一轮 reconcile：IsRunning 对已恢复的 project 返回 true
     → diff 不产生差异，StartTask 不被调用
     → 若 etmemd 同时崩溃重启，其侧 project 已丢失，但 Agent 侧认为任务仍在运行
       （此场景需参见 §11.5 的恢复路径）
```

**关键点**：Bootstrap 仅恢复 Agent 内存中的 running map，不影响 etmemd 侧的实际任务状态。情形 A 下 StopAll 保证配置文件清理，下一轮 reconcile 可全量重新注册；情形 B 下 Bootstrap 可成功恢复 running map，但若 etmemd 同时崩溃，则需通过 §11.5 所述路径主动触发重新注册。

### 11.3 Operator 重启场景

**触发条件**：Operator Pod 重启（Deployment 滚动升级、节点故障、手动删除 Pod 等）。

**恢复逻辑**：

```
1. Operator 重启后，controller-runtime 框架自动重新列举所有 Pod 事件
2. PodReconciler 对每个命名空间重新计算并收敛 EtmemPolicy
3. 若 EtmemPolicy 已存在且配置正确，无需修改（幂等）
4. Agent 侧不受影响，继续按现有 Policy 运行任务
```

**关键点**：Operator 与 Agent 之间通过 EtmemPolicy CRD 解耦，Operator 重启不会中断 Agent 侧的任务执行。EtmemPolicy 持久化在 etcd 中，重启后无需重建。

### 11.4 镜像升级场景

**触发条件**：通过 `helm upgrade` 更新 Operator 或 Agent 镜像版本。

**恢复逻辑**：

```
Operator 升级（Deployment 滚动更新）：
  → 新 Operator Pod 启动，接管 controller-runtime 监听
  → 重新 reconcile 所有 Pod 事件，幂等收敛 EtmemPolicy
  → 无服务中断

Agent 升级（DaemonSet RollingUpdate）：
  → 每节点依次：旧 Agent Pod 终止前执行 StopAll（信号处理）
    → 停止所有 etmem project，清理配置文件
  → 新 Agent Pod 启动：Bootstrap 尝试恢复任务 map，但配置文件已被 StopAll 删除，recovered=0
  → 第一轮 reconcile：重新解析 PID，全量重新注册 etmem project
  → NodeState 更新为新状态
```

**容错**：Agent 升级期间，目标节点上的 etmem 任务存在短暂中断（旧 Pod 停止到新 Pod 完成第一轮 reconcile 之间）。中断时间上限为 `reconcileInterval`（默认 30s）加上 Pod 启动时间。

### 11.5 etmemd 异常场景

**触发条件**：etmemd 进程崩溃或 socket 不可达。

**恢复逻辑**：

```
1. Agent reconcile 时通路探测失败（ProjectShow 返回错误）
2. NodeState 中 socketReachable=false, etmemdReady=false
3. Agent 继续 reconcile 但无法启动新任务（StartTask 的 ObjAdd 步骤失败）
4. 待 etmemd 恢复后，下一轮 reconcile 中：
   a. 通路探测成功，transportHealthy=true
   b. NodeState 更新为健康状态
   c. 现有任务的 StartTask 不会被自动重新调用（TaskManager.running map 仍保存旧
      project 记录，IsRunning 返回 true，diff 不会检测到差异）
```

**关键限制**：etmemd 重启后，其侧的 project 信息丢失，但 Agent 的 running map 仍保留这些记录（配置文件未消失，Bootstrap 已写入或运行期间写入的状态均在内存中）。diff 逻辑依赖 running map 判断是否需要 StartTask，无法感知 etmemd 内部状态，因此现有 project 不会自动重新注册。

**恢复路径**：etmemd 重启后，Agent 侧的 running map 仍保有旧 project 记录，diff 逻辑无法自动感知 etmemd 内部状态，因此现有 project 不会被自动重新注册。当前代码支持以下可靠的手动恢复路径：

```
路径 1：通过 suspend 标志触发全量重新注册（推荐）
  1. 将 EtmemPolicy 的 spec.suspend 置为 true：
       kubectl patch etmempolicy <name> -n <namespace> \
         -p '{"spec":{"suspend":true}}' --type=merge
  2. 下一轮 reconcile：suspend=true 使所有期望任务集合为空
     → diff 阶段对 running map 中全部 project 调用 StopTask
     → StopTask 清除 running map 条目并删除 hostPath 上的配置文件
  3. 将 spec.suspend 置回 false：
       kubectl patch etmempolicy <name> -n <namespace> \
         -p '{"spec":{"suspend":false}}' --type=merge
  4. 下一轮 reconcile：IsRunning 对所有期望任务返回 false
     → 对所有期望任务执行 StartTask，向已恢复的 etmemd 重新注册 project

路径 2：优雅重启 Agent Pod（触发 SIGTERM → StopAll）
  1. kubectl rollout restart daemonset/<agent-ds> -n <agent-ns>
  2. SIGTERM → StopAll 删除配置文件并清空 running map
  3. 新 Pod 启动：Bootstrap 找不到配置文件，recovered=0
  4. 第一轮 reconcile：全量执行 StartTask 重新注册
```

**注意**：崩溃式重启 Agent（OOM/kill -9）不触发 StopAll，配置文件保留，Bootstrap 将恢复 running map，reconcile 不会重新注册；此时若 etmemd 同时崩溃，需采用上述路径 1 或路径 2 主动干预。

### 11.6 回滚（Rollback）

**暂停**（停止 etmem 管理行为，不修改工作负载）：

```bash
kubectl patch etmempolicy etmem-auto -n <namespace> \
  -p '{"spec":{"suspend":true}}' --type=merge
```

**完整回滚**（恢复到未启用 etmem 的工作负载状态）：

```bash
# 1. 移除工作负载的 etmem 标签（触发 Operator 删除 EtmemPolicy）
kubectl patch deployment <name> -n <namespace> \
  -p '{"spec":{"template":{"metadata":{"labels":{"etmem.openeuler.io/enable":null}}}}}'

# 2. 验证 EtmemPolicy 已清理
kubectl get etmempolicy -n <namespace>

# 3. 验证 NodeState 任务已清空
kubectl get etmemnodestate <node-name> -o jsonpath='{.status.tasks}'
```

回滚后，etmem 扫描停止，进程已换出的内存页在后续被访问时由内核自动换入，业务进程无需感知。

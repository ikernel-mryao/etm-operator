## 1. 背景

etmem-operator 是一个 Kubernetes Operator，用于管理 etmem（openEuler 开源的内存分级工具），实现对目标 Pod 进程中冷内存页的自动换出（swap）。其核心工作流如下：

```
用户创建带标签的 Pod
  → Operator 自动创建 EtmemPolicy
    → Agent 发现 Policy + Pod
      → Agent 生成 etmemd 配置文件
        → etmemd 扫描进程冷页
          → 冷页被换出
            → VmSwap 增长
```

在控制面验证（Operator / Agent / CRD 生命周期）通过后，我们发现**数据面实际上并未工作**——etmemd 可以扫描目标 PID，但进程的 VmSwap 始终为 0，没有实际的页换出发生。本文档记录了从发现问题到最终修复并验证通过的完整过程。

---

## 2. 问题描述

### 现象

- 控制面验证全部通过：Operator 正确创建 EtmemPolicy，Agent 正确生成配置文件，etmemd 正确启动。
- 但数据面不工作：etmemd 可以扫描目标 PID，但 `VmSwap` 始终为 **0**，无任何实际页换出。

### 预期行为

在 Agent 生成配置并启动 etmemd 项目后，目标进程的 `VmSwap` 应逐渐增长，`VmRSS` 应相应下降。

### 实际行为

```bash
$ grep -E 'VmSwap|VmRSS' /proc/<PID>/status
VmRSS:    270792 kB
VmSwap:        0 kB   # 长时间停留在 0，不增长
```

## 4. 根因分析

经过排查，共发现 **3 个根因**，三者共同导致了数据面不工作。

### 4.1 根因一：sysmem_threshold 在大内存机器上阈值过低

#### 原理

etmemd 配置中的 `sysmem_threshold=N` 含义为：**只有当系统可用内存（MemAvailable）低于总内存（MemTotal）的 N% 时，才触发页换出**。这是一种保护机制，避免在内存充足时做无意义的换出。

#### 问题

在我们的验证环境（1TB 内存的物理机）上，系统可用内存占比高达 **82%**：

```bash
$ grep -E 'MemTotal|MemAvailable' /proc/meminfo
MemTotal:       1056309200 kB   # ≈ 1TB
MemAvailable:    866558832 kB   # ≈ 82% 可用
```

旧的 Profile 配置为：

| Profile | 旧 sysmem_threshold |
|---|---|
| conservative | 30 |
| moderate | 50 |
| aggressive | 70 |

由于 82% > 70% > 50% > 30%，所有 Profile 的阈值**都低于实际可用内存占比**，条件永远不满足，etmemd **永远不会触发换出**。

#### 修复

将各 Profile 的 `sysmem_threshold` 上调，确保在大内存机器上也能触发：

| Profile | 修复后 sysmem_threshold |
|---|---|
| conservative | 85 |
| moderate | 90 |
| aggressive | 95 |
| extreme | 99 |

#### 验证

手动测试：设置 `sysmem_threshold=99`，`type=pid`，指定目标 PID。启动 etmemd 项目后 30 秒内，VmSwap 从 0 增长至 **273MB** ✅

---

### 4.2 根因二：etmemd INI 解析器只加载最后一个 [task] 段

#### 原理

etmemd 的配置文件采用 INI 格式，支持 `[project]`、`[engine]`、`[task]` 三种段。理论上每个配置文件可以有多个 `[task]` 段来同时管理多个进程。

#### 问题

**etmemd 的 INI 解析器实际上只加载最后一个 `[task]` 段**，之前的 `[task]` 段会被静默丢弃（不报错、不告警）。

Agent 在为一个 Pod 生成配置时，会为 Pod 中的每个容器创建一个 `[task]` 段。Kubernetes Pod 通常包含两个容器：

1. **pause 容器**（Kubernetes 基础设施容器，几乎不占内存）
2. **memhog 容器**（用户应用容器，实际目标）

由于 Agent 按容器顺序生成配置，**pause 容器的 `[task]` 总是排在最后**。因此 etmemd 实际上只加载了 pause 容器的任务，而 pause 容器的内存接近于 0——自然换不出任何东西。

生成的有问题的配置文件示意：

```ini
[project]
name=default-memhog-test
...

[engine]
name=slide
project=default-memhog-test

[task]
project=default-memhog-test
engine=slide
name=default-memhog-test_task_0
type=name
value=memhog          # ← 应用容器（实际目标），但被丢弃了！
...

[task]
project=default-memhog-test
engine=slide
name=default-memhog-test_task_1
type=name
value=pause           # ← 基础设施容器（几乎无内存），这才是被加载的！
...
```

#### 修复

在 Agent 的配置生成逻辑中增加 `isInfraProcess()` 过滤函数，跳过以下基础设施容器：

- `pause`（Kubernetes pause 容器）
- `sandbox`（containerd sandbox 容器）

这确保每个配置文件只包含**一个有效的 `[task]` 段**，且指向真正的应用容器。

#### 验证

修复后，生成的配置文件中只有一个 `[task]` 段，`value=memhog`，etmemd 正确加载并开始换出 ✅

---

### 4.3 根因三：残留进程导致 name 匹配干扰

#### 原理

当 `[task]` 中配置 `type=name, value=memhog` 时，etmemd 会匹配**宿主机上所有 comm 名称为 memhog 的进程**，而不仅仅是当前目标 Pod 中的进程。

#### 问题

在反复测试过程中，之前已删除 Pod 的 memhog 进程可能仍然存活（成为孤儿进程或僵尸进程）。这些残留进程：

- 已经被之前的 etmemd 实例换出过，VmSwap 已经很高
- 占据了 etmemd 的匹配结果，导致 etmemd 实际上在处理旧进程
- 新创建的目标 Pod 中的 memhog 进程没有被有效处理

可以通过以下命令确认残留进程是否存在：

```bash
$ pgrep -a memhog
1234567 memhog 256   # ← 旧 Pod 残留的进程
1548266 memhog 256   # ← 新 Pod 的目标进程
```

#### 解决

1. **短期措施**：在测试前清理残留进程：`kill <旧PID>`
2. **长期改进方向**：使用 `type=pid` 替代 `type=name` 进行精确匹配（需要 Agent 获取容器主进程 PID 并填入配置）

## 5. 其他发现

在排查过程中，还发现了以下重要细节：

### 5.1 etmemd 配置文件权限要求

etmemd **严格要求**配置文件权限为 `600`（仅 owner 可读写）或 `400`（仅 owner 可读）。

```bash
# 权限为 644 的文件会被拒绝
$ sudo /usr/bin/etmem project add -f /path/to/config.conf -s etmemd_socket
error: invalid parameters

# 权限为 600 的文件可以正常加载
$ chmod 600 /path/to/config.conf
$ sudo /usr/bin/etmem project add -f /path/to/config.conf -s etmemd_socket
# 成功
```

**注意**：Agent 代码中已正确使用 `0600` 权限写入配置文件 ✅

### 5.2 etmemd 抽象 Unix 套接字

etmemd 使用**抽象 Unix 域套接字**（abstract Unix domain socket）进行通信，而非文件系统上的套接字文件。

- 抽象套接字存在于网络命名空间中，不在文件系统中
- Agent DaemonSet 配置了 `hostNetwork: true`，与宿主机共享网络命名空间
- 因此 Agent 可以直接访问 etmemd 的抽象套接字，**无需额外的 volume mount**

### 5.3 swap_flag=yes 在 FusionOS 5.10 内核上不可用

在 FusionOS（基于 openEuler）的 5.10 内核上，设置 `swap_flag=yes` 会导致 ioctl 失败，`idle_pages` 返回 0 字节，整个扫描流程失效。

**所有 Profile 均已正确设置 `swap_flag=no`** ✅

## 6. 最终端到端验证结果

在全部 3 个根因修复后，进行了完整的端到端验证。

### 环境信息

| 项目 | 值 |
|---|---|
| 节点 | cp0 |
| cgroup 版本 | v1 |
| 操作系统 | FusionOS（基于 openEuler） |
| 内核版本 | 5.10 |
| 物理内存 | 1TB |
| etmemd | 已安装并运行 |

### 测试用例

- **Pod 名称**：memhog-test
- **Pod 标签**：`etmem.openeuler.io/enable: "true"`
- **Pod 注解**：`etmem.openeuler.io/profile: "aggressive"`
- **目标进程 PID**：1548266

### 验证结果

```
初始状态:
  VmRSS:   270,792 kB (约 264 MB)
  VmSwap:        0 kB

约 40 秒后:
  VmRSS:     1,540 kB (约 1.5 MB)
  VmSwap: 272,156 kB (约 265 MB)
```

**结论**：冷内存页被成功换出，VmSwap 从 0 增长至 265MB，VmRSS 相应下降。数据面工作正常 ✅

### 完整管道验证

```
Pod(标签 etmem.openeuler.io/enable: "true")
  → Operator 自动创建 EtmemPolicy
    → Agent 发现 Policy + Pod
      → Agent 生成配置（单 [task]，value=memhog）
        → etmemd 加载配置（slide engine，started=true）
          → VmSwap 增长（0 → 265MB）
```

### 生效的配置文件

```ini
[project]
name=default-memhog-test
scan_type=page
loop=1
interval=1
sleep=1
sysmem_threshold=95
swapcache_high_wmark=5
swapcache_low_wmark=3

[engine]
name=slide
project=default-memhog-test

[task]
project=default-memhog-test
engine=slide
name=default-memhog-test_task_0
type=name
value=memhog
T=1
max_threads=2
swap_flag=no
```

## 8. Profile 参数定义表

修复后各 Profile 的完整参数如下：

| Profile | sysmem_threshold (%) | swapcache_high_wmark | swapcache_low_wmark | loop | interval | sleep | T | max_threads | swap_flag |
|---|---|---|---|---|---|---|---|---|---|
| conservative | 85 | 15 | 10 | 3 | 5 | 3 | 3 | 1 | no |
| moderate | 90 | 10 | 6 | 1 | 1 | 1 | 1 | 1 | no |
| aggressive | 95 | 5 | 3 | 1 | 1 | 1 | 1 | 2 | no |
| extreme | 99 | 2 | 1 | 1 | 1 | 1 | 1 | 2 | no |

**参数说明**：

| 参数 | 含义 |
|---|---|
| `sysmem_threshold` | 当系统可用内存 < 总内存的 N% 时才触发换出 |
| `swapcache_high_wmark` | swap cache 高水位标记（百分比），超过此值时开始回收 swap cache |
| `swapcache_low_wmark` | swap cache 低水位标记（百分比），回收至此值以下 |
| `loop` | 每轮扫描的循环次数 |
| `interval` | 两次扫描之间的间隔（秒） |
| `sleep` | 每轮扫描结束后的休眠时间（秒） |
| `T` | 冷热判定阈值——页面连续 T 次扫描未被访问则判定为冷页 |
| `max_threads` | 扫描使用的最大线程数 |
| `swap_flag` | 是否使用 swap flag 机制（FusionOS 5.10 内核上必须为 no） |

## 10. 总结

### 三个根因及其修复

| 编号 | 根因 | 影响 | 修复方式 |
|---|---|---|---|
| 1 | sysmem_threshold 在大内存机器上过低 | 换出条件永远不满足，VmSwap 始终为 0 | 上调各 Profile 的阈值（85/90/95/99） |
| 2 | etmemd INI 解析器只加载最后一个 [task] | 实际目标被 pause 容器覆盖 | Agent 过滤基础设施容器，确保单 [task] |
| 3 | 残留进程导致 name 匹配干扰 | etmemd 处理旧进程而非新目标 | 测试前清理残留进程；长期改用 type=pid |

### 关键经验

1. **sysmem_threshold 需要根据实际环境调整**：在大内存机器上（如 1TB），可用内存占比通常很高，阈值设置必须留有足够裕度。
2. **etmemd 的 INI 解析器有隐性限制**：多 [task] 段会被静默丢弃，这是 etmemd 自身的已知行为，必须在配置生成侧规避。
3. **name 匹配是全局的**：`type=name` 会匹配宿主机上所有同名进程，需要确保没有干扰进程，或改用 `type=pid` 精确定位。
4. **配置文件权限至关重要**：etmemd 严格检查文件权限，644 会被拒绝，必须为 600 或 400。
5. **内核兼容性需要关注**：`swap_flag=yes` 在某些内核版本上不可用，需要根据实际内核选择合适的参数。

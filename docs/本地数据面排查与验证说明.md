## 1. 背景

etmem-operator 是一个 Kubernetes Operator，用于管理 etmem（openEuler 开源的内存分级工具），实现对目标 Pod 进程中冷内存页的自动换出（swap）。其核心工作流如下：

```
用户创建带标签的 Pod
  → Operator 自动创建 EtmemPolicy
    → Agent 发现 Policy + Pod
      → Agent 生成 etmemd 配置文件
        → etmemd 扫描进程冷页
          → 冷页被换出
            → VmSwap 增长
```

在控制面验证（Operator / Agent / CRD 生命周期）通过后，我们发现**数据面实际上并未工作**——etmemd 可以扫描目标 PID，但进程的 VmSwap 始终为 0，没有实际的页换出发生。本文档记录了从发现问题到最终修复并验证通过的完整过程。

---

## 2. 问题描述

### 现象

- 控制面验证全部通过：Operator 正确创建 EtmemPolicy，Agent 正确生成配置文件，etmemd 正确启动。
- 但数据面不工作：etmemd 可以扫描目标 PID，但 `VmSwap` 始终为 **0**，无任何实际页换出。

### 预期行为

在 Agent 生成配置并启动 etmemd 项目后，目标进程的 `VmSwap` 应逐渐增长，`VmRSS` 应相应下降。

### 实际行为

```bash
$ grep -E 'VmSwap|VmRSS' /proc/<PID>/status
VmRSS:    270792 kB
VmSwap:        0 kB   # 长时间停留在 0，不增长
```

## 4. 根因分析

经过排查，共发现 **3 个根因**，三者共同导致了数据面不工作。

### 4.1 根因一：sysmem_threshold 在大内存机器上阈值过低

#### 原理

etmemd 配置中的 `sysmem_threshold=N` 含义为：**只有当系统可用内存（MemAvailable）低于总内存（MemTotal）的 N% 时，才触发页换出**。这是一种保护机制，避免在内存充足时做无意义的换出。

#### 问题

在我们的验证环境（1TB 内存的物理机）上，系统可用内存占比高达 **82%**：

```bash
$ grep -E 'MemTotal|MemAvailable' /proc/meminfo
MemTotal:       1056309200 kB   # ≈ 1TB
MemAvailable:    866558832 kB   # ≈ 82% 可用
```

旧的 Profile 配置为：

| Profile | 旧 sysmem_threshold |
|---|---|
| conservative | 30 |
| moderate | 50 |
| aggressive | 70 |

由于 82% > 70% > 50% > 30%，所有 Profile 的阈值**都低于实际可用内存占比**，条件永远不满足，etmemd **永远不会触发换出**。

#### 修复

将各 Profile 的 `sysmem_threshold` 上调，确保在大内存机器上也能触发：

| Profile | 修复后 sysmem_threshold |
|---|---|
| conservative | 85 |
| moderate | 90 |
| aggressive | 95 |
| extreme | 99 |

#### 验证

手动测试：设置 `sysmem_threshold=99`，`type=pid`，指定目标 PID。启动 etmemd 项目后 30 秒内，VmSwap 从 0 增长至 **273MB** ✅

---

### 4.2 根因二：etmemd INI 解析器只加载最后一个 [task] 段

#### 原理

etmemd 的配置文件采用 INI 格式，支持 `[project]`、`[engine]`、`[task]` 三种段。理论上每个配置文件可以有多个 `[task]` 段来同时管理多个进程。

#### 问题

**etmemd 的 INI 解析器实际上只加载最后一个 `[task]` 段**，之前的 `[task]` 段会被静默丢弃（不报错、不告警）。

Agent 在为一个 Pod 生成配置时，会为 Pod 中的每个容器创建一个 `[task]` 段。Kubernetes Pod 通常包含两个容器：

1. **pause 容器**（Kubernetes 基础设施容器，几乎不占内存）
2. **memhog 容器**（用户应用容器，实际目标）

由于 Agent 按容器顺序生成配置，**pause 容器的 `[task]` 总是排在最后**。因此 etmemd 实际上只加载了 pause 容器的任务，而 pause 容器的内存接近于 0——自然换不出任何东西。

生成的有问题的配置文件示意：

```ini
[project]
name=default-memhog-test
...

[engine]
name=slide
project=default-memhog-test

[task]
project=default-memhog-test
engine=slide
name=default-memhog-test_task_0
type=name
value=memhog          # ← 应用容器（实际目标），但被丢弃了！
...

[task]
project=default-memhog-test
engine=slide
name=default-memhog-test_task_1
type=name
value=pause           # ← 基础设施容器（几乎无内存），这才是被加载的！
...
```

#### 修复

在 Agent 的配置生成逻辑中增加 `isInfraProcess()` 过滤函数，跳过以下基础设施容器：

- `pause`（Kubernetes pause 容器）
- `sandbox`（containerd sandbox 容器）

这确保每个配置文件只包含**一个有效的 `[task]` 段**，且指向真正的应用容器。

#### 验证

修复后，生成的配置文件中只有一个 `[task]` 段，`value=memhog`，etmemd 正确加载并开始换出 ✅

---

### 4.3 根因三：残留进程导致 name 匹配干扰

#### 原理

当 `[task]` 中配置 `type=name, value=memhog` 时，etmemd 会匹配**宿主机上所有 comm 名称为 memhog 的进程**，而不仅仅是当前目标 Pod 中的进程。

#### 问题

在反复测试过程中，之前已删除 Pod 的 memhog 进程可能仍然存活（成为孤儿进程或僵尸进程）。这些残留进程：

- 已经被之前的 etmemd 实例换出过，VmSwap 已经很高
- 占据了 etmemd 的匹配结果，导致 etmemd 实际上在处理旧进程
- 新创建的目标 Pod 中的 memhog 进程没有被有效处理

可以通过以下命令确认残留进程是否存在：

```bash
$ pgrep -a memhog
1234567 memhog 256   # ← 旧 Pod 残留的进程
1548266 memhog 256   # ← 新 Pod 的目标进程
```

#### 解决

1. **短期措施**：在测试前清理残留进程：`kill <旧PID>`
2. **长期改进方向**：使用 `type=pid` 替代 `type=name` 进行精确匹配（需要 Agent 获取容器主进程 PID 并填入配置）

## 5. 其他发现

在排查过程中，还发现了以下重要细节：

### 5.1 etmemd 配置文件权限要求

etmemd **严格要求**配置文件权限为 `600`（仅 owner 可读写）或 `400`（仅 owner 可读）。

```bash
# 权限为 644 的文件会被拒绝
$ sudo /usr/bin/etmem project add -f /path/to/config.conf -s etmemd_socket
error: invalid parameters

# 权限为 600 的文件可以正常加载
$ chmod 600 /path/to/config.conf
$ sudo /usr/bin/etmem project add -f /path/to/config.conf -s etmemd_socket
# 成功
```

**注意**：Agent 代码中已正确使用 `0600` 权限写入配置文件 ✅

### 5.2 etmemd 抽象 Unix 套接字

etmemd 使用**抽象 Unix 域套接字**（abstract Unix domain socket）进行通信，而非文件系统上的套接字文件。

- 抽象套接字存在于网络命名空间中，不在文件系统中
- Agent DaemonSet 配置了 `hostNetwork: true`，与宿主机共享网络命名空间
- 因此 Agent 可以直接访问 etmemd 的抽象套接字，**无需额外的 volume mount**

### 5.3 swap_flag=yes 在 FusionOS 5.10 内核上不可用

在 FusionOS（基于 openEuler）的 5.10 内核上，设置 `swap_flag=yes` 会导致 ioctl 失败，`idle_pages` 返回 0 字节，整个扫描流程失效。

**所有 Profile 均已正确设置 `swap_flag=no`** ✅

## 8. Profile 参数定义表

修复后各 Profile 的完整参数如下：

| Profile | sysmem_threshold (%) | swapcache_high_wmark | swapcache_low_wmark | loop | interval | sleep | T | max_threads | swap_flag |
|---|---|---|---|---|---|---|---|---|---|
| conservative | 85 | 15 | 10 | 3 | 5 | 3 | 3 | 1 | no |
| moderate | 90 | 10 | 6 | 1 | 1 | 1 | 1 | 1 | no |
| aggressive | 95 | 5 | 3 | 1 | 1 | 1 | 1 | 2 | no |
| extreme | 99 | 2 | 1 | 1 | 1 | 1 | 1 | 2 | no |

**参数说明**：

| 参数 | 含义 |
|---|---|
| `sysmem_threshold` | 当系统可用内存 < 总内存的 N% 时才触发换出 |
| `swapcache_high_wmark` | swap cache 高水位标记（百分比），超过此值时开始回收 swap cache |
| `swapcache_low_wmark` | swap cache 低水位标记（百分比），回收至此值以下 |
| `loop` | 每轮扫描的循环次数 |
| `interval` | 两次扫描之间的间隔（秒） |
| `sleep` | 每轮扫描结束后的休眠时间（秒） |
| `T` | 冷热判定阈值——页面连续 T 次扫描未被访问则判定为冷页 |
| `max_threads` | 扫描使用的最大线程数 |
| `swap_flag` | 是否使用 swap flag 机制（FusionOS 5.10 内核上必须为 no） |

## 七、多进程数据面验证

### 7.1 背景：为什么不能在一个配置文件中写多个 [task]

在 [4.2 节](#42-根因二etmemd-ini-解析器只加载最后一个-task-段) 中已经指出，etmemd 的 INI 解析器**只加载最后一个 `[task]` 段**，之前的段会被静默丢弃。这意味着：

```ini
# ❌ 错误：多 [task] 配置，etmemd 只会加载 task_2，task_1 被丢弃
[project]
name=my-project
...
[engine]
name=slide
project=my-project

[task]
name=task_1
value=process_a    # ← 被丢弃！

[task]
name=task_2
value=process_b    # ← 只有这个被加载
```

因此，**如果一个 Pod 中有多个用户进程，不能把它们写在同一个配置文件的多个 [task] 中**。

### 7.2 正确的多进程模型：一个进程一个项目

Agent v0.3.0-mp 采用的方案是：**为每个用户进程创建独立的 etmemd 项目**，每个项目有自己的配置文件，每个配置文件只包含一个 `[project]`、一个 `[engine]`、一个 `[task]`。

```
Pod: multiproc-test（包含 memeat1、memeat2 两个进程）
  ├── 项目 1: default-multiproc-test-memeat1
  │   └── 配置文件: default-multiproc-test-memeat1.conf（[task] value=memeat1）
  ├── 项目 2: default-multiproc-test-memeat2
  │   └── 配置文件: default-multiproc-test-memeat2.conf（[task] value=memeat2）
  └── 项目 3: default-multiproc-test-sh
      └── 配置文件: default-multiproc-test-sh.conf（[task] value=sh，容器入口进程）
```

### 7.3 Agent 实现方式

Agent 在 reconcile 循环中，对每个 Pod 的进程列表进行遍历：

1. **过滤基础设施进程**：跳过 `pause`、`sandbox` 等 Kubernetes 基础设施进程（`isInfraProcess()` 函数）
2. **逐进程生成项目**：对每个用户进程调用 `ProjectNameForProcess()` 生成项目名，格式为 `{namespace}-{podName}-{processName}`
3. **名称长度保护**：当项目名超过 64 字符时，使用 SHA256 截断确保不超长
4. **独立配置文件**：每个进程对应一个 `.conf` 文件，确保每个文件只有单个 `[task]` 段

### 7.4 实际验证结果

**测试日期**：2026-04-17
**测试环境**：FusionOS 23 / 5.10 内核 / 1TB RAM / cgroup v1

#### 多进程测试 Pod（multiproc-test）

Pod 内包含 2 个内存消费进程（memeat1 + memeat2），各分配 150MB 内存。

| PID | 进程名 | VmRSS | VmSwap | 说明 |
|---|---|---|---|---|
| 2338929 | memeat1 | 1,788 kB | 152,424 kB (148 MB) | ✅ 冷页已换出 |
| 2338930 | memeat2 | 1,784 kB | 152,424 kB (148 MB) | ✅ 冷页已换出 |
| **合计** | | | **304,848 kB (297 MB)** | 两个进程均被管理 |

#### etmemd 项目列表

```bash

```

显示 4 个独立项目：

| 项目名 | 说明 |
|---|---|
| `default-multiproc-test-memeat1` | 多进程 Pod 的 memeat1 进程 |
| `default-multiproc-test-memeat2` | 多进程 Pod 的 memeat2 进程 |
| `default-multiproc-test-sh` | 多进程 Pod 的容器入口（sh wrapper） |
| `default-memhog-test-memhog` | 单进程基线 Pod |

每个配置文件均只包含一个 `[project]`、一个 `[engine]`、一个 `[task]`，无多 `[task]` 回归：


### 7.5 与单进程基线对比

| 测试场景 | Pod | 进程数 | 总 VmSwap | 说明 |
|---|---|---|---|---|
| 单进程基线 | memhog-test | 1（memhog） | 287,424 kB (280 MB) | 单进程正常换出 |
| 多进程测试 | multiproc-test | 2（memeat1 + memeat2） | 304,848 kB (297 MB) | 两个进程均被独立管理和换出 |

**结论**：多进程场景下，每个进程的冷页都被独立换出，总 VmSwap 为各进程之和，与预期一致。

### 7.6 如何判断多进程是否正常工作

1. **检查 etmemd 项目数**：

   ```bash
   sudo /usr/bin/etmem project show -s etmemd_socket
   ```

   如果一个 Pod 有 N 个用户进程，应该看到 N 个（或 N+1 个，包含 sh wrapper）对应的项目。

2. **逐进程检查 VmSwap**：

   ```bash
   # 获取 Pod 中所有进程的 PID（从 Agent 日志或 cgroup tasks 文件获取）
   for pid in <PID1> <PID2>; do
     echo "PID=$pid:"
     grep -E 'VmSwap|VmRSS' /proc/$pid/status
   done
   ```

   每个进程的 VmSwap 都应该 > 0。

3. **如果只看到一个项目**：说明 Agent 可能仍在使用旧的单进程代码（`processes[:1]` 截断），需要升级到 v0.3.0-mp 或更新版本。

## 10. 总结

### 三个根因及其修复

| 编号 | 根因 | 影响 | 修复方式 |
|---|---|---|---|
| 1 | sysmem_threshold 在大内存机器上过低 | 换出条件永远不满足，VmSwap 始终为 0 | 上调各 Profile 的阈值（85/90/95/99） |
| 2 | etmemd INI 解析器只加载最后一个 [task] | 实际目标被 pause 容器覆盖 | Agent 过滤基础设施容器，确保单 [task] |
| 3 | 残留进程导致 name 匹配干扰 | etmemd 处理旧进程而非新目标 | 测试前清理残留进程；长期改用 type=pid |

### 关键经验

1. **sysmem_threshold 需要根据实际环境调整**：在大内存机器上（如 1TB），可用内存占比通常很高，阈值设置必须留有足够裕度。
2. **etmemd 的 INI 解析器有隐性限制**：多 [task] 段会被静默丢弃，这是 etmemd 自身的已知行为，必须在配置生成侧规避。
3. **name 匹配是全局的**：`type=name` 会匹配宿主机上所有同名进程，需要确保没有干扰进程，或改用 `type=pid` 精确定位。
4. **配置文件权限至关重要**：etmemd 严格检查文件权限，644 会被拒绝，必须为 600 或 400。
5. **内核兼容性需要关注**：`swap_flag=yes` 在某些内核版本上不可用，需要根据实际内核选择合适的参数。

### PID 作用域修复

#### 旧方案的问题：type=name 节点级匹配

在 v0.4.0-pid 之前，Agent 生成的 etmemd 配置使用 `type=name` + `value=<进程名>`。etmemd 内部通过 `pgrep -x <name>` 查找目标进程。这意味着：

- **节点级全局匹配**：`pgrep -x memhog` 会匹配宿主机上所有名为 `memhog` 的进程，而不仅是目标 Pod 中的进程。
- **跨 Pod 干扰**：如果同一节点上有两个 Pod 运行同名进程（如两个 memhog），etmemd 无法区分它们。
- **只取第一个匹配**：etmemd 的 `pgrep` 匹配只取第一个结果，可能换出错误的进程。

这在多租户集群中是不可接受的安全和正确性问题。

#### 修复方案：type=pid + cgroup PID 解析

Agent 现在通过以下流程精确定位 Pod 内的进程：

1. **从 cgroup 路径获取 PID 列表**：读取 `/sys/fs/cgroup/memory/<kubepods-path>/cgroup.procs`，仅获取属于目标 Pod 容器的进程。
2. **使用 `type=pid`**：每个 PID 在 etmemd 配置中使用 `type=pid` + `value=<PID>`，精确定位到具体进程。
3. **每个 PID 独立项目**：项目命名格式为 `{namespace}-{podName}-{processName}-p{pid}`。如果名称超过 64 字符，使用 SHA256 截断保证唯一性。

示例配置（修复后）：

```ini
[project]
name=etmem-test-memhog-test-memhog-p1548266

[engine]
name=slide
project=etmem-test-memhog-test-memhog-p1548266

[task]
type=pid
value=1548266
name=slide
engine=slide
...
```

#### E2E 隔离验证结果

在同一节点上运行 3 个 memhog 进程，仅对其中一个 Pod 启用 etmem：

| PID | Pod | 启用标签 | VmSwap | VmRSS | 结果 |
|---|---|---|---|---|---|
| 1548266 | memhog-test | ✅ `enable=true` | 287 MB | 1.6 MB | ✅ 冷页被换出 |
| 494856 | memhog-noenable | ❌ 无标签 | 0 | 270 MB | ✅ 未被干扰 |
| 2484659 | 其他 Pod | ❌ 无标签 | 0 | 277 MB | ✅ 未被干扰 |

多进程场景同样验证通过：

| Pod | 进程 | VmSwap |
|---|---|---|
| multiproc-test | memeat1 | 152 MB |
| multiproc-test | memeat2 | 152 MB |

Agent 日志中仅出现已启用 Pod 的 PID，未出现 memhog-noenable 或其他 Pod 的进程。

#### 项目命名规则

- 格式：`{namespace}-{podName}-{processName}-p{pid}`
- 示例：`etmem-test-memhog-test-memhog-p1548266`
- 长度限制：etmemd 项目名最大 64 字符
- 超长处理：对原始名称进行 SHA256 哈希，取前缀拼接哈希截断值，保证唯一性且不超长
- 与旧命名 `{namespace}-{podName}-{processName}` 的区别：新命名包含 PID 后缀，每个进程独立可追踪

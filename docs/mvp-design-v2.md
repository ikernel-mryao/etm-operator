# etmem-operator v0.2.0 设计说明（Pod-driven 架构）

## 1. 设计变更背景

### 1.1 从 Policy-driven 到 Pod-driven 的转变

v0.1.0 采用 Policy-driven 模式，用户需要手工创建和管理 EtmemPolicy CRD 资源。这种设计存在以下问题：

- **用户体验复杂**：需要理解 CRD 概念和 YAML 配置
- **生命周期管理困难**：Policy 与 Pod 生命周期不同步，容易出现孤儿资源
- **权限要求高**：用户需要 etmempolicies 的创建权限

v0.2.0 改为 Pod-driven 模式，用户只需在 Pod 上添加 label 和 annotation：

- **简化用户 API**：仅需 `kubectl label` 和 `kubectl annotate`
- **自动生命周期管理**：Policy 随 Pod 创建/删除自动管理
- **权限友好**：用户只需 pods 标签权限

### 1.2 namespace 级策略整合

v0.2.0 在每个 namespace 内维护一个自动生成的 EtmemPolicy（名称固定为 `etmem-auto`），整合该 namespace 下所有启用 etmem 的 Pod 需求。

## 2. 用户 API 设计

### 2.1 启用 etmem

```yaml
apiVersion: v1
kind: Pod
metadata:
  labels:
    etmem.openeuler.io/enable: "true"  # 必需：启用 etmem
  annotations:
    etmem.openeuler.io/profile: "aggressive"  # 可选：指定性能档位
```

### 2.2 API 语义

- **Label `etmem.openeuler.io/enable`**：
  - `"true"`：启用 etmem 内存分层
  - 其他值或不存在：不启用
  - 变更会触发 PodReconciler 重新计算 namespace 策略

- **Annotation `etmem.openeuler.io/profile`**：
  - 值范围：`conservative` | `moderate` | `aggressive` | `extreme`
  - 默认值：`aggressive`
  - 用于选择内存分层激进程度

### 2.3 用户不直接操作 CRD

用户无需了解 EtmemPolicy 和 EtmemNodeState CRD，这些由 Operator 自动管理：

- **EtmemPolicy**：Operator 根据 Pod labels/annotations 自动生成
- **EtmemNodeState**：Agent 自动创建和更新

## 3. Profile 参数映射表

### 3.1 四档性能配置

| Profile | Loop | Interval | Sleep | SysMem% | HighWmark | LowWmark | T | MaxThreads | SwapFlag |
|---------|------|----------|-------|---------|-----------|----------|---|------------|----------|
| conservative | 3 | 5 | 3 | 70 | 15 | 10 | 3 | 1 | no |
| moderate | 1 | 1 | 1 | 50 | 10 | 6 | 1 | 1 | no |
| aggressive | 1 | 1 | 1 | 30 | 5 | 3 | 1 | 2 | no |
| extreme | 1 | 1 | 1 | 20 | 2 | 1 | 1 | 2 | no |

### 3.2 参数说明

- **Loop**：扫描轮次，值越小频率越高
- **Interval**：扫描间隔（秒），值越小越频繁
- **Sleep**：休眠时间（秒），扫描后的等待时间
- **SysMem%**：系统内存使用阈值，低于此值开始 swap
- **HighWmark/LowWmark**：swapcache 高/低水位（%），控制刷盘行为
- **T**：冷页判断阈值（访问次数），值越小越激进
- **MaxThreads**：并发线程数，影响 swap 速度
- **SwapFlag**：固定为 `no`，规避 FusionOS 内核 idle_pages ioctl 问题

### 3.3 swapcache 行为链路

```
etmem slide 扫描冷页 → 标记换出 → 进入 swapcache（内存中）
    → swapcache > high_wmark% 时触发刷盘
    → 写入 swap 设备  
    → 刷至 < low_wmark%
    → 物理页框释放 → available 内存增长
```

**关键理解**：
- swapcache 是内存中的缓冲区，不等于真正写入磁盘
- high_wmark 越低，越快触发刷盘，内存释放越快
- extreme 档位 (2%) 最激进，conservative 档位 (15%) 最保守

## 4. PodReconciler 逻辑

### 4.1 监听机制

PodReconciler 通过 Kubernetes Controller 模式监听：

```go
// 监听 Pod label 变更事件
func (r *PodReconciler) watchPods() error {
    return ctrl.NewControllerManagedBy(r.Manager).
        For(&corev1.Pod{}).
        WithOptions(controller.Options{}).
        Complete(r)
}
```

### 4.2 Reconcile 逻辑

每次 Pod 事件触发时，PodReconciler 执行以下步骤：

1. **获取 namespace 内所有启用 Pod**：
   ```go
   enabledPods := []corev1.Pod{}
   for _, pod := range pods.Items {
       if pod.Labels["etmem.openeuler.io/enable"] == "true" && 
          pod.Status.Phase != corev1.PodSucceeded && 
          pod.Status.Phase != corev1.PodFailed {
           enabledPods = append(enabledPods, pod)
       }
   }
   ```

2. **解析最激进 profile**（most-aggressive-wins）：
   ```go
   profiles := []string{"conservative", "moderate", "aggressive", "extreme"}
   finalProfile := "aggressive" // 默认值
   for _, pod := range enabledPods {
       if profile := pod.Annotations["etmem.openeuler.io/profile"]; profile != "" {
           if getProfileIndex(profile) > getProfileIndex(finalProfile) {
               finalProfile = profile
           }
       }
   }
   ```

3. **创建/更新 etmem-auto 策略**：
   ```go
   policy := &etmemv1alpha1.EtmemPolicy{
       ObjectMeta: metav1.ObjectMeta{
           Name:      "etmem-auto",
           Namespace: namespace,
           Labels:    map[string]string{"etmem.openeuler.io/auto-generated": "true"},
       },
       Spec: etmemv1alpha1.EtmemPolicySpec{
           NodePolicyTemplate: generateNodePolicy(finalProfile),
       },
   }
   ```

4. **清理策略**（当无启用 Pod 时）：
   ```go
   if len(enabledPods) == 0 {
       return r.deleteAutoPolicy(ctx, namespace)
   }
   ```

### 4.3 生命周期管理

- **创建**：首个 Pod 启用 etmem 时创建 `etmem-auto` 策略
- **更新**：Pod profile 变更时更新策略参数
- **删除**：最后一个 Pod 禁用或删除时清理策略
- **保护**：不会覆盖用户手工创建的策略（通过 `LabelAutoGenerated` 标识区分）

## 5. Agent 变更

### 5.1 全进程模式

v0.2.0 Agent 默认对匹配 Pod 内的**所有进程**执行内存分层：

```go
// v0.2.0: ProcessFilter 为空，表示所有进程
nodePolicy := &etmemv1alpha1.NodePolicy{
    SlideConfig: &etmemv1alpha1.SlideConfig{
        ProcessFilter: nil, // 全进程模式
    },
}
```

### 5.2 进程去重

Agent 按进程名去重，避免重复处理：

```go
func (a *Agent) deduplicateProcesses(processes []Process) []Process {
    seen := make(map[string]bool)
    result := []Process{}
    for _, p := range processes {
        if !seen[p.Name] {
            seen[p.Name] = true
            result = append(result, p)
        }
    }
    return result
}
```

### 5.3 Bootstrap 恢复机制

Agent 启动时调用 `BootstrapFromNodeState()` 恢复运行中的任务状态：

```go
func (a *Agent) BootstrapFromNodeState(ctx context.Context) error {
    nodeState := &etmemv1alpha1.EtmemNodeState{}
    if err := a.Client.Get(ctx, types.NamespacedName{
        Name: a.NodeName,
    }, nodeState); err != nil {
        return err
    }
    
    // 恢复运行中的任务
    for _, task := range nodeState.Status.RunningTasks {
        if err := a.restoreTask(task); err != nil {
            log.Error(err, "failed to restore task", "task", task.Name)
        }
    }
    return nil
}
```

### 5.4 cgroup v1 路径解析

Agent 通过 cgroup v1 路径查找 Pod 容器进程：

```bash
# cgroupfs 驱动路径
/sys/fs/cgroup/memory/kubepods/burstable/pod<UID>/<container-id>/cgroup.procs

# systemd 驱动路径  
/sys/fs/cgroup/memory/kubepods.slice/kubepods-burstable.slice/kubepods-burstable-pod<UID>.slice/cri-containerd-<container-id>.scope/cgroup.procs
```

## 6. RBAC 设计

### 6.1 Operator 权限

```yaml
rules:
- apiGroups: ["etmem.openeuler.io"]
  resources: ["etmempolicies"]
  verbs: ["get", "list", "watch", "create", "update", "delete"]
- apiGroups: ["etmem.openeuler.io"] 
  resources: ["etmempolicies/status"]
  verbs: ["get", "update", "patch"]
- apiGroups: ["etmem.openeuler.io"]
  resources: ["etmemnodestates"]
  verbs: ["get", "list", "watch"]
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
- apiGroups: [""]
  resources: ["events"]
  verbs: ["create", "patch"]
```

**权限理由**：
- **etmempolicies 全权限**：需要创建/更新/删除自动策略
- **etmemnodestates 只读**：监听节点状态变化
- **pods 只读**：监听 Pod label/annotation 变更
- **events 创建**：记录操作事件

### 6.2 Agent 权限

```yaml  
rules:
- apiGroups: ["etmem.openeuler.io"]
  resources: ["etmempolicies"]
  verbs: ["get", "list", "watch"]
- apiGroups: ["etmem.openeuler.io"]
  resources: ["etmemnodestates"]
  verbs: ["get", "list", "create", "update", "delete"]
- apiGroups: ["etmem.openeuler.io"]
  resources: ["etmemnodestates/status"] 
  verbs: ["get", "update", "patch"]
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
- apiGroups: [""]
  resources: ["nodes"]
  verbs: ["get"]
- apiGroups: [""]
  resources: ["events"]
  verbs: ["create", "patch"]
```

**权限理由**：
- **etmempolicies 只读**：获取策略配置
- **etmemnodestates 全权限**：管理节点状态资源
- **pods 只读**：查找目标 Pod 进程
- **nodes 只读**：获取节点信息
- **events 创建**：记录执行事件

## 7. 资源限制默认值

### 7.1 Operator 资源配置

```yaml
resources:
  requests:
    cpu: 50m
    memory: 64Mi
  limits:
    cpu: 200m  
    memory: 256Mi
```

### 7.2 Agent 资源配置

```yaml
resources:
  requests:
    cpu: 50m
    memory: 64Mi
  limits:
    cpu: 500m
    memory: 512Mi
```

### 7.3 资源规划理由

- **Operator**：轻量级控制器，主要处理 Pod 事件和策略 CRUD
- **Agent**：需要与 etmemd 交互、进程扫描、cgroup 解析，CPU 和内存需求更高
- **Requests**：保证最小可用资源
- **Limits**：防止资源滥用，确保集群稳定性

## 8. 当前限制与后续扩展

### 8.1 当前限制

1. **cgroup 版本**：仅支持 cgroup v1，v2 支持计划在 v1.0.0
2. **Swap 设备**：需要预配置 swap 分区或文件
3. **内核模块**：依赖 etmemd 内核模块和 idle_pages 支持
4. **容器运行时**：当前验证 containerd，其他运行时待测试
5. **Profile 固化**：四个档位参数不支持用户自定义
6. **单租户**：暂无 namespace 隔离和多租户安全机制

### 8.2 扩展规划

1. **v0.3.0**：SlideOverrides 支持用户自定义 swapcache 水位
2. **v0.4.0**：Agent informer 节点过滤，减少不必要的 API 调用
3. **v1.0.0**：cgroup v2 支持，多租户安全加固
4. **v1.1.0**：支持 CRI-O、Docker 等多容器运行时
5. **v1.2.0**：动态 Profile 配置，支持用户自定义参数

### 8.3 兼容性承诺

- **API 兼容**：v0.2.x 系列保持 Pod label/annotation API 稳定
- **CRD 兼容**：EtmemPolicy 和 EtmemNodeState schema 向前兼容
- **升级路径**：提供平滑升级脚本，保证零停机时间升级

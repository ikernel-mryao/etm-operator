> **决策状态：** 已确认

---

## 1. 问题陈述

当前 etmem-operator 已经在真实环境中验证了以下闭环：

- Pod/工作负载打标签后，Operator 自动创建/收敛 EtmemPolicy
- Agent 以 DaemonSet 形式在节点侧匹配目标 Pod、解析 PID、下发 etmem task
- NodeState 可以回写节点任务状态与 etmemd 通路状态
- 目标 Pod 多进程 `VmSwap > 0`，整机 `Swap used` 增长，说明数据面真实生效

但在进入 v1.0 正式开发前，仍缺少一套面向研发交付的正式文档，去明确：

1. v1.0 为什么做、范围是什么、验收标准是什么
2. 当前 Operator + Agent(DaemonSet) 方案的总体架构、部署模型、恢复模型和交付模型
3. 具体到 CRD、reconcile、PID 解析、NodeState、Helm、离线交付与验证脚本的详细实现边界

本设计规格用于约束后续三份正式文档的结构、边界和核心立场。

## 3. 核心事实基线

三份正式文档必须建立在以下事实之上：

### 3.1 已通过验证的事实

1. **控制面闭环通过**
   - 标签驱动自动生成 Policy
   - Pod 重建后可重新纳管
   - suspend / rollback 路径已验证

2. **节点执行面通过**
   - Agent 以 DaemonSet 方式运行
   - 能根据 cgroup v1 路径解析目标 Pod 进程
   - NodeState 能反映任务与节点健康状态

3. **数据面通过**
   - 真实目标环境中目标进程出现 `VmSwap > 0`
   - `Swap used` 增长，证明换出已发生

4. **交付链路已跑通**
   - Helm 安装/升级路径可用
   - 离线镜像导入 + 本地 tag 部署路径可用
   - 现有工作负载（如 DaemonSet）可通过 patch 纳管

### 3.2 已知但不构成架构失败的现象

1. `MemAvailable` 可能不随单次验证稳定增长  
   原因是它是整机净值，会受其他内存活动、页缓存和内核回收影响。

2. 不同进程的 `VmSwap` 增长幅度可能不同  
   这与进程实际 RSS、冷热页分布、生命周期有关，不代表方案失效。

3. 单次观测中的 sleep / wrapper 进程可能 `VmSwap = 0`  
   这属于正常现象，重点应看主要业务进程和总体换出结果。

## 6. 正式文档中的关键立场

三份正式文档必须统一使用以下立场：

### 6.1 关于方案可行性

当前真实验证结果已经足以证明 **Operator + Agent(DaemonSet) + etmemd** 方案可行，可直接作为 v1.0 开发基线。

### 6.2 关于 `MemAvailable`

文档中将其定义为：

- **主机级辅助观测项**
- 不是单次验收硬门槛
- 验证时应与 `VmSwap`、`Swap used`、NodeState、Agent 日志联合判断

### 6.3 关于工程化重点

v1.0 的重点是：

- 稳定部署
- 可升级/可恢复
- 可观测
- 可复现验证
- 可离线交付

而不是：

- 推翻 MVP 架构
- 大幅扩展高级参数面
- 把所有未来特性都塞入第一阶段

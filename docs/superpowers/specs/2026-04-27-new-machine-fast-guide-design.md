# 新机器部署与验证快速说明设计规格

> **决策状态：** 已确认
> **文档定位：** 方案 1 —— 在 `docs/` 下新增独立专题文档
> **目标读者：** 需要在第 3 台或其他相似新机器上快速复现 etmem-operator 部署与验证的实施人员

---

## 1. 问题陈述

在新的真实环境机器上复用已验证交付物时，现场最容易出现三类困惑：

1. 已经离线导入镜像，为什么还需要 Helm。
2. 执行 `helm upgrade` 时，为什么会报 `has no deployed releases`。
3. `etmem-operator` 代码仓除了脚本，实际还承担什么作用。

现有《其他机器快速上手手册》覆盖了完整流程，但篇幅较长，不适合在现场针对单次“新机器快速跑通”问题做高效排障和转发。因此需要一份更短、更聚焦的新文档，直接回答这次真实问题，并给出最短执行路径。

---

## 2. 已确认设计决策

| # | 主题 | 结论 |
|---|------|------|
| 1 | 文档位置 | 放在 `etmem-operator/docs/` 目录下 |
| 2 | 文档形式 | 独立新文档，不并入现有长手册 |
| 3 | 文档范围 | 只覆盖新机器最快落地、报错解释、代码仓作用、镜像与代码关系 |
| 4 | 写作基线 | 基于已真实验证通过的 MVP 行为，不虚构未验证路径 |
| 5 | 环境前提 | 默认目标机器已具备 etmem、swap、相似运行环境 |
| 6 | 操作风格 | 小白可照做，强调“这一步做什么、为什么做、看到什么算正常” |
| 7 | 参数建议 | `extreme` 作为更易触发观测的推荐策略，在启用 etmem 之前设置 |

---

## 3. 文档职责边界

新文档必须回答以下问题：

1. 为什么 `helm upgrade` 会在新机器上失败。
2. 什么情况下应使用 `helm install` 或 `helm upgrade --install`。
3. 代码仓在部署、验证、构建中的实际作用分别是什么。
4. 离线镜像 tar 与代码仓源码、二进制、Helm chart 之间是什么关系。
5. 在“环境已类似、镜像已准备”的前提下，最快执行步骤是什么。
6. 是否需要 `git pull`、Go、`rollout restart`，以及这些动作分别在什么情况下需要。

新文档不应展开以下内容：

- 不重新讲完整架构设计
- 不重复完整环境安装说明
- 不扩写为通用产品手册
- 不把 `MemAvailable` 写成单次硬验收门槛

---

## 4. 文档章节设计

正式文档固定为 6 节：

1. **适用场景与结论**
2. **为什么 `helm upgrade` 会报 `has no deployed releases`**
3. **`etmem-operator` 代码仓的作用**
4. **镜像 tar 与代码仓的关系**
5. **新机器最快执行步骤**
6. **常见误区与判断口径**

其中第 5 节必须给出按顺序执行的最短路径，并明确：

- `nerdctl load` 只负责导入镜像，不负责改集群对象
- Helm 负责安装/升级 Deployment、DaemonSet、CRD 关联资源
- 当镜像 tag 不变或不确定 Pod 是否已更新时，才建议 `rollout restart`
- `extreme` 应在启用 `etmem.openeuler.io/enable=true` 之前设置

---

## 5. 关键立场

文档必须统一使用以下立场：

1. **这次 Helm 报错不是代码坏了，而是 release 状态不满足 upgrade 前提。**
2. **第 3 台机器如果只是复用现成交付物，不需要安装 Go。**
3. **复制最新代码仓目录不是为了“只有脚本可用”，而是为了同时带上 Helm chart、脚本、文档和源码交付基线。**
4. **离线镜像 tar 是基于代码仓内容构建出来的交付物，不是与代码仓无关的独立产物。**
5. **当前真实验证结果已足以支撑“Operator + Agent(DaemonSet)”方案可行。**

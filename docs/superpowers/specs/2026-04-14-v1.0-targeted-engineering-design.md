# etmem-operator v1.0 定向工程补齐设计规格

> **决策状态：** 全部已确认  
> **架构路径：** 方案 A（渐进增强），不做大规模重构  
> **基线：** Pod-driven MVP（Task 1 已完成，ProcessFilter 已改为可选指针）

---

## 1. 问题陈述

当前 Pod-driven MVP 核心架构已验证正确，但距离 v1.0 正式版本仍缺少：
- Profile annotation 支持（用户无法为不同 Pod 选择不同策略）
- extreme profile（swapcache_high_wmark=2/low_wmark=1）
- Agent 启动恢复（重启后 TaskManager.running 丢失）
- Helm 工程化（无 resources 限制、无交付脚本）
- 完整文档（设计、验证、使用指导、Post-MVP 路线图）

本设计在现有代码基础上做定向修改，最大化复用已有验证结果。

## 2. 已确认设计决策

| # | 决策 | 结论 |
|---|------|------|
| 1 | 用户 API | 只有 Pod label（enable/disable）+ annotation（profile 选择） |
| 2 | EtmemPolicy | 内部实现细节，用户不创建不操作，文档不暴露 |
| 3 | 配置分层 | Pod annotation `etmem.openeuler.io/profile` 选择 profile |

| 10 | 进程范围 | v1.0：Pod 内所有进程。后续版本支持按进程名筛选 |

## 3. 用户 API 设计

### 3.1 Enable 标签

```yaml
metadata:
  labels:
    etmem.openeuler.io/enable: "true"    # 必须，启用 etmem
  annotations:
    etmem.openeuler.io/profile: "aggressive"  # 可选，默认 aggressive
```

### 3.2 行为规则

| Pod 状态 | 结果 |
|----------|------|
| enable=true, 无 profile annotation | 使用默认 profile（aggressive） |
| enable=true, profile=extreme | 使用 extreme profile |
| enable=false 或无 label | 不触发 etmem |
| Pod 删除 | 策略自动回收 |
| Pod 重建 | 策略自动重建 |

### 3.3 不支持的操作（v1.0 scope out）

- 用户手动创建/修改 EtmemPolicy
- 同一 Pod 内不同进程使用不同配置
- 按进程名筛选（ProcessFilter.Names）

## 4. Profile 参数模型

### 4.1 完整 Profile 映射

| 参数 | conservative | moderate | aggressive | **extreme** |
|------|-------------|----------|------------|-------------|
| loop | 3 | 1 | 1 | 1 |
| interval | 5 | 1 | 1 | 1 |
| sleep | 3 | 1 | 1 | 1 |
| sysmem_threshold | 70% | 50% | 30% | 20% |
| swapcache_high_wmark | 15 | 10 | 5 | **2** |
| swapcache_low_wmark | 10 | 6 | 3 | **1** |
| T | 3 | 1 | 1 | 1 |
| max_threads | 1 | 1 | 2 | 2 |
| swap_flag | no | no | no | no |

### 4.2 swapcache → available 行为链路

extreme profile 的预期行为：
1. etmem slide 扫描进程内存页，标记冷页
2. 冷页被迁移到 swapcache
3. swapcache_high_wmark=2：当 swapcache 达到总内存 2% 时开始刷入 swapfile
4. swapcache_low_wmark=1：刷到 swapcache 降至 1% 时停止
5. 数据从 swapcache 写入 swapfile 后，物理内存释放
6. `free -h` 中 available 增长

**注意：** swap_flag=no 是 FusionOS 5.10 内核兼容性要求，不可更改。

### 4.3 参数暴露策略

- v1.0：swapcache_high/low_wmark 由 profile 派生，不在 CRD 暴露
- 理由：减少 API 面复杂度，用户通过 profile 选择即可获得合理默认值
- 未来：如有高级用户需求，在 SlideOverrides 中增加 override 字段

## 5. PodReconciler 设计

### 5.1 职责

- Watch 所有 Pod 的 create/update/delete 事件
- 识别 `etmem.openeuler.io/enable: "true"` label
- **按 namespace 聚合**：同一 namespace 内所有 enable=true 的 Pod 共享一个 `etmem-auto` EtmemPolicy
- **读取 profile annotation**：从触发 reconcile 的 Pod 或同 namespace 中第一个 enable Pod 的 annotation 读取 profile
- 当 namespace 内不再有 enable Pod 时，自动删除 `etmem-auto` policy

### 5.2 Profile annotation 处理

自动生成的 EtmemPolicy 的 profile 取值逻辑：
1. 扫描 namespace 内所有 enable=true 的 Pod 的 `etmem.openeuler.io/profile` annotation
2. 如果所有 Pod 指定相同 profile → 使用该 profile
3. 如果不同 Pod 指定不同 profile → 使用最激进的那个（extreme > aggressive > moderate > conservative）
4. 如果无 Pod 指定 profile → 使用默认值 aggressive

### 5.3 Auto-policy 格式

```yaml
apiVersion: etmem.openeuler.io/v1alpha1
kind: EtmemPolicy
metadata:
  name: etmem-auto
  namespace: <pod-namespace>
  labels:
    etmem.openeuler.io/auto-generated: "true"
spec:
  selector:
    matchLabels:
      etmem.openeuler.io/enable: "true"
  engine:
    type: slide
    profile: <resolved-profile>
  # processFilter: nil → 所有进程
```

## 6. Agent 启动恢复

### 6.1 当前问题

TaskManager.running 是纯内存 map，Agent 重启后丢失。当前依赖下一次 reconcile 周期（30s）重建。

### 6.2 v1.0 方案

Agent 启动时执行 bootstrap 恢复：
1. 读取本节点的 EtmemNodeState
2. 检查 NodeState.Status.Tasks 中记录的 running 任务
3. 对每个记录的任务，检查配置文件是否仍存在于 `/var/run/etmem/configs/`
4. 如果配置文件存在，将 projectName → configPath 加入 running map
5. 如果配置文件不存在（说明 etmemd 也已重启），标记为需重建
6. 首次 reconcile 会重建所有缺失任务

### 6.3 接口

```go
// BootstrapFromNodeState 从 NodeState 恢复 TaskManager 的 running map
func (tm *TaskManager) BootstrapFromNodeState(tasks []etmemv1.NodeTask) int
```

## 9. 未来扩展预留

### 9.1 当前限制（v1.0 有意 scope out）

- **进程名筛选：** 当前对 Pod 内所有进程统一 swap，不支持按进程名筛选
- **多配置：** 同一 Pod 内不同进程无法使用不同 etmem 配置
- **Profile override：** swapcache_high/low_wmark 不可单独覆盖

### 9.2 后续版本扩展方向

**进程级控制（v1.1+）：**
```yaml
annotations:
  etmem.openeuler.io/process-filter: "mysqld,java"
  # 或
  etmem.openeuler.io/match-mode: "include"  # include | exclude | all
```

**内部模型预留：**
- ProcessFilter 已改为 `*ProcessFilter`（nil = 全部进程）
- 后续添加进程名列表只需设置 ProcessFilter.Names
- Agent 的 ResolvePIDs 已支持 processNames 为空（返回全部）和非空（按名过滤）两种模式

**多配置支持（v1.2+）：**
- 同一 namespace 可能需要多个 auto-policy（不同 profile）
- 策略名可扩展为 `etmem-auto-<profile>`
- Agent 已通过 policy selector + Pod matching 支持多策略匹配

## 10. 验证策略

### 10.1 机制验证（必须通过）

| 场景 | 验证内容 | 成功标准 |
|------|---------|---------|
| A: Enable 触发 | Pod label enable=true → auto-policy 创建 → Agent 执行 swap | VmSwap > 0 |
| B: Disable 停止 | 移除 enable label → auto-policy 删除 → 任务停止 | NodeState 清空 |
| C: Pod 删除 | 删除 Pod → 无残留策略 | auto-policy 删除 |
| D: Pod 重建 | 重建 Pod → 自动恢复 | VmSwap 再次增长 |
| E: Profile 选择 | annotation profile=extreme → 使用 extreme 参数 | 配置文件中 swapcache_high_wmark=2 |

### 10.2 效果增强验证（观察性）

| 指标 | 观察方法 | 预期 |
|------|---------|------|
| VmSwap 增长 | `/proc/<PID>/status` | VmSwap > 0 |
| available 增长 | `free -h` | 取决于 swapcache 刷入速度 |
| swapcache 行为 | `/proc/meminfo` SwapCached | 先增长后下降（刷入 swapfile） |

**注意：** available 增长是 swapcache → swapfile 的结果。使用 extreme profile 时效果最明显，但受内核调度影响，不保证立即可观察。

## 三、控制面验证结果（7 个场景）

### 场景 A：添加启用标签 → 自动创建 Policy

**验证目的**：验证 Operator 能否根据 Pod 标签自动创建 EtmemPolicy。

**操作步骤**：

```bash
# 第 1 步：创建不带 etmem 标签的测试 Pod
kubectl run memhog-test --image=polinux/stress \
  --overrides='{"spec":{"nodeSelector":{"kubernetes.io/hostname":"cp0"}}}' \
  -- stress --vm 1 --vm-bytes 512M --vm-hang 0

# 第 2 步：确认无 EtmemPolicy 被创建
kubectl get etmempolicies -A

# 第 3 步：为 Pod 添加启用标签
kubectl label pod memhog-test etmem.openeuler.io/enable=true
```

**观测结果**：

1. 无标签时，`kubectl get etmempolicies -A` 返回空结果（无资源）。
2. 添加标签后，Operator 立即创建了名为 `etmem-auto` 的 EtmemPolicy：

```
NAMESPACE   NAME         PROFILE      AGE
default     etmem-auto   aggressive   3s
```

3. Agent 匹配到 `memhog-test` Pod，解析出 2 个目标 PID，生成配置文件：

```
配置文件路径：/var/run/etmem/configs/default-memhog-test.conf
```

4. NodeState 资源更新，包含任务信息。

**判定**：✅ **通过**

**通过标准**：无标签时不创建 Policy；添加标签后自动创建 Policy，profile 为默认值 aggressive，Agent 正确发现 Pod 并生成配置。

2. 添加 `extreme` 注解后，Policy 的 profile 字段更新为 `extreme`。

| moderate | 1 | 1 | 1 | 50 | 10 | 6 | 1 | 1 | no |
| aggressive | 1 | 1 | 1 | 30 | 5 | 3 | 1 | 2 | no |
| extreme | 1 | 1 | 1 | 20 | 2 | 1 | 1 | 2 | no |

**判定**：✅ **通过**

**通过标准**：通过 annotation 切换 profile 后，Policy 的 profile 字段和对应参数实时更新。

### 4.6 排查过程中发现的根因与修复

在数据面验证过程中，发现并修复了以下三个问题：

#### 根因 1：sysmem_threshold 语义问题

| 项目 | 说明 |
|------|------|
| **问题** | `sysmem_threshold=N` 含义为"当系统可用内存低于 N% 时触发换出"。在 1TB 内存的机器上，可用内存高达 82%，旧 profile 的阈值（conservative=70, moderate=50, aggressive=30, extreme=20）永远无法触发 |
| **现象** | etmemd 扫描目标 PID 但从不执行换出，VmSwap 始终为 0 |
| **修复** | 调整所有 profile 的 sysmem_threshold：conservative=85, moderate=90, aggressive=95, extreme=99 |
| **效果** | 在 82% 可用内存的环境下，aggressive（95%）和 extreme（99%）profile 均可正常触发换出 |

#### 根因 2：etmemd INI 解析器限制

| 项目 | 说明 |
|------|------|
| **问题** | etmemd 的 INI 解析器在遇到多个 `[task]` 段时，只加载最后一个。Agent 之前为每个容器（包括 pause 容器）都生成一个 `[task]` 段 |
| **现象** | Agent 生成 2 个 task（pause + app），etmemd 只加载了 pause 容器的 task，实际应用进程未被管理 |
| **修复** | Agent 过滤基础设施容器（名称包含 pause 或 sandbox 的容器），确保只为应用容器生成 `[task]` 段 |
| **效果** | 配置文件中仅包含单个 `[task]` 段，etmemd 正确加载并管理目标应用进程 |

#### 根因 3：宿主机残留进程干扰

| 项目 | 说明 |
|------|------|
| **问题** | `type=name` 匹配方式会匹配宿主机上所有同名进程。旧的调试过程中遗留的孤儿进程与目标进程同名 |
| **现象** | etmemd 同时管理了目标进程和宿主机上的残留进程，行为不可预测 |
| **解决** | 清理宿主机上的残留同名进程，确保 `type=name` 仅匹配目标容器内的进程 |
| **效果** | etmemd 精确管理目标 Pod 内的进程，VmSwap 正常增长 |

| 3 | **宿主机同名进程**：type=name 全局匹配，残留/其他 Pod 的同名进程干扰 | etmemd 扫描到错误 PID | 中 |

### 7.3 为什么数据面现在能工作

修复后的完整链路：

1. **Operator** 检测到 Pod label `etmem.openeuler.io/enable=true`，自动创建 EtmemPolicy（profile=aggressive）
2. **Agent** 匹配 Pod，通过 cgroup v1 路径解析容器 PID
3. **Agent** 过滤掉 pause 容器，只保留业务容器（memhog），生成**单 [task]** 配置
4. **Agent** 设置 `sysmem_threshold=95`，在 82% 可用内存的机器上满足触发条件（82 < 95）
5. **etmemd** 加载配置，启动项目，扫描 memhog 进程冷页
6. **etmemd** 将冷页通过 etmem_swap 内核模块换出到 swap 设备
7. **结果**：VmSwap 0 → 272MB，VmRSS 270MB → 1.5MB

### 7.4 复现步骤（精简版）

```bash
# 前提：FusionOS/openEuler + etmem 内核模块 + etmemd + swap + cgroup v1

# 1. 启动干净的 etmemd

# 3. 部署带 enable label + aggressive profile 的测试 Pod
```

### 7.5 最小必要条件

| 条件 | 为什么必要 |
|------|-----------|
| etmem_scan + etmem_swap 内核模块 | 提供冷页扫描和 swap 能力 |
| etmemd 守护进程运行中 | 执行实际的内存扫描和换出 |
| swap 设备已启用 | 冷页的目的地 |
| vm.swappiness >= 60 | 内核愿意使用 swap |
| sysmem_threshold > 当前可用% | 触发条件（aggressive=95 适用于大多数服务器） |
| 单 [task] 配置 | etmemd INI 解析器限制 |
| 无同名进程干扰 | type=name 全局匹配 |
| swap_flag=no | FusionOS 5.10 内核兼容性 |
| 配置文件权限 600/400 | etmemd 强制安全要求 |

## 多进程数据面验证结果

> **验证日期**：2026-04-17
> **Agent 版本**：v0.3.0-mp（多进程支持）
> **验证人**：etmem-operator 开发团队
> **文档性质**：实际集群验证结果记录，非模拟数据

### 测试背景

在 v0.2.0 中，Agent 对每个 Pod 仅管理一个进程（代码中 `processes[:1]` 截断）。v0.3.0-mp 升级后，Agent 为 Pod 中的每个用户进程创建独立的 etmemd 项目和配置文件，实现真正的多进程数据面管理。

### 测试环境

| 项目 | 详情 |
|------|------|
| 操作系统 | FusionOS 23（openEuler 衍生发行版） |
| 内核版本 | 5.10.0-278.0.0.181.u183.fos23.x86_64 |
| 物理内存 | 1TB RAM |
| Swap | 16GB LVM swap 分区 |
| cgroup | v1 |
| Agent 版本 | v0.3.0-mp |

### 测试设置

部署两个测试 Pod：

1. **多进程 Pod**（`multiproc-test`）：包含 2 个内存消费进程（memeat1 + memeat2），各分配约 150MB 内存
2. **单进程基线 Pod**（`memhog-test`）：包含 1 个 memhog 进程，分配约 256MB 内存

两个 Pod 均打上 `etmem.openeuler.io/enable: "true"` 标签，使用 `aggressive` profile。

### 配置文件结构

每个配置文件严格包含 **1 个 `[project]`、1 个 `[engine]`、1 个 `[task]`**，无多 `[task]` 回归。

示例（`default-multiproc-test-memeat1.conf`）：

```ini
[project]
name=default-multiproc-test-memeat1
scan_type=page
loop=1
interval=1
sleep=1
sysmem_threshold=95
swapcache_high_wmark=5
swapcache_low_wmark=3

[engine]
name=slide
project=default-multiproc-test-memeat1

[task]
project=default-multiproc-test-memeat1
engine=slide
name=default-multiproc-test-memeat1_task_0
type=name
value=memeat1
T=1
max_threads=2
swap_flag=no
```

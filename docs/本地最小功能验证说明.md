# etmem-operator v0.2.0 本地最小功能验证说明

> **版本**：v0.2.0（Pod-driven 架构）
> **文档性质**：验证方法论说明——解释我们验证了什么、为什么这样验证、每个场景证明了什么

---

## 一、什么是"本地最小功能验证"？

### 1.1 定义

"本地最小功能验证"是指：在一个**真实但受限的本地 Kubernetes 集群**上，验证 etmem-operator 的**核心控制逻辑**是否按设计工作。

用更通俗的话说：

> 我们在一台真机器上搭了一个小集群，然后一步步操作，看 etmem-operator 的"大脑"（控制面）
> 是不是能正确地自动管理资源。我们**不要求**内存真的被换出去——那是"手脚"（数据面）的事情。

### 1.2 为什么叫"最小"？

因为我们的验证环境有约束：

| 约束 | 说明 |
|------|------|
| 节点数量 | 只有 2 个节点，且实际只用了 1 个（cp0） |
| 镜像分发 | wk1 节点无法接收镜像，无法测试多节点调度 |
| 数据面 | 宿主机 swap 配置有问题，etmem 无法实际换出内存页 |
| 负载类型 | 只用了简单的 stress 工具模拟内存负载 |

所以我们验证的是**最小但完整**的控制面功能集——覆盖了所有设计的用户交互场景，但不包含大规模、多节点、真实生产负载的验证。

### 1.3 验证边界

明确列出本次验证**覆盖**和**不覆盖**的范围：

| ✅ 覆盖 | ❌ 不覆盖 |
|----------|-----------|
| Pod 标签触发 Policy 创建 | 多 namespace 同时启用 |
| Pod 标签移除触发 Policy 删除 | 大量 Pod（100+）并发启用 |
| Pod 删除触发清理 | 多节点 Agent 协同 |
| Pod 重建触发恢复 | 真实的内存页换出（数据面） |
| Profile 动态切换 | 长时间运行稳定性（持续数天） |
| Operator 重启恢复 | 网络分区、节点故障等异常场景 |
| Agent 重启恢复 | 与其他 Operator 的兼容性 |

## 二、控制面 vs. 数据面——一个关键区别

这是理解本次验证最重要的概念。

### 2.1 什么是控制面（Control Plane）？

控制面是 etmem-operator 的"大脑"——负责**决策和资源管理**：

- 监听 Pod 标签变化
- 自动创建/删除 EtmemPolicy CRD
- 根据 annotation 选择 profile
- 生成 etmem 配置文件
- 恢复中断的状态

**类比**：控制面就像交通指挥中心，它决定哪条路该通、哪条路该封，但它自己不开车。

### 2.2 什么是数据面（Data Plane）？

数据面是 etmem 的"手脚"——负责**实际执行内存操作**：

- etmemd 守护进程扫描进程内存页
- 根据策略将冷内存页换出到 swap
- 进程的 VmSwap 值增加

**类比**：数据面就像实际在路上开的车，真正完成"运送"任务。

### 2.3 为什么要分开验证？

| 原因 | 解释 |
|------|------|
| **依赖不同** | 控制面只依赖 Kubernetes API；数据面依赖内核模块、etmemd、swap 配置 |
| **故障隔离** | 控制面有 bug ≠ 数据面有问题，反之亦然 |
| **可独立验证** | 即使 swap 不工作，我们也能完整验证控制面逻辑 |
| **价值分层** | 控制面验证证明"Operator 代码正确"；数据面验证证明"整个系统端到端工作" |

### 2.4 本次验证的定位

```
控制面验证 ✅ → 证明 Operator/Agent 代码逻辑正确
数据面验证 ❌ → 宿主机配置问题，待修复后补充
```

> **重要**：控制面通过 + 数据面未通过 = Operator 代码没问题，但还不能交给用户用于生产。
> 这就像：导航软件路线规划是对的，但车还没加油，暂时开不动。

## 三、每个验证场景的目的与意义

### 场景 A：添加启用标签 → 自动创建 Policy

#### 为什么要测试这个？

这是 v0.2.0 最核心的设计：用户只需要给 Pod 加一个标签，Operator 就自动完成所有事情。如果这个不工作，整个 Pod-driven 架构就失去意义。

#### 这个场景证明了什么？

| ✅ 证明了 | ❌ 没有证明 |
|-----------|------------|
| Operator 能监听 Pod 标签变化 | 大量 Pod 同时添加标签时的性能 |
| 收到标签后能自动创建 EtmemPolicy | 创建的 Policy 能被 etmemd 正确执行 |
| 默认 profile 为 aggressive | 多 namespace 下的 Policy 隔离 |
| Agent 能发现目标 Pod 并解析 PID | PID 解析在所有容器运行时下都正确 |
| Agent 能生成正确的配置文件 | 配置文件能被 etmemd 正确加载 |

#### 成功标准

1. 无标签时：`kubectl get etmempolicies -A` 返回空
2. 添加标签后：出现 `etmem-auto` Policy，profile 为 `aggressive`
3. Agent 日志显示匹配到 Pod 和 PID
4. 配置文件生成在 `/var/run/etmem/configs/` 目录下

#### 操作方法

```bash
# 创建不带标签的 Pod
kubectl run memhog-test --image=polinux/stress \
  --overrides='{"spec":{"nodeSelector":{"kubernetes.io/hostname":"cp0"}}}' \
  -- stress --vm 1 --vm-bytes 512M --vm-hang 0

# 确认无 Policy
kubectl get etmempolicies -A

# 添加标签
kubectl label pod memhog-test etmem.openeuler.io/enable=true

# 确认 Policy 创建
kubectl get etmempolicies -A
```

---

### 场景 B：移除标签 → 自动删除 Policy

#### 为什么要测试这个？

自动创建只是一半——自动**清理**同样重要。如果用户取消标签后 Policy 还留着，就会有"孤儿资源"问题，这正是 v0.1.0 的痛点之一。

#### 这个场景证明了什么？

| ✅ 证明了 | ❌ 没有证明 |
|-----------|------------|
| 移除标签后 Policy 自动删除 | 频繁添加/移除标签时的竞态安全 |
| Agent 清理了配置文件 | 配置清理在磁盘 I/O 异常时是否安全 |
| 无孤儿资源残留 | 跨 namespace 清理的正确性 |

#### 成功标准

1. 移除标签后：`etmem-auto` Policy 消失
2. Agent 日志显示 `policies=0`
3. `/var/run/etmem/configs/` 目录为空

#### 操作方法

```bash
# 移除标签（注意末尾的减号 - 表示删除）
kubectl label pod memhog-test etmem.openeuler.io/enable-

# 确认 Policy 已删除
kubectl get etmempolicies -A

# 确认配置文件已清理
ls /var/run/etmem/configs/
```

---

### 场景 C：删除 Pod → 自动清理

#### 为什么要测试这个？

Pod 会因为各种原因被删除（手动删除、Deployment 缩容、节点故障驱逐……）。Operator 必须能感知到 Pod 消失并清理关联资源。

#### 这个场景证明了什么？

| ✅ 证明了 | ❌ 没有证明 |
|-----------|------------|
| Pod 删除后 Policy 自动清理 | Pod 被强制删除（--force --grace-period=0）时的行为 |
| Operator 日志记录清理操作 | 节点故障导致 Pod 丢失时的清理 |
| 配置文件被清理 | Finalizer 机制的鲁棒性 |

#### 成功标准

1. 删除 Pod 后：`etmem-auto` Policy 消失
2. Operator 日志输出 `"No enabled Pods, auto-policy cleaned up"`
3. 配置文件被清理

#### 操作方法

```bash
# 先确保有启用标签的 Pod 和对应 Policy
kubectl label pod memhog-test etmem.openeuler.io/enable=true
kubectl get etmempolicies -A

# 删除 Pod
kubectl delete pod memhog-test

# 确认清理
kubectl get etmempolicies -A
```

### 场景 D：Pod 重建 → 自动恢复

#### 为什么要测试这个？

在 Kubernetes 中，Pod 经常被删除重建（滚动更新、重调度等）。如果 Operator 不能在新 Pod 出现后恢复 Policy，用户就需要手动重新操作——这违背了自动化的初衷。

#### 这个场景证明了什么？

| ✅ 证明了 | ❌ 没有证明 |
|-----------|------------|
| 删除后重建 Pod 可无缝恢复 Policy | Deployment 滚动更新时的无缝切换 |
| 不会创建重复的 Policy | StatefulSet 等有状态工作负载的恢复 |
| Operator 日志正确记录恢复 | 大规模 Pod 同时重建时的性能 |

#### 成功标准

1. 删除旧 Pod 后 Policy 消失
2. 创建新 Pod（带标签）后 Policy 自动恢复
3. 不出现重复的 `etmem-auto` Policy
4. Operator 日志输出 `"Auto-policy ensured", enabledPods=1`

#### 操作方法

```bash
# 创建带标签的 Pod
kubectl run memhog-test --image=polinux/stress \
  -l etmem.openeuler.io/enable=true \
  --overrides='{"spec":{"nodeSelector":{"kubernetes.io/hostname":"cp0"}}}' \
  -- stress --vm 1 --vm-bytes 512M --vm-hang 0

# 确认 Policy
kubectl get etmempolicies -A

# 删除 Pod
kubectl delete pod memhog-test

# 重建 Pod（相同名称和标签）
kubectl run memhog-test --image=polinux/stress \
  -l etmem.openeuler.io/enable=true \
  --overrides='{"spec":{"nodeSelector":{"kubernetes.io/hostname":"cp0"}}}' \
  -- stress --vm 1 --vm-bytes 512M --vm-hang 0

# 确认 Policy 恢复
kubectl get etmempolicies -A
```

### 场景 E：Profile 注解切换

#### 为什么要测试这个？

不同的业务负载需要不同强度的内存分层策略。用户需要能够**动态调整**策略，而不是删了重建。annotation 机制让用户可以随时切换。

#### 这个场景证明了什么？

| ✅ 证明了 | ❌ 没有证明 |
|-----------|------------|
| annotation 能动态更新 Policy 的 profile | profile 切换后 etmemd 行为的实际变化 |
| 支持 4 种 profile 切换 | 自定义 profile（非预定义）的处理 |
| Operator 实时响应 annotation 变化 | 高频切换时的稳定性 |

#### 4 种 Profile 的参数含义

给不熟悉 etmem 的读者解释：

| 参数 | 含义 |
|------|------|
| loop | 扫描循环次数 |
| interval | 扫描间隔（秒） |
| sleep | 扫描后休眠时间（秒） |
| sysMem% | 系统内存使用率阈值——超过此值才开始换出 |
| highWmark | 高水位线（百分比）——达到时开始换出 |
| lowWmark | 低水位线（百分比）——降到此值时停止换出 |
| T | 页面冷热判定阈值 |
| maxThreads | 最大并发线程数 |
| swapFlag | 是否强制换出 |

**简单理解**：
- `conservative`（保守）：扫描慢、阈值高，只在内存非常紧张时才动作
- `moderate`（适中）：中等力度
- `aggressive`（激进）：扫描快、阈值低，更积极地换出冷内存
- `extreme`（极端）：最激进，最大力度换出

#### 成功标准

1. 默认 profile 为 `aggressive`
2. 添加 `extreme` 注解后 Policy 更新为 `extreme`
3. 覆盖为 `conservative` 后 Policy 更新为 `conservative`
4. 每次变更 Operator 日志都有记录

#### 操作方法

```bash
# 确认默认 profile
kubectl get etmempolicy etmem-auto -o jsonpath='{.spec.profile}'

# 切换到 extreme
kubectl annotate pod memhog-test etmem.openeuler.io/profile=extreme

# 确认切换
kubectl get etmempolicy etmem-auto -o jsonpath='{.spec.profile}'

# 切换到 conservative
kubectl annotate pod memhog-test etmem.openeuler.io/profile=conservative --overwrite

# 确认切换
kubectl get etmempolicy etmem-auto -o jsonpath='{.spec.profile}'
```

### 场景 F：Operator 重启恢复

#### 为什么要测试这个？

在生产环境中，Operator 会因为升级、节点重启、OOM 等原因重启。重启后必须能正确恢复状态，否则：
- 已有的 Policy 可能丢失
- 可能创建重复的 Policy
- 可能和当前集群状态不同步

#### 这个场景证明了什么？

| ✅ 证明了 | ❌ 没有证明 |
|-----------|------------|
| Operator 重启后能恢复已有 Policy | 长时间运行后的内存泄漏 |
| 不会创建重复 Policy | Operator 和 API Server 断连后的恢复 |
| 保留之前的 profile 设置 | 多个 Operator 副本的 Leader Election |

#### 成功标准

1. 重启后 Operator 日志显示 `"Auto-policy ensured"`
2. `etmem-auto` Policy 保持不变（不是新创建的）
3. profile 仍为重启前的值

#### 操作方法

```bash
# 重启 Operator
kubectl -n etmem-system rollout restart deploy/etmem-operator

# 等待新 Pod 就绪
kubectl -n etmem-system rollout status deploy/etmem-operator

# 查看日志
kubectl -n etmem-system logs deploy/etmem-operator --tail=20

# 确认 Policy 仍存在
kubectl get etmempolicies -A
```

### 场景 G：Agent 重启恢复

#### 为什么要测试这个？

Agent 以 DaemonSet 运行在每个节点上，同样会面临重启。Agent 重启后需要：
- 重新发现节点上的目标 Pod
- 重新解析 PID
- 重新生成配置文件
- 重新尝试执行 etmem 任务

#### 这个场景证明了什么？

| ✅ 证明了 | ❌ 没有证明 |
|-----------|------------|
| Agent 重启后能重新发现 Pod | Agent 在高负载下的恢复速度 |
| 能重新解析 PID | PID 变化时的无缝切换 |
| 控制面恢复逻辑正确 | 与 etmemd 的长连接稳定性 |

#### 成功标准

1. 新 Agent Pod 启动后发现 `memhog-test` Pod
2. 正确解析出目标 PID
3. 尝试恢复 etmem 任务（即使因 etmemd 连接问题失败，Agent 自身逻辑正确即算通过）

#### 操作方法

```bash
# 删除 Agent Pod
kubectl -n etmem-system delete pod -l app=etmem-agent

# 等待新 Pod 启动
kubectl -n etmem-system get pods -l app=etmem-agent -w

# 查看日志确认恢复行为
kubectl -n etmem-system logs -l app=etmem-agent --tail=30
```

## 四、CRD 标签与注解参考

本节面向不熟悉 Kubernetes 标签/注解体系的读者。

### 4.1 什么是标签（Label）？

标签是附加到 Kubernetes 资源上的键值对，用于标识和选择资源。etmem-operator 用标签来判断"这个 Pod 是否需要启用 etmem"。

```
键：etmem.openeuler.io/enable
值："true" 表示启用；其他值或不存在表示不启用
```

### 4.2 什么是注解（Annotation）？

注解也是键值对，但用于附加**非标识性**的元数据。etmem-operator 用注解来指定策略参数。

```
键：etmem.openeuler.io/profile
值：conservative | moderate | aggressive | extreme
默认值：aggressive（不设置注解时的默认行为）
```

### 4.3 操作示例

```bash
# 启用 etmem（添加标签）
kubectl label pod <pod-name> etmem.openeuler.io/enable=true

# 禁用 etmem（移除标签）
kubectl label pod <pod-name> etmem.openeuler.io/enable-

# 设置 profile（添加注解）
kubectl annotate pod <pod-name> etmem.openeuler.io/profile=conservative

# 修改 profile（覆盖注解）
kubectl annotate pod <pod-name> etmem.openeuler.io/profile=extreme --overwrite
```

## 五、如果验证失败怎么办？

本节提供排障指南，帮助你在自己的环境上复现验证时排查问题。

### 5.1 场景 A 失败：添加标签后没有 Policy 创建

**可能原因与排查步骤**：

| 排查项 | 命令 | 期望结果 |
|--------|------|----------|
| Operator 是否在运行？ | `kubectl -n etmem-system get pods` | Operator Pod 状态为 Running |
| Operator 日志有无错误？ | `kubectl -n etmem-system logs deploy/etmem-operator` | 无 panic/crash 日志 |
| CRD 是否已安装？ | `kubectl get crd etmempolicies.etmem.openeuler.io` | CRD 存在 |
| RBAC 是否正确？ | `kubectl auth can-i create etmempolicies -n default --as=system:serviceaccount:etmem-system:etmem-operator` | yes |
| 标签拼写是否正确？ | `kubectl get pod <name> --show-labels` | 包含 `etmem.openeuler.io/enable=true` |

### 5.2 场景 B/C 失败：资源未清理

**可能原因与排查步骤**：

| 排查项 | 命令 | 期望结果 |
|--------|------|----------|
| Finalizer 是否存在？ | `kubectl get etmempolicy etmem-auto -o yaml` | 检查 finalizers 字段 |
| Operator 是否收到删除事件？ | 查看 Operator 日志 | 应有 reconcile 日志 |
| 是否有其他 Pod 仍有标签？ | `kubectl get pods -l etmem.openeuler.io/enable=true` | 应为空 |

### 5.3 场景 D 失败：Pod 重建后 Policy 未恢复

**可能原因与排查步骤**：

| 排查项 | 说明 |
|--------|------|
| 新 Pod 是否带有正确标签？ | 确认 `kubectl get pod <name> --show-labels` |
| Operator 是否观测到新 Pod？ | 查看 Operator 日志中的 reconcile 事件 |
| 是否存在命名冲突？ | 确认旧 Pod 已完全删除（Terminating 状态也算未删除） |

### 5.4 场景 E 失败：Profile 未更新

**可能原因与排查步骤**：

| 排查项 | 命令 |
|--------|------|
| annotation 是否正确？ | `kubectl get pod <name> -o jsonpath='{.metadata.annotations}'` |
| 值是否为合法 profile？ | 只支持 conservative、moderate、aggressive、extreme |
| 是否使用了 --overwrite？ | 修改已存在的 annotation 需要 `--overwrite` 参数 |

### 5.5 场景 F/G 失败：重启后未恢复

**可能原因与排查步骤**：

| 排查项 | 说明 |
|--------|------|
| 新 Pod 是否成功启动？ | 检查 Pod 状态和事件 `kubectl describe pod ...` |
| 镜像是否能拉取？ | `imagePullPolicy` 设置和镜像是否在节点上 |
| ServiceAccount 和 RBAC？ | 确认权限未丢失 |
| API Server 是否可达？ | 新 Pod 日志中是否有连接错误 |

### 5.6 通用排障技巧

```bash
# 查看 Operator 实时日志
kubectl -n etmem-system logs -f deploy/etmem-operator

# 查看 Agent 实时日志
kubectl -n etmem-system logs -f -l app=etmem-agent

# 查看所有 etmem 相关资源
kubectl get etmempolicies -A
kubectl get etmemnodestates -A

# 查看 Pod 事件
kubectl describe pod <pod-name>

# 查看配置文件目录
ls -la /var/run/etmem/configs/

# 检查 etmemd 是否运行
systemctl status etmemd
# 或
ps aux | grep etmemd
```

## 六、关于数据面验证的补充说明

### 6.1 为什么数据面没有工作？

简单回答：**宿主机的内核参数配置不正确**。

详细解释：

1. **swappiness=0**：Linux 内核的 `swappiness` 参数控制内核使用 swap 的倾向。值为 0 意味着内核**尽可能不使用 swap**。
2. **swap_extension=0**：etmem 的 swap 扩展功能需要此内核参数启用。
3. **内核模块 refcount=0**：虽然 etmem_swap 和 etmem_scan 模块已加载，但没有活跃引用。

### 6.2 数据面问题和 Operator 代码有关系吗？

**没有关系**。证据如下：

- etmemd **确实在扫描**目标进程的内存页（日志可见）
- Agent **确实生成了**正确格式的配置文件
- Operator **确实创建了**包含正确参数的 Policy
- 在 v0.1.0 阶段，**相同的系统**上成功实现了 229MB 的内存换出

问题在于 etmemd → 内核模块 → swap 分区这条链路的**宿主机配置**，不在 Operator/Agent 的控制范围内。

### 6.3 数据面什么时候能验证？

需要满足以下条件：

1. 正确配置 `swappiness`（建议 ≥ 60）
2. 启用 `swap_extension`（值设为 1）
3. 确认 swap 分区可用（`swapon -s` 应显示活跃 swap）
4. 重启 etmemd 守护进程
5. 可能还需要排查其他内核参数

## 跨 Pod 隔离验证（type=pid 作用域）

### 背景

etmem-operator v0.4.0-pid 将进程匹配方式从 `type=name`（节点级 pgrep）改为 `type=pid`（PID 精确定位）。本场景验证同一节点上多个同名进程时，仅启用标签的 Pod 中的进程被换出。

### 为什么要测试这个？

`type=name` 使用 `pgrep -x <name>` 匹配进程，会命中节点上所有同名进程。在多租户集群中，这可能导致未启用 etmem 的 Pod 进程被误换出。`type=pid` 通过 cgroup 精确定位 Pod 内的 PID，彻底消除跨 Pod 干扰。

### 操作方法

#### 步骤 1：创建已启用的 Pod

```bash
# 创建启用 etmem 的 Pod（带标签）
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: memhog-enabled
  namespace: etmem-test
  labels:
    etmem.openeuler.io/enable: "true"
  annotations:
    etmem.openeuler.io/process-name: "memhog"
    etmem.openeuler.io/profile: "default"
spec:
  containers:
  - name: memhog
    image: <your-memhog-image>
    command: ["memhog", "256M"]
EOF
```

#### 步骤 2：创建未启用的同名进程 Pod

```bash
# 创建不带启用标签的 Pod（进程名相同）
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: memhog-noenable
  namespace: etmem-test
spec:
  containers:
  - name: memhog
    image: <your-memhog-image>
    command: ["memhog", "256M"]
EOF
```

确保两个 Pod 调度到同一节点（可使用 `nodeSelector` 或 `nodeName`）。

#### 步骤 3：等待换出生效并检查

```bash
# 等待 Agent reconcile 完成（约 30-60 秒）
sleep 60

# 获取两个 Pod 中 memhog 进程的 PID
ENABLED_PID=$(kubectl exec -n etmem-test memhog-enabled -- pgrep -x memhog)
NOENABLE_PID=$(kubectl exec -n etmem-test memhog-noenable -- pgrep -x memhog)

# 在节点上检查 VmSwap（需要 SSH 到节点或通过 Agent Pod 执行）
# 启用的 Pod 进程
cat /proc/${ENABLED_PID}/status | grep -E "VmSwap|VmRSS"
# 预期：VmSwap >> 0，VmRSS 很小

# 未启用的 Pod 进程
cat /proc/${NOENABLE_PID}/status | grep -E "VmSwap|VmRSS"
# 预期：VmSwap: 0 kB，VmRSS ≈ 256 MB
```

#### 步骤 4：检查 Agent 日志

```bash
# 查看 Agent 日志中的项目创建记录
kubectl -n etmem-system logs ds/etmem-agent --tail=50 | grep -E "project|pid"

# 预期输出应包含：
#   项目名包含 "-p{PID}" 后缀（如 etmem-test-memhog-enabled-memhog-p12345）
#   type=pid, value=<ENABLED_PID>
# 预期输出不应包含：
#   memhog-noenable 的 PID
#   type=name 的配置
```

### 成功标准

| 检查项 | 预期结果 |
|---|---|
| 启用 Pod 的 VmSwap | > 0（冷页被换出） |
| 未启用 Pod 的 VmSwap | = 0（未被干扰） |
| Agent 日志中项目名 | 包含 `-p{pid}` 后缀 |
| Agent 日志中配置 | 使用 `type=pid`，不含 `type=name` |
| Agent 日志中 PID | 仅包含启用 Pod 的 PID |

### 这个场景证明了什么？

- `type=pid` 精确定位了目标进程，不会误匹配同名进程
- PID 来源于 Pod cgroup 路径，天然限定了 Pod 作用域
- 在多租户场景下，未启用 etmem 的 Pod 不会受到任何影响
- VmSwap=0 证明未启用 Pod 的进程**完全没有被 etmemd 处理**

## 使用现有工作负载进行最小验证

除了创建专用测试 Pod，也可以直接利用集群中**已有的高负载工作负载**进行最小功能验证。

### 适用场景

- 集群中已经运行着内存占用较大的服务（如 DaemonSet、Deployment、StatefulSet）
- 不想额外构建测试镜像
- 希望在更真实的场景下验证 etmem 的控制面和数据面

### 基本原理

etmem 的策略选择器为 `etmem.openeuler.io/enable: "true"`。只要给工作负载的 Pod 模板添加此标签，Agent 就会自动发现并管理对应的 Pod 进程。

### 验证范围

- **控制面**：任何添加了 etmem 标签的 Pod 都可以验证控制面（Agent 匹配、EtmemPolicy 创建、NodeState 更新）
- **数据面**：需要环境支持 etmem 内核模块，且目标进程有足够的内存占用（建议 >100MB RSS），同时 `sysmem_threshold` 需要满足触发条件

# etmem-operator v1.0 概要设计说明书

**文档版本**：v1.0  
**状态**：正式  
**文档定位**：描述 v1.0 的总体架构、模块分工与部署/恢复/交付边界。  
**适用范围**：用于说明控制面、Agent、数据面和交付链路的系统边界，不展开字段级实现细节。

---

## 已验证事实基线

本文档以如下已在真实目标环境（openEuler/FusionOS + etmemd + swap）中验证通过的事实为架构输入：

1. **控制面闭环已通过**：Pod/工作负载打标签后，Operator 自动创建并收敛 EtmemPolicy；Pod 重建后可重新接管；suspend/rollback 路径已验证。
2. **节点执行面已通过**：Agent 以 DaemonSet 方式运行，基于 cgroup v1 路径完成节点内 PID 解析；NodeState 写回与健康状态观测已验证。
3. **数据面已通过**：真实目标进程出现 `VmSwap > 0`，整机 `Swap used` 增长，证明换出真实发生。
4. **交付链路已通过**：Helm install/upgrade、离线镜像导入、workload patching、验证脚本/NodeState 观测路径均已跑通。
5. **`MemAvailable` 定位**：属于主机级净值辅助观测，受页缓存、内核回收等多因素影响，不作为单次验收硬门槛。不同进程 VmSwap 增长幅度可以不同；wrapper/sleep 类进程 VmSwap=0 属于正常现象。

## 1. 设计目标

v1.0 在已验证 MVP 能力基础上，以工程化和可交付性为核心目标，不推翻现有架构。

### 1.1 核心目标

| 目标 | 说明 |
|------|------|
| 稳定部署 | Operator 以 Deployment 运行，Agent 以 DaemonSet 覆盖所有目标节点，保证控制面与执行面的可用性 |
| 可升级 / 可恢复 | Helm 管理全生命周期；Agent 启动时执行 Bootstrap 读取 NodeState 重建任务状态，首轮 reconcile 完成任务重新注册 |
| 可观测 | NodeState、Agent 日志、验证脚本共同构成主观测面；`MemAvailable` 作为辅助参考 |
| 可复现验证 | 提供标准化验证脚本，验证结果可独立重现 |
| 可离线交付 | 提供镜像 tar 包 + Helm Chart + 导入脚本，支持无外网环境部署 |

### 1.2 非目标（本版本不展开）

- 多租户极限安全隔离
- 高级 slide 参数用户自定义扩展（预留扩展位，不作为主线）
- cgroup v2 支持
- 长期版本演进路线

## 2. 总体架构

etmem-operator v1.0 采用 **Operator + Agent(DaemonSet) + etmemd** 三层架构。

```
┌─────────────────────────────────────────────────────────────┐
│                      Kubernetes 集群                         │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │                   控制面（etmem-system）               │   │
│  │                                                      │   │
│  │  ┌────────────────┐    ┌──────────────────────────┐  │   │
│  │  │  etmem-operator │    │      EtmemPolicy CRD     │  │   │
│  │  │  (Deployment)   │◄──►│   EtmemNodeState CRD     │  │   │
│  │  │                 │    │                          │  │   │
│  │  │  • 监听 Pod 标签  │    └──────────────────────────┘  │   │
│  │  │  • reconcile    │                                   │   │
│  │  │    Policy       │                                   │   │
│  │  └────────────────┘                                   │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  ┌──────────────────────────────────────────────────────┐   │
│  │                  节点执行面（每节点）                    │   │
│  │                                                      │   │
│  │  ┌────────────────┐    ┌───────────────────────────┐ │   │
│  │  │  etmem-agent    │    │  etmemd（节点守护进程）     │ │   │
│  │  │  (DaemonSet)    │───►│  • 内核模块扫描冷页        │ │   │
│  │  │                 │    │  • 换出至 swap 设备        │ │   │
│  │  │  • 监听 Policy   │    └───────────────────────────┘ │   │
│  │  │  • 解析 PID     │                                   │   │
│  │  │  • 下发任务     │    ┌───────────────────────────┐  │   │
│  │  │  • 写回 State   │    │  宿主机 cgroup v1 / swap   │  │   │
│  │  └────────────────┘    └───────────────────────────┘  │   │
│  └──────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

### 2.1 层次职责说明

| 层次 | 组件 | 运行方式 | 职责 |
|------|------|----------|------|
| 控制面 | etmem-operator | Deployment（etmem-system 命名空间） | 监听 Pod 标签变化，自动生成/收敛/删除 EtmemPolicy |
| 节点执行面 | etmem-agent | DaemonSet（部署至所有目标节点） | 在节点侧监听 Policy、解析 PID、下发 etmem 任务、写回 NodeState |
| 数据面 | etmemd + 内核模块 | 节点守护进程（独立于 Kubernetes） | 执行冷页扫描和内存换出，写入 swap 设备 |
| 配置管理 | Helm Chart | 集群外运维工具 | 统一管理 CRD 安装、Operator/Agent 部署、升级和卸载 |

### 2.2 关键分界线

- **Operator 与 Agent 之间**：通过 EtmemPolicy CRD 异步解耦，Operator 写 Policy，Agent 读 Policy。
- **Agent 与 etmemd 之间**：通过节点本地 Unix socket 同步通信，Agent 负责配置组装和命令下发。
- **Kubernetes 与数据面之间**：etmemd 及内核模块不受 Kubernetes 管理，由节点运维预先就绪。

## 3. 逻辑组件划分

### 3.1 etmem-operator（控制面）

**部署形式**：Deployment，副本数为 1，运行于 `etmem-system` 命名空间。

**核心职责**：

- 监听带有 `etmem.openeuler.io/enable=true` 标签的 Pod 事件（创建/更新/删除）。
- 在每个 namespace 内自动维护一个名为 `etmem-auto` 的 EtmemPolicy，整合该 namespace 下所有启用 etmem 的 Pod 需求。
- 采用"最激进档位优先（most-aggressive-wins）"策略合并同一 namespace 内多个 Pod 的 profile。
- 当 namespace 内无启用 Pod 时，自动删除 `etmem-auto` 策略。
- 用户手工创建的 EtmemPolicy 通过标签标识与自动生成策略区分，Operator 不覆盖手工策略。

**不负责的边界**：

- 不直接与 etmemd 通信。
- 不感知节点级执行状态（由 NodeState 体现）。

### 3.2 etmem-agent（节点执行面）

**部署形式**：DaemonSet，部署至所有具备 etmem 能力的目标节点（通过节点标签或 tolerations 控制范围）。

**核心职责**：

- 监听 EtmemPolicy，提取本节点相关的任务配置。
- 通过 cgroup v1 路径（`/sys/fs/cgroup/memory/kubepods/...`）解析目标 Pod 容器的进程 PID。
- 组装 etmem 任务配置并通过 Unix socket 下发给 etmemd。
- 在 EtmemNodeState CRD 中回写本节点的任务状态和 etmemd 通路健康状态。
- 启动时调用 Bootstrap 恢复逻辑，从 NodeState 读取任务状态重建内存记录，首轮 reconcile 完成任务重新注册。

**不负责的边界**：

- 不修改 EtmemPolicy，只读取。
- 不管理 etmemd 进程本身的启停。
- 不直接操作内核模块。

### 3.3 CRD（自定义资源）

| CRD | 写入方 | 读取方 | 核心职责 |
|-----|--------|--------|----------|
| EtmemPolicy | Operator | Agent | 承载内存分层策略配置（目标 Pod 选择、profile 参数等） |
| EtmemNodeState | Agent | Operator / 运维人员 | 记录节点任务状态、etmemd 通路健康、最后更新时间 |

### 3.4 Helm Chart

**职责边界**：

- 安装/升级 CRD。
- 部署 Operator Deployment 和 Agent DaemonSet。
- 管理 RBAC（ServiceAccount、ClusterRole、ClusterRoleBinding）。
- 通过 `values.yaml` 提供镜像版本、资源限制、节点选择器等可配置项。
- **不管理** etmemd 安装及内核模块加载（属于节点运维范围）。

### 3.5 验证脚本

**职责边界**：

- 提供标准化验证步骤，辅助运维人员确认环境就绪和功能正常。
- 检查项包括：NodeState 状态、Agent 日志中的任务执行记录、目标进程 VmSwap、整机 Swap used 变化。
- 脚本属于可交付物的一部分，不属于运行时组件。

## 4. 部署架构

### 4.1 Kubernetes 资源拓扑

```
etmem-system 命名空间
├── Deployment: etmem-operator（副本数 1）
│   └── ServiceAccount: etmem-operator-sa
│       └── ClusterRoleBinding → ClusterRole: etmem-operator-role
│
└── DaemonSet: etmem-agent（每目标节点一个 Pod）
    └── ServiceAccount: etmem-agent-sa
        └── ClusterRoleBinding → ClusterRole: etmem-agent-role

集群范围
├── CRD: etmempolicies.etmem.openeuler.io
└── CRD: etmemnodestates.etmem.openeuler.io

各业务命名空间
└── EtmemPolicy: etmem-auto（由 Operator 自动维护）

集群范围（每节点一个）
└── EtmemNodeState: <node-name>（由 Agent 创建和维护）
```

### 4.2 节点级部署关系

```
节点（openEuler/FusionOS）
├── kubelet
├── containerd
├── etmemd（独立守护进程，节点运维预装）
│   └── Unix socket: /var/run/etmemd.sock
├── 内核模块: etmem_scan + etmem_swap
├── swap 设备（节点运维预配置）
└── etmem-agent Pod（由 DaemonSet 调度）
    ├── hostPath 挂载: /sys/fs/cgroup
    └── hostPath 挂载: /var/run/etmemd.sock
```

### 4.3 Operator 使用 Deployment 的设计依据

Operator 运行于控制面，对节点资源无直接依赖，无需特权访问宿主机，适合以标准 Deployment 形式运行。单副本满足 Leader Election 语义要求，可在需要时扩展为多副本高可用。

### 4.4 Agent 使用 DaemonSet 的设计依据

Agent 需要访问每个目标节点的 cgroup 文件系统和 etmemd socket，与节点强绑定，必须以 DaemonSet 形式部署以保证每节点一个 Pod 的覆盖关系。Agent Pod 需挂载宿主机路径并以特权模式运行。

## 5. 关键业务流程

### 5.1 工作负载纳管流程

```
用户对 Pod 打标签（etmem.openeuler.io/enable=true）
    │
    ▼
Operator（PodReconciler）
    ├── 获取 namespace 内所有启用 etmem 的 Pod
    ├── 合并 profile（most-aggressive-wins）
    └── 创建/更新 EtmemPolicy（etmem-auto）
            │
            ▼
        Agent（PolicyReconciler，每目标节点）
            ├── 监听到 Policy 变更
            ├── 通过 cgroup v1 路径解析目标 Pod 容器 PID
            ├── 组装 etmem 任务配置
            ├── 通过 Unix socket 下发任务至 etmemd
            └── 更新 EtmemNodeState（任务状态 + 节点健康）
                    │
                    ▼
                etmemd（节点守护进程）
                    ├── 接收任务配置
                    ├── 调用内核模块扫描目标进程冷页
                    └── 将冷页换出至 swap 设备
```

### 5.2 工作负载退出流程

```
Pod 被删除 / 标签被移除
    │
    ▼
Operator 重新 reconcile 该 namespace
    ├── 若 namespace 内无剩余启用 Pod → 删除 etmem-auto Policy
    └── 若仍有其他启用 Pod → 更新 etmem-auto Policy（移除已删除 Pod）
            │
            ▼
        Agent 监听到 Policy 变更
            ├── 向 etmemd 下发 remove task 命令
            └── 更新 EtmemNodeState
```

### 5.3 Agent 启动恢复流程

```
Agent Pod 启动（节点重启 / DaemonSet 滚动更新）
    │
    ▼
Bootstrap 恢复阶段
    ├── 读取本节点 EtmemNodeState
    └── 重建内存中的运行任务状态（不重新向 etmemd 下发）
            │
            ▼
正常 reconcile 循环
    └── 与 Policy 对账，补齐差量或清理多余任务
```

### 5.4 异常处理原则

- **etmemd 不可达**：Agent 标记节点 etmemd 通路为 Unhealthy，写入 NodeState，定期重试。
- **PID 解析失败**：跳过该进程，记录日志，不影响其他进程任务。
- **Policy 冲突**：以 Operator 最新 reconcile 结果为准，Agent 以 Policy 为单一事实来源。
- **任务幂等**：Agent 重复下发同一任务视为幂等操作，不产生副作用。

## 6. 状态与数据模型概览

### 6.1 EtmemPolicy（策略资源）

EtmemPolicy 是控制面和执行面之间的核心契约，承载以下核心信息：

- **目标选择**：指定哪些 Pod（通过 namespace + name 或 label selector）需要进行内存分层。
- **策略参数**：profile 档位（conservative / moderate / aggressive / extreme），映射为 slide 扫描参数。
- **状态**：当前策略是否处于激活（active）、暂停（suspended）或降级状态。

### 6.2 EtmemNodeState（节点状态资源）

EtmemNodeState 是节点执行面向控制面和运维人员暴露状态的唯一渠道，每个节点对应一个实例，记录以下信息：

- **etmemd 通路健康状态**：Agent 能否正常与 etmemd 通信。
- **运行中任务列表**：当前已下发至 etmemd 的任务及其状态（运行中 / 失败 / 待重试）。
- **最后同步时间**：Agent 最后一次成功与 etmemd 对账的时间戳。

### 6.3 状态流转概览

```
Policy 状态:  Pending → Active → Suspended / Deleted

NodeState 任务状态:  Pending → Running → Failed → (Retry → Running)
                                        → Removed
```

### 6.4 数据面核心指标（验证基准）

| 指标 | 来源 | 意义 | 说明 |
|------|------|------|------|
| `VmSwap > 0` | 目标进程 `/proc/<pid>/status` | 进程冷页已换出 | 主要验证指标 |
| `Swap used` 增长 | 整机 `/proc/meminfo` | 整机换出已发生 | 主要验证指标 |
| `MemAvailable` 变化 | 整机 `/proc/meminfo` | 可用内存变化参考 | **辅助观测，非硬门槛** |
| NodeState 健康 | EtmemNodeState | Agent 与 etmemd 通路正常 | 观测面之一 |
| Agent 日志 | Agent Pod 日志 | 任务下发与执行记录 | 观测面之一 |

## 7. 可观测性与恢复机制

### 7.1 主观测面

v1.0 的主观测面由三个维度共同构成，缺少任何一个维度都不能完整判断系统状态：

| 维度 | 观测手段 | 覆盖内容 |
|------|----------|----------|
| NodeState | `kubectl get etmemnodestates -A` | 每节点 etmemd 通路健康、任务运行状态 |
| Agent 日志 | `kubectl logs -n etmem-system <agent-pod>` | PID 解析结果、任务下发记录、异常告警 |
| 验证脚本 | `scripts/verify-etmem.sh <pod-name>` | VmSwap、Swap used、通路连通性综合检查 |

### 7.2 辅助观测项

`MemAvailable` 属于主机级净值，反映整机可用内存，受页缓存回收、其他进程内存活动影响，**不能作为单次验收的硬门槛**。验证时应以 `VmSwap > 0` 和 `Swap used` 增长为主要判据，`MemAvailable` 变化作为辅助参考。

### 7.3 Operator 恢复机制

- Operator 以 Deployment 运行，由 Kubernetes 控制面保障其可用性。
- Operator 重启后，controller-manager 框架会自动触发所有 Policy 的 reconcile，恢复正常工作状态。
- reconcile 设计为幂等，重复执行不产生副作用。

### 7.4 Agent 恢复机制

- Agent 以 DaemonSet 运行，节点重启或 Pod 重建后由 Kubernetes 自动重新调度。
- Agent 启动时执行 Bootstrap 恢复逻辑：读取本节点 NodeState，重建内存中的运行任务状态记录；实际任务重新注册由首轮 reconcile 完成。
- Bootstrap 完成后进入正常 reconcile 循环，与当前 Policy 对账，补齐差量。
- 若 etmemd 在节点重启后尚未就绪，Agent 会持续重试并在 NodeState 中记录通路状态。

### 7.5 etmemd 异常处理

etmemd 不受 Kubernetes 管理，其健康状态通过 Agent 的 Unix socket 探测反映至 NodeState。若 etmemd 重启，Agent 会检测到 socket 不可达并在 NodeState 中记录通路异常；待 etmemd 恢复后，下次 reconcile 更新健康状态，具体任务重新注册路径见详细设计。

## 8. 交付与升级设计

### 8.1 标准交付物清单

v1.0 的正式交付物包括以下内容：

| 交付物 | 说明 |
|--------|------|
| `etmem-operator-<version>.tgz` | Helm Chart，包含 CRD、Deployment、DaemonSet、RBAC 模板 |
| `etmem-operator-<version>-images.tar.gz` | 镜像 tar 包（Operator 镜像 + Agent 镜像） |
| `scripts/load-images.sh` | 离线镜像导入脚本 |
| `scripts/verify-etmem.sh` | 部署后验证脚本 |
| 本文档（v1.0-overview-design.md）及需求说明、详细设计文档 | 工程文档 |

### 8.2 在线安装路径

适用于可访问外部镜像仓库的环境：

```bash
# 1. Helm 安装（CRD 由 Helm Chart 内置 crds/ 目录自动部署）
helm install etmem-operator etmem-operator-<version>.tgz \
  --namespace etmem-system --create-namespace

# 2. 验证
kubectl get pods -n etmem-system
kubectl get etmemnodestates -A
```

### 8.3 离线安装路径（正式交付路径之一）

适用于无外网环境，离线镜像导入是 v1.0 的正式交付路径之一：

```bash
# 1. 在可访问机器上构建产物（或由发布流程生成）
# 2. 将 tar 包拷贝至目标节点
# 3. 导入镜像
bash scripts/load-images.sh

# 4. 为 Helm 指定本地镜像（无需从外部拉取）
helm install etmem-operator etmem-operator-<version>.tgz \
  --namespace etmem-system --create-namespace \
  --set image.pullPolicy=Never \
  --set agent.image.pullPolicy=Never

# 5. 验证
bash scripts/verify-etmem.sh
```

### 8.4 现有工作负载纳管路径

对已运行的工作负载（如 DaemonSet）通过 patch 方式纳管，无需重建工作负载：

```bash
kubectl patch daemonset <name> -n <namespace> \
  --type=merge \
  -p '{"spec":{"template":{"metadata":{"labels":{"etmem.openeuler.io/enable":"true"}}}}}'
```

### 8.5 升级路径

Helm 负责版本升级的统一入口：

```bash
# 在线升级
helm upgrade etmem-operator etmem-operator-<new-version>.tgz \
  --namespace etmem-system

# 离线升级（先导入新版镜像）
bash scripts/load-images.sh <new-version>
helm upgrade etmem-operator etmem-operator-<new-version>.tgz \
  --namespace etmem-system \
  --set image.pullPolicy=Never \
  --set agent.image.pullPolicy=Never
```

**升级策略**：

- Operator 采用滚动更新（RollingUpdate），保证升级期间控制面不中断。
- Agent DaemonSet 采用滚动更新，逐节点更新，避免全量同时重启。
- CRD 升级向前兼容，新增字段为 optional，不破坏已有资源。
- Helm uninstall 不自动删除 CRD，防止数据丢失；CRD 删除需人工确认。

### 8.6 回滚路径

```bash
helm rollback etmem-operator [REVISION] --namespace etmem-system
```

Helm rollback 会恢复 Operator 和 Agent 至目标版本镜像，但不回滚 CRD schema（CRD 回滚需手动操作）。

## 9. 安全与权限边界

### 9.1 权限最小化原则

各组件遵循最小权限原则，权限不相互混用：

| 组件 | 核心权限 | 权限依据 |
|------|----------|----------|
| Operator | EtmemPolicy（全权限）、EtmemNodeState（只读）、Pod（只读）、Events（创建） | 需要创建/收敛自动策略，不需要节点访问 |
| Agent | EtmemPolicy（只读）、EtmemNodeState（全权限）、Pod（只读）、Node（只读）、Events（创建） | 需要写回节点状态，不需要修改策略 |

### 9.2 Agent 特权说明

Agent Pod 需要以下宿主机访问权限：

- **挂载 `/sys/fs/cgroup`**：用于 cgroup v1 路径解析和 PID 读取。
- **挂载 `/var/run/etmemd.sock`**：用于与 etmemd 进行 Unix socket 通信。
- **privileged 模式**：访问宿主机 cgroup 文件系统所必需。

这些权限是功能性必要权限，不可缩减。

### 9.3 隔离边界

- Operator 与 Agent 各自使用独立的 ServiceAccount，互不共享权限。
- etmem-system 命名空间为系统专用命名空间，业务命名空间中的 EtmemPolicy 由 Operator 自动管理。
- CRD 资源属于集群级别，RBAC 通过 ClusterRole 控制访问粒度。

### 9.4 本版本安全范围

v1.0 实现基础 RBAC 最小权限设计，以下内容不在本版本展开：

- 多租户 namespace 级极限隔离
- Pod Security Standards 的全面加固
- 网络策略（NetworkPolicy）的精细管控

## 10. 设计权衡

### 10.1 Operator 采用 Deployment 而非 DaemonSet

**决策**：Operator 负责控制面 reconcile，不依赖节点资源，以 Deployment 运行更符合控制器组件的标准模式，也便于 Leader Election 和滚动升级管理。

### 10.2 Agent 采用 DaemonSet 而非 Deployment

**决策**：Agent 的核心职责是节点本地执行（cgroup 路径解析、etmemd socket 通信），必须与节点一一绑定。DaemonSet 天然保证每节点一个 Pod 的语义，是唯一合理选择。

### 10.3 namespace 内一个 etmem-auto Policy 而非每 Pod 一个

**决策**：同一 namespace 内多个 Pod 的 etmem 配置合并为一个 Policy（most-aggressive-wins），降低 API 对象数量，简化 Agent 的对账逻辑。代价是同一 namespace 内无法为不同 Pod 设置完全独立的参数（本版本不需要此能力）。

### 10.4 Helm 作为唯一安装/升级入口

**决策**：Helm 提供版本化的安装和升级语义，值驱动配置，支持 rollback，是 Kubernetes 应用管理的事实标准。避免使用裸 kubectl apply，确保升级路径可预期、可追溯。

### 10.5 离线镜像导入作为正式交付路径

**决策**：目标环境（openEuler/FusionOS 企业内网）通常无法直接访问外部镜像仓库。离线镜像 tar 包 + 导入脚本是确保可交付性的必要路径，与 Helm 在线安装路径并列，共同构成完整交付方案。

### 10.6 MemAvailable 定位为辅助观测

**决策**：`MemAvailable` 是整机净值，受页缓存回收、buddy 系统、其他进程内存活动等多重因素影响，单次观测结果不稳定，不能作为"etmem 是否生效"的单一判据。以 `VmSwap > 0` 和 `Swap used` 增长为主要验证指标，`MemAvailable` 仅作辅助参考。

### 10.7 etmemd 不纳入 Kubernetes 管理

**决策**：etmemd 依赖特定内核模块，属于节点级系统服务，生命周期与 Kubernetes 工作负载不同。将其保留为节点守护进程，由节点运维负责安装和维护，可以与 Kubernetes 版本升级解耦，降低相互影响风险。

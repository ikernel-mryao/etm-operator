## ⚠️ 重要说明：「本地验证」与「真实 swap 验证」的边界

etmem-operator 的功能分为两层：

| 层级 | 说明 | 在哪验证 |
|------|------|----------|
| **控制面** | Operator 监听 Pod label → 自动创建 EtmemPolicy；Agent 匹配 Pod → 解析 PID → 生成配置 | **任意 K8s 集群**（包括 Kind、Minikube、单节点集群等） |
| **数据面** | etmemd 扫描冷页 → 写入 swap → 可用内存增长 | **仅限真实目标环境**（openEuler/FusionOS 内核 + etmem 内核模块 + swap 设备） |

- 如果您的机器**没有** openEuler 内核和 etmem 内核模块，您仍然可以验证**控制面**的全部流程。
- 如果您的机器**有** openEuler 内核和 etmem 内核模块，您可以验证**完整的数据面**，即真正的内存扫描和 swap 操作。

本手册会在每个步骤中明确标注该步骤属于哪个层级。

## 1. 环境准备（硬件/软件要求）

### 1.1 硬件要求

| 资源 | 最低要求 | 建议 |
|------|----------|------|
| CPU  | 2 核     | 4 核+ |
| 内存 | 4 GB     | 8 GB+ |
| 磁盘 | 20 GB 可用空间 | 40 GB+ |
| 网络 | 能访问互联网（拉取基础镜像） | — |

### 1.2 软件要求

| 软件 | 版本要求 | 用途 |
|------|----------|------|
| **操作系统** | openEuler 22.03+ 或 FusionOS 23（数据面验证必需）；其他 Linux 发行版可验证控制面 | 提供 etmem 内核模块 |
| **Kubernetes** | 1.26+ | 集群运行环境 |
| **Helm** | 3.x | 部署 Operator 和 Agent |
| **Go** | **1.25+**（推荐 1.26）| 编译源码（go.mod 要求 ≥1.25.0，yum 安装的 1.21 **不够**） |
| **containerd** | 1.6+ | 容器运行时 |
| **nerdctl** 或 **docker** | 最新版 | 构建容器镜像 |
| **git** | 任意版本 | 获取源码 |

### 1.3 数据面额外要求（仅 openEuler/FusionOS 内核需要）

| 条件 | 说明 |
|------|------|
| 内核模块 `etmem_scan`、`etmem_swap` | 必须加载 |
| `etmemd` 二进制 | 必须安装并在每个 Worker 节点运行 |
| swap 设备 | 必须启用（`swapon -s` 应有输出） |
| cgroup v1 | 当前版本仅支持 cgroup v1（不支持 cgroup v2） |

## 2. 环境检查（检查所有前提条件）

> 在开始部署之前，请逐项检查以下条件。任何一项不满足都可能导致部署失败或功能受限。

### 2.1 检查 Kubernetes 集群

**做什么：** 确认集群可用且版本满足要求。

```bash
kubectl version --short 2>/dev/null || kubectl version
```

**预期输出（示例）：**
```
Client Version: v1.28.2
Server Version: v1.28.2
```

**异常处理：**
- 如果提示 `command not found`：请先安装 kubectl。
- 如果 Server Version 为空或报错：请确认集群已启动（`systemctl status kubelet`）。
- 如果版本低于 1.26：建议升级集群。

### 2.2 检查 Helm

**做什么：** 确认 Helm 3 已安装。

```bash
helm version --short
```

**预期输出（示例）：**
```
v3.13.1+g3547a4b
```

**异常处理：**
- 如果提示 `command not found`：请安装 Helm 3（参考 https://helm.sh/docs/intro/install/）。

### 2.3 检查 Go 编译器

**做什么：** 确认 Go 版本满足要求（**≥ 1.25.0**，推荐 1.26）。

> ⚠️ **重要**：`go.mod` 要求 `go 1.25.0`，如果你的 Go 版本低于 1.25（例如 yum 安装的 1.21.x），编译会报错：
> `go: go.mod requires go >= 1.25.0 (running go 1.21.4; GOTOOLCHAIN=local)`

```bash
go version
```

**预期输出（示例）：**
```
go version go1.26.0 linux/amd64
```

**异常处理：**
- 如果版本低于 1.25（例如 yum 安装的 1.21.x）：**必须手动升级 Go**，yum/dnf 仓库通常没有 1.25+ 版本。
- 升级方法（手动安装 tarball）：

```bash
# 1. 下载 Go 1.26 二进制包
wget https://go.dev/dl/go1.26.0.linux-amd64.tar.gz

# 2. 删除旧版本并解压新版本
sudo rm -rf /usr/local/go
sudo tar -C /usr/local -xzf go1.26.0.linux-amd64.tar.gz

# 3. 确保 PATH 包含 /usr/local/go/bin（添加到 ~/.bashrc）
export PATH=/usr/local/go/bin:$PATH
echo 'export PATH=/usr/local/go/bin:$PATH' >> ~/.bashrc

# 4. 验证版本
go version
# 预期输出：go version go1.26.0 linux/amd64
```

- 如果在离线/内网环境中无法直接下载，可从能上网的机器下载后通过 scp 传输：
```bash
# 在能上网的机器上
wget https://go.dev/dl/go1.26.0.linux-amd64.tar.gz
scp go1.26.0.linux-amd64.tar.gz root@<目标机IP>:/tmp/

# 在目标机上
sudo rm -rf /usr/local/go
sudo tar -C /usr/local -xzf /tmp/go1.26.0.linux-amd64.tar.gz
export PATH=/usr/local/go/bin:$PATH
```

### 2.4 检查容器运行时

**做什么：** 确认 containerd 正在运行，且 nerdctl 或 docker 可用。

```bash
# 检查 containerd
systemctl status containerd | head -5

# 检查 nerdctl（推荐）
sudo nerdctl version

# 或者检查 docker
docker version
```

**预期输出：** containerd 状态为 `active (running)`，nerdctl/docker 能正常输出版本号。

**异常处理：**
- containerd 未运行：`sudo systemctl start containerd`
- nerdctl 和 docker 都不可用：请安装其中之一。

### 2.5 检查 cgroup 版本（控制面 + 数据面）

**做什么：** 确认系统使用 cgroup v1。当前版本的 Agent **不支持 cgroup v2**。

```bash
# 方法一：检查 /sys/fs/cgroup 挂载类型
stat -fc %T /sys/fs/cgroup/
```

**预期输出：**
```
tmpfs
```

> 如果输出是 `tmpfs`，说明是 cgroup v1。如果输出是 `cgroup2fs`，说明是 cgroup v2，Agent 将**无法解析容器 PID**。

```bash
# 方法二：检查 memory cgroup 子系统是否存在
ls /sys/fs/cgroup/memory/
```

**预期输出：** 应有文件列表（`cgroup.procs`、`memory.usage_in_bytes` 等）。

**异常处理：**
- 如果是 cgroup v2：需要在内核启动参数中添加 `systemd.unified_cgroup_hierarchy=0` 切换到 v1，然后重启。
- 这是**已知限制**，v2 支持将在未来版本添加。

### 2.6 【数据面】检查 etmem 内核模块

> ⚠️ 此步骤仅在 openEuler/FusionOS 内核上可用。非 openEuler 内核可跳过，但将无法验证数据面。

```bash
# 尝试加载内核模块
sudo modprobe etmem_scan etmem_swap

# 确认加载成功
lsmod | grep etmem
```

**预期输出：**
```
etmem_swap            xxxxx  0
etmem_scan            xxxxx  0
```

**异常处理：**
- `modprobe: FATAL: Module etmem_scan not found`：您的内核不包含 etmem 模块，无法验证数据面。
- 确认您运行的是 openEuler 内核：`uname -r` 应包含 `oe` 字样。

### 2.7 【数据面】检查 etmemd 守护进程

```bash
# 检查 etmemd 是否已安装
which etmemd

# 检查 etmemd 是否正在运行
ps aux | grep etmemd | grep -v grep
```

**预期输出：**
```
/usr/bin/etmemd
root  xxxxx  ... etmemd -s etmemd_socket -l 0
```

**如果 etmemd 未运行，手动启动：**

```bash
# 后台启动 etmemd
sudo etmemd -s etmemd_socket -l 0 &

# 确认启动成功
ps aux | grep etmemd | grep -v grep
```

**异常处理：**
- `etmemd: command not found`：需要在 openEuler 上安装 etmemd 包（`yum install etmem`）。
- Agent 启动时会连接 `/var/run/etmemd_socket`，如果 etmemd 没启动，Agent 功能将受限。

### 2.8 【数据面】检查 swap 状态

```bash
swapon -s
cat /proc/swaps
free -h | grep -i swap
```

**预期输出（示例）：**
```
Filename                Type        Size      Used    Priority
/dev/sda2               partition   4194300   0       -2

              total        used        free
Swap:         4.0Gi       0B          4.0Gi
```

**如果 swap 未启用：**

```bash
# 创建 swap 文件（示例：4GB）
sudo fallocate -l 4G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile

# 确认生效
swapon -s
```

**异常处理：**
- 如果 `swapon -s` 无输出：swap 未配置，etmemd 无法将冷页交换出去。

## 3. 获取代码与构建

### 3.1 获取源码

**做什么：** 克隆 etmem-operator 仓库。

```bash
# 如果已有代码，跳过此步
git clone https://github.com/ikernel-mryao/etm-operator.git etmem-operator
cd etmem-operator
```

### 3.2 编译二进制文件

根据您的网络和 Go 环境，选择以下**三种方式之一**：

#### 方式 A：在线编译（网络正常 + Go ≥ 1.25）

```bash
export PATH=/usr/local/go/bin:$PATH
CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build -o bin/operator ./cmd/operator
CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build -o bin/agent ./cmd/agent
```

#### 方式 B：离线编译（无网络 + Go ≥ 1.25）⭐ 推荐

仓库已包含 `vendor/` 目录（所有依赖已本地化），**无需联网**：

```bash
export PATH=/usr/local/go/bin:$PATH
mkdir -p bin

# 使用 vendor 目录离线编译（关键参数：-mod=vendor）
CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build -mod=vendor -o bin/operator ./cmd/operator
CGO_ENABLED=0 GOOS=linux GOARCH=amd64 go build -mod=vendor -o bin/agent ./cmd/agent
```

> ⚠️ 仍然需要 Go ≥ 1.25.0。如果 Go 版本太旧，见 [2.3 检查 Go 编译器](#23-检查-go-编译器) 中的手动升级方法。

#### 方式 C：直接使用预编译二进制（无 Go 也可以）⭐⭐ 最快

仓库 `dist/` 目录下已提供**预编译的静态二进制文件**（Go 1.26.0, linux/amd64, CGO_ENABLED=0），
**不需要安装 Go**，直接使用即可：

```bash
mkdir -p bin
cp dist/operator bin/operator
cp dist/agent bin/agent
chmod +x bin/operator bin/agent
```

> 说明：dist/ 下的二进制是**静态链接**的，不依赖任何系统动态库，可在任何 linux/amd64 系统上运行。

### 3.3 验证二进制文件

**预期输出：** `bin/` 目录下出现 `operator` 和 `agent` 两个二进制文件。

```bash
ls -la bin/
```

**预期输出：**
```
-rwxr-xr-x 1 user user xxxxxxx  ... operator
-rwxr-xr-x 1 user user xxxxxxx  ... agent
```

**异常处理：**

| 错误信息 | 原因 | 解决方法 |
|---------|------|---------|
| `go.mod requires go >= 1.25.0` | Go 版本太旧（如 yum 安装的 1.21.x） | 手动安装 Go 1.26 tarball（见 2.3 节），或直接用方式 C |
| `dial tcp: lookup proxy.golang.org: no such host` | DNS 解析失败，无法下载依赖 | 使用方式 B（`-mod=vendor`）离线编译 |
| `x509: certificate is not authorized` | 企业代理 TLS 拦截 | 使用方式 B 或方式 C |
| `unexpected EOF` / 下载中断 | 代理不稳定 | 使用方式 B 或方式 C |

## 4. 构建镜像

### 4.1 准备二进制文件

**做什么：** 确保 `bin/` 目录下有编译好的二进制文件。

如果您在第 3 步中使用了**方式 C（预编译二进制）**，需要先复制到 bin/：

```bash
mkdir -p bin
cp dist/operator bin/operator
cp dist/agent bin/agent
chmod +x bin/operator bin/agent
```

### 4.2 构建容器镜像

**做什么：** 使用 nerdctl（或 docker）构建 Operator 和 Agent 的容器镜像。

> ⚠️ **关键：** 必须使用 `--namespace k8s.io` 参数，这样 containerd（K8s 的容器运行时）才能找到镜像。
>
> ⚠️ **再次注意：** `nerdctl build` 不是独立工作的，它依赖 **BuildKit**。
> 如果机器上缺少 `buildctl` 或 `buildkitd`，会直接报错：
> `buildctl: executable file not found in $PATH`
> 或
> `no buildkit host is available`

先检查 BuildKit 是否可用：

```bash
which buildctl
which buildkitd
systemctl status buildkit 2>/dev/null || true
```

- 如果 `buildctl` 和 `buildkitd` 都存在，并且 `buildkitd` 已运行：可以继续执行下面的 `nerdctl build`
- 如果缺少 `buildctl/buildkitd`：**不要继续卡在 4.2**，请直接跳到 **4.4 离线镜像方案**
- 如果本机装了 Docker，但没有 BuildKit：也可以改用 `docker build`，然后再导入到 containerd

```bash
# 构建 Operator 镜像（使用预编译二进制，不需要 Go 环境）
sudo nerdctl --namespace k8s.io build -t etmem-operator:v0.4.0-pid -f build/operator/Dockerfile .

# 构建 Agent 镜像
sudo nerdctl --namespace k8s.io build -t etmem-agent:v0.4.0-pid -f build/agent/Dockerfile .
```

**预期输出：** 构建成功，最后一行类似：
```
naming to etmem-operator:v0.4.0-pid done
naming to etmem-agent:v0.4.0-pid done
```

> 说明：Dockerfile 已优化为**直接 COPY 预编译二进制**，不再需要多阶段 Go 编译，因此不需要网络也不需要 Go 环境。唯一需要的基础镜像（`gcr.io/distroless/static:nonroot` 和 `openeuler/openeuler:22.03-lts`）如果本地没有缓存，需要能拉取。如果也无法拉取，请参考 4.4 节的离线方案。

### 4.3 确认镜像存在

```bash
sudo nerdctl --namespace k8s.io images | grep etmem
```

**预期输出：**
```
etmem-operator   v0.4.0-pid   xxxxxx   ...
etmem-agent      v0.4.0-pid   xxxxxx   ...
```

**异常处理：**
- 构建失败提示 Dockerfile 不存在：确认您在 `etmem-operator` 项目根目录，且使用了 `-f build/operator/Dockerfile` 路径。
- 使用 docker 的话：`docker build -t etmem-operator:v0.4.0-pid -f build/operator/Dockerfile .`
- 如果报 `buildctl: executable file not found in $PATH`：这不是二进制有问题，而是 **BuildKit 没装**。最快办法是直接走 **4.4 离线镜像方案**，在另一台能构建镜像的机器上先构建并导出 tar，再在目标机 `nerdctl load`。

### 4.4 离线镜像方案（无法拉取基础镜像时）

如果目标服务器出现以下任一情况，都建议直接使用本节方案：

- 无法拉取 `gcr.io/distroless/static:nonroot` 或 `openeuler/openeuler:22.03-lts`
- `nerdctl build` 报 `buildctl` / `buildkitd` 缺失
- 目标服务器网络或代理环境不稳定

做法是在**另一台能正常构建镜像的机器**上预构建后导出：

```bash
# 在有网络的机器上构建并导出
sudo nerdctl --namespace k8s.io build -t etmem-operator:v0.4.0-pid -f build/operator/Dockerfile .
sudo nerdctl --namespace k8s.io build -t etmem-agent:v0.4.0-pid -f build/agent/Dockerfile .
sudo nerdctl --namespace k8s.io save etmem-operator:v0.4.0-pid -o etmem-operator.tar
sudo nerdctl --namespace k8s.io save etmem-agent:v0.4.0-pid -o etmem-agent.tar

# scp 到目标服务器
scp etmem-operator.tar etmem-agent.tar root@<目标服务器IP>:/tmp/

# 在目标服务器上导入
sudo nerdctl --namespace k8s.io load -i /tmp/etmem-operator.tar
sudo nerdctl --namespace k8s.io load -i /tmp/etmem-agent.tar
```

### 4.5 多节点集群：分发镜像

> 如果您的集群有多个 Worker 节点，需要将镜像复制到每个节点。

```bash
# 导出镜像
sudo nerdctl --namespace k8s.io save etmem-operator:v0.4.0-pid -o etmem-operator.tar
sudo nerdctl --namespace k8s.io save etmem-agent:v0.4.0-pid -o etmem-agent.tar

# 复制到其他节点（替换 <node-ip>）
scp etmem-operator.tar etmem-agent.tar root@<node-ip>:/root/

# 在其他节点上导入
ssh root@<node-ip> "nerdctl --namespace k8s.io load -i /root/etmem-operator.tar"
ssh root@<node-ip> "nerdctl --namespace k8s.io load -i /root/etmem-agent.tar"
```

## 5. 部署 etmem-operator（Helm）

### 5.1 安装 CRDs 和组件

**做什么：** 使用 Helm 一键部署 Operator、Agent、CRDs、RBAC 等全部组件。

> ⚠️ 请先确认当前目录是 **etmem-operator 仓库根目录**（也就是能看到 `deploy/helm/` 的目录）。
> 如果您当前在上一级目录，例如 `/home/ygz/work/etmem-work`，请先执行：
> `cd /home/ygz/work/etmem-work/etm-operator`
> 或者直接把 Chart 路径写成 `./etm-operator/deploy/helm`

```bash
helm install etmem-operator ./deploy/helm \
  --namespace etmem-system \
  --create-namespace
```

**预期输出：**
```
NAME: etmem-operator
LAST DEPLOYED: ...
NAMESPACE: etmem-system
STATUS: deployed
REVISION: 1
```

**异常处理：**
- `Error: INSTALLATION FAILED: cannot re-use a name that is still in use`：之前已安装过，先卸载：`helm uninstall etmem-operator -n etmem-system`。
- 如果 Pod 出现 `ErrImagePull` / `ImagePullBackOff`，而本地已经通过 `nerdctl load` 导入了 `etmem-operator:v0.4.0-pid` 和 `etmem-agent:v0.4.0-pid`，请执行：
  ```bash
  helm upgrade etmem-operator ./deploy/helm \
    --namespace etmem-system \
    --set operator.image=etmem-operator:v0.4.0-pid \
    --set agent.image=etmem-agent:v0.4.0-pid
  ```
- 如果需要自定义配置（例如修改 Agent 的 reconcileInterval），可以通过 `--set` 参数：
  ```bash
  helm install etmem-operator ./deploy/helm \
    --namespace etmem-system \
  ```

#### 9.5.2 检查进程的 VmSwap

1. etmemd 未运行或 socket 不通

5. etmem 内核模块未加载

```bash

# 检查 swap_extension（openEuler 特有）
cat /proc/sys/vm/swap_extension 2>/dev/null

# 检查 etmemd socket

# 检查 etmemd 日志
journalctl -u etmemd --tail=50
```

## Pod 内多进程场景说明

> 🏷️ **层级：控制面 + 数据面** — 本节说明当 Pod 内运行多个进程时，etmem-operator 如何自动处理。

### 当前版本的多进程支持

从 Agent v0.3.0-mp 开始，etmem-operator **自动支持 Pod 内多进程**。您无需做任何额外配置——Agent 会自动发现 Pod 中的所有用户进程，并为每个进程创建独立的 etmemd 项目。

**工作原理：**

1. Agent 扫描目标 Pod 的 cgroup，获取所有进程列表
2. 自动过滤掉 Kubernetes 基础设施进程（`pause`、`sandbox`）
3. 为每个用户进程创建一个独立的 etmemd 项目和配置文件
4. 每个配置文件只包含一个 `[task]` 段（避免 etmemd INI 解析器的限制）
5. 项目命名格式：`{namespace}-{podName}-{processName}`

**对用户完全透明** — 您只需像平常一样给 Pod 打标签，Agent 会自动处理多进程。

### 如何创建多进程测试 Pod

如果您想测试多进程场景，可以使用以下 YAML 创建一个包含 2 个内存消费进程的 Pod：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: multiproc-test
  namespace: default
  labels:
    etmem.openeuler.io/enable: "true"
  annotations:
    etmem.openeuler.io/profile: "aggressive"
spec:
  nodeSelector:
    kubernetes.io/hostname: <您的节点名>    # 替换为实际节点名
  containers:
  - name: dual-memory
    image: busybox
    command:
    - sh
    - -c
    - |
      # 启动两个独立的内存消费进程
      # memeat1: 分配 150MB 内存
      sh -c 'exec -a memeat1 tail -f /dev/zero' &
      # memeat2: 分配 150MB 内存
      sh -c 'exec -a memeat2 tail -f /dev/zero' &
      wait
    resources:
      requests:
        memory: "512Mi"
      limits:
        memory: "512Mi"
```

> **提示**：以上仅为示意。实际测试中建议使用 `polinux/stress` 等专用工具来分配真实的常驻内存。

### 如何验证多进程正在工作

#### 步骤 1：检查 etmemd 项目列表

```bash
sudo /usr/bin/etmem project show -s etmemd_socket
```

**预期输出**：如果 Pod 有 2 个用户进程（如 memeat1 和 memeat2），应看到 2-3 个对应项目：

```
default-multiproc-test-memeat1   # 用户进程 1
default-multiproc-test-memeat2   # 用户进程 2
default-multiproc-test-sh        # 容器入口 sh（无害的 wrapper 进程）
```

> **关于 `sh` 项目**：这是容器的入口进程（entrypoint shell），它本身几乎不占内存。Agent 无法区分它和用户进程，所以也会为它创建项目。这是无害的——etmemd 会扫描它但几乎不会换出任何东西。

#### 步骤 2：检查每个进程的 VmSwap

```bash
# 从 Agent 日志或 cgroup 获取各进程的 PID
# 然后逐一检查：
for pid in <PID1> <PID2>; do
  echo "--- PID=$pid ---"
  grep -E 'Name|VmRSS|VmSwap' /proc/$pid/status
done
```

**预期输出**（等待 1-2 分钟后）：

```
--- PID=2338929 ---
Name:   memeat1
VmRSS:      1788 kB
VmSwap:   152424 kB
--- PID=2338930 ---
Name:   memeat2
VmRSS:      1784 kB
VmSwap:   152424 kB
```

每个进程的 `VmSwap` 都应 > 0，表示各自的冷页都被独立换出。

#### 步骤 3：确认总 VmSwap

**总 VmSwap = 各进程 VmSwap 之和**。在上面的例子中：

- memeat1 VmSwap = 152,424 kB (148 MB)
- memeat2 VmSwap = 152,424 kB (148 MB)
- 总计 = 304,848 kB (297 MB)

如果只有一个进程有 VmSwap 而另一个为 0，说明多进程管理可能未完全生效，请检查 Agent 版本和日志。

## 12. 常见问题排查

### Q1: Pod 状态为 ImagePullBackOff

**原因：** K8s 节点找不到镜像。

**解决：**
```bash
# 确认镜像在节点上存在（注意 namespace 必须是 k8s.io）
sudo nerdctl --namespace k8s.io images | grep etmem

# 如果不存在，重新构建或导入
sudo nerdctl --namespace k8s.io build -t etmem-operator:v0.2.0 -f Dockerfile.operator .
```

### Q2: Operator 日志报错 "failed to list/watch"

**原因：** CRD 未正确安装或 RBAC 权限不足。

**解决：**
```bash
# 重新检查 CRD
kubectl get crd | grep etmem

# 检查 ServiceAccount
kubectl -n etmem-system get serviceaccount

# 检查 ClusterRole/ClusterRoleBinding
kubectl get clusterrole | grep etmem
kubectl get clusterrolebinding | grep etmem
```

### Q3: 添加 label 后没有自动创建 EtmemPolicy

**可能原因：**
1. Operator 未运行或异常
2. Label 拼写错误

**排查：**
```bash
# 确认 label 正确
kubectl -n etmem-test get pod memory-hog --show-labels

# 确认 Operator 日志
kubectl -n etmem-system logs deploy/etmem-operator --tail=30

# label 必须是 etmem.openeuler.io/enable=true（注意大小写）
```

### Q4: Agent 日志中无 "Matched pod" 信息

**可能原因：**
1. Agent 未运行在 Pod 所在节点
2. EtmemPolicy 未创建
3. Agent 的 reconcileInterval 较长，需等待

**排查：**
```bash
# 确认 Pod 所在节点
kubectl -n etmem-test get pod memory-hog -o wide

# 确认该节点上有 Agent Pod
kubectl -n etmem-system get pods -o wide | grep agent
```

### Q5: Agent 找不到容器 PID

**可能原因：** cgroup v2 环境，或 cgroup 路径不匹配。

**排查：**
```bash
# 在节点上手动检查 cgroup 路径
# 获取 Pod UID
kubectl -n etmem-test get pod memory-hog -o jsonpath='{.metadata.uid}'

# 在节点上查找对应 cgroup（需要 ssh 到节点）
find /sys/fs/cgroup/memory/kubepods -name "cgroup.procs" | head -20
```

**cgroup v1 PID 解析路径说明：**

Agent 通过以下路径解析容器 PID：
1. 从 Pod metadata 获取 UID
2. 根据 QoS 等级确定路径前缀：
   - Guaranteed: `/sys/fs/cgroup/memory/kubepods/<pod-uid>/`
   - Burstable: `/sys/fs/cgroup/memory/kubepods/burstable/<pod-uid>/`
   - BestEffort: `/sys/fs/cgroup/memory/kubepods/besteffort/<pod-uid>/`
3. 在路径下找到 `<container-id>/cgroup.procs` 文件
4. 从文件中读取 PID 列表

### Q6: etmemd 扫描成功但不 swap

**现象：** Agent 日志正常，VmSwap 始终为 0。

**这是我们在真实验证中发现的问题。** 原因可能是：
1. 系统内存使用率未达到 profile 阈值 — 尝试使用 `extreme` profile
2. swappiness 设置过低 — `sysctl vm.swappiness=60`
3. swap 设备未正确配置 — 检查 `swapon -s`
4. etmemd 版本与内核不匹配

### Q7: 多节点环境部署失败

**解决：**
1. 确保每个 Worker 节点都有镜像（参考 4.3 节）
2. 确保每个 Worker 节点都运行了 etmemd（数据面必需）
3. 确保每个 Worker 节点都加载了 etmem 内核模块（数据面必需）

### Q8: Helm 升级后配置不生效

```bash
# 使用 upgrade 而非 install
helm upgrade etmem-operator ./deploy/helm \
  --namespace etmem-system \
  --set agent.reconcileInterval=60s
```

## 13. 数据面验证指导

> 🏷️ **层级：数据面** — 本节内容仅适用于具备 openEuler/FusionOS 内核 + etmem 内核模块的真实目标环境。
>
> 前面的步骤（第 1–12 节）已经覆盖了**控制面**的完整验证。如果您已经验证控制面正常工作（EtmemPolicy 已创建、Agent 日志显示 "Matched pod"），但目标进程的 `VmSwap` 始终为 0，请仔细阅读本节。

### 13.1 数据面验证前提条件

要让 etmemd 真正执行内存扫描和 swap 操作，**必须同时满足**以下所有条件：

| # | 前提条件 | 检查方法 | 说明 |
|---|----------|----------|------|
| 1 | openEuler 系内核 | `uname -r`（应包含 `oe` 字样） | 支持的内核包括：FusionOS、openEuler 22.03 LTS 等。普通 Linux 内核（如 Ubuntu、CentOS 原生内核）**不包含** etmem 内核模块 |
| 2 | `etmem_scan` 和 `etmem_swap` 内核模块已加载 | `lsmod \| grep etmem` | 如果没有输出，请尝试 `sudo modprobe etmem_scan etmem_swap` 加载 |
| 3 | `etmemd` 和 `etmem` CLI 已安装 | `which etmemd && which etmem` | 可通过 `yum install etmem` 安装，或从源码编译 |
| 4 | swap 分区/文件已启用 | `swapon -s`（应有输出） | 如果没有 swap，etmemd 没有地方写入冷页数据。详见 2.8 节创建 swap 文件的方法 |
| 5 | cgroup v1 | `stat -fc %T /sys/fs/cgroup/`（应输出 `tmpfs`） | 当前 MVP 版本仅支持 cgroup v1，cgroup v2 下 Agent 无法正确解析容器 PID |
| 6 | etmemd 在每个目标节点上运行 | `ps aux \| grep etmemd` | 如未运行，请手动启动（见下方命令） |

**启动 etmemd 的推荐命令：**

```bash
# 在每个需要执行 swap 的 Worker 节点上运行
# -s 指定 socket 名称，Agent 会通过此 socket 与 etmemd 通信
# -l 2 表示日志级别为 2（较详细），方便排查问题
sudo nohup /usr/bin/etmemd -s etmemd_socket -l 2 &

# 确认 etmemd 已启动
ps aux | grep etmemd | grep -v grep
```

> 💡 **提示：** 日志级别 `-l 0` 为静默模式，`-l 2` 为详细模式。初次部署排查问题时建议使用 `-l 2`，稳定运行后可改为 `-l 0`。

### 13.2 常见数据面问题排查

#### 问题 1：VmSwap 始终为 0

这是数据面验证中**最常见的问题**。以下是按可能性从高到低排列的原因及排查方法：

---

**原因 ①（最常见）：`sysmem_threshold` 设置过低**

`sysmem_threshold`（即 profile 中的 `sysMem`）的含义是：**只有当系统可用内存 (MemAvailable) 低于 MemTotal 的 N% 时，etmemd 才会触发 swap 操作**。

换句话说：`threshold=N` 表示 "当可用内存 < 总内存的 N% 时才触发"。

**举例说明：**
- 假设您的机器有 **1TB 内存**，当前 `MemAvailable` 占 `MemTotal` 的 **82%**
- 如果使用 `conservative` profile（`sysMem=70`），那么阈值条件是 "可用内存 < 70%"
- 当前可用内存 82% > 70%，**条件不满足，etmemd 不会触发任何 swap 操作**
- 即使使用 `aggressive` profile（`sysMem=30`），82% 仍然远高于 30%，依然不会触发
- 只有 `extreme` profile（`sysMem=20`）也不够——需要阈值 > 82 才能触发

**解决办法：**
- 使用 `aggressive` 或 `extreme` profile 进行测试（`aggressive` 的阈值为 95，详见第 10.3 节）
- 或者在测试环境中运行一个**占用大量内存**的程序，人为降低可用内存比例

```bash
# 检查当前系统可用内存比例
awk '/MemTotal/{t=$2} /MemAvailable/{a=$2} END{printf "MemTotal: %.1f GB\nMemAvailable: %.1f GB\n可用内存比例: %.1f%%\n", t/1024/1024, a/1024/1024, a/t*100}' /proc/meminfo
```

> ⚠️ **注意：** 第 10.3 节中的 profile 参数表里 `sysMem%` 列显示的是"系统内存使用 > X% 才触发"的简化描述。实际实现中，这个值对应配置文件中的 `sysmem_threshold` 参数，含义是"可用内存低于总内存的 threshold% 时触发"。`aggressive` profile 的实际 `sysmem_threshold` 值为 **95**，即系统可用内存低于 95% 时就会触发——这在大多数场景下都能满足。

---

**原因 ②：etmemd 加载了错误的任务（task）**

etmemd 的配置文件支持多个 `[task]` 段，但 etmemd **只会加载最后一个** `[task]` 段。如果配置文件中有多个 `[task]`，前面的任务会被忽略。

**排查方法：**

```bash
# 查看 etmemd 当前加载的项目和任务
sudo /usr/bin/etmem project show -s etmemd_socket
```

**关注点：**
- 输出中的 `task` 值应该是您的**应用进程名称**（例如 `stress`、`java`、`python` 等）
- 如果 `task` 值是 `pause` 或其他非预期值，说明加载了错误的任务段

**解决办法：** 检查 Agent 生成的配置文件内容（见 13.3 节的诊断命令）。

---

**原因 ③：存在残留的同名进程**

etmemd 使用 `type=name` 进行进程匹配时，会匹配节点上**所有**同名进程——不仅仅是目标 Pod 中的进程。如果之前的 Pod 没有被正确清理，可能存在残留的同名进程，导致 etmemd 扫描到错误的进程。

**排查方法：**

```bash
# 检查宿主机上所有同名进程
# 将 <process-name> 替换为您的应用进程名
pgrep -a <process-name>

# 示例：查找所有名为 stress 的进程
pgrep -a stress
```

如果发现有不属于当前 Pod 的残留进程，使用 `kill` 命令清理。

---

**原因 ④：配置文件权限不正确**

etmemd 要求配置文件的权限为 **600**（仅所有者可读写）或 **400**（仅所有者只读）。如果权限不正确，etmemd 会拒绝加载配置。

**排查方法：**

```bash
# 检查 Agent 生成的配置文件权限
ls -la /var/run/etmem/configs/
```

**预期输出：** 每个 `.conf` 文件的权限应为 `-rw-------`（600）或 `-r--------`（400）。

**解决办法：** 如果权限不正确，手动修正：

```bash
sudo chmod 600 /var/run/etmem/configs/*.conf
```

---

**原因 ⑤：在不兼容内核上使用了 `swap_flag=yes`**

如果您的内核是 **FusionOS 5.10** 系列，**必须**使用 `swap_flag=no`。设置为 `yes` 会导致 etmemd 行为异常。

> 当前所有内置 profile 均默认使用 `swap_flag=no`（见第 10.3 节），通常不需要手动修改。

---

#### 问题 2：如何判断是控制面问题还是数据面问题

当 `VmSwap` 为 0 时，首先需要确认问题出在哪个层级：

**✅ 控制面正常的标志（以下条件全部满足）：**

| 检查项 | 命令 | 预期结果 |
|--------|------|----------|
| EtmemPolicy 已创建 | `kubectl get etmempolicy -A` | 有对应 Pod 的 Policy 记录 |
| Agent 日志显示匹配 | `kubectl -n etmem-system logs ds/etmem-agent \| grep "Matched"` | 显示 "Matched pod xxx" |
| etmemd 项目已启动 | `sudo /usr/bin/etmem project show -s etmemd_socket` | 显示 `started=true` |

**✅ 数据面正常的标志（以下条件至少满足一个）：**

| 检查项 | 命令 | 预期结果 |
|--------|------|----------|
| 进程有 swap 活动 | `grep VmSwap /proc/<PID>/status` | `VmSwap` > 0 |
| 进程 RSS 减少 | `grep VmRSS /proc/<PID>/status` | 值比初始值小 |

**判断流程：**

```
控制面正常？
├── 否 → 排查控制面：检查 Operator/Agent 是否运行、label 是否正确、CRD 是否安装
└── 是 → 数据面正常？
    ├── 否 → 排查数据面：检查 sysmem_threshold、配置文件内容、etmemd 日志
    └── 是 → 一切正常！VmSwap > 0，内存回收已生效
```

**如果控制面正常但数据面不工作，重点检查：**
1. `sysmem_threshold` 是否足够高（原因 ① 中有详细说明）
2. 配置文件中的 `task` 值是否正确（原因 ②）
3. etmemd 自身日志是否有报错：`journalctl -u etmemd --tail=50`

### 13.3 关键诊断命令

以下是数据面排查中最常用的诊断命令，建议收藏备用：

```bash
# ========================================
# 进程级别诊断
# ========================================

# 检查目标进程的 swap 和内存状态
# 将 <PID> 替换为实际进程 ID（可从 Agent 日志中获取）
grep -E 'VmSwap|VmRSS' /proc/<PID>/status

# ========================================
# etmemd 级别诊断
# ========================================

# 查看 etmemd 当前加载的所有项目和任务
sudo /usr/bin/etmem project show -s etmemd_socket

# 查看 Agent 生成的配置文件内容
# <project-name> 通常是 Pod 名称或 Policy 名称
sudo cat /var/run/etmem/configs/<project-name>.conf

# 检查配置文件权限（必须是 600 或 400）
ls -la /var/run/etmem/configs/

# ========================================
# 系统级别诊断
# ========================================

# 检查系统可用内存比例（判断是否满足 sysmem_threshold）
grep -E 'MemTotal|MemAvailable' /proc/meminfo

# 检查 swap 使用情况
swapon -s
free -h

# ========================================
# 进程匹配诊断
# ========================================

# 检查宿主机上同名进程（排除残留进程干扰）
# 将 <process-name> 替换为目标应用进程名
pgrep -a <process-name>

# ========================================
# Kubernetes 日志诊断
# ========================================

# 检查 Agent 日志（关注 Matched/PID/config 相关信息）
kubectl -n etmem-system logs -l app=etmem-agent --tail=50

# 检查 Operator 日志（关注 Policy 创建/更新信息）
kubectl -n etmem-system logs deploy/etmem-operator --tail=50
```

> 💡 **提示：** 可以将上述命令保存为一个诊断脚本，在排查问题时一键执行。

### 13.4 本地控制面验证 ≠ 真实 swap 验证

> 这一节特别重要，请务必理解两者的区别，避免误判。

在前面的步骤中，我们通过以下操作验证了**控制面**：

| 操作 | 验证了什么 |
|------|-----------|
| 修改 Deployment/DaemonSet 并触发 Pod 重建 | Operator 能够识别新 Pod |
| 给 Pod 添加 `etmem.openeuler.io/enable=true` label | label/annotation 机制正常工作 |
| 检查 EtmemPolicy 自动创建 | Operator 能自动生成 EtmemPolicy |
| 检查 Agent 日志中的 "Matched pod" | Agent 能接收并处理 Policy |

**但这些操作 ≠ "真实的内存 swap 已经发生"。**

控制面验证证明的是"信号传递链路"通畅：

```
用户添加 label → Operator 创建 Policy → Agent 匹配 Pod → Agent 解析 PID → Agent 生成配置 → Agent 调用 etmemd
```

**真实 swap 验证**还需要额外证明：

```
etmemd 扫描冷页 → etmemd 将冷页写入 swap 设备 → 进程 VmSwap 增长 → 系统可用内存增加
```

**判断真实 swap 是否发生的唯一可靠方法：**

```bash
# 1. 确认 etmem 内核模块已加载
lsmod | grep etmem

# 2. 确认内存压力条件满足（或 sysmem_threshold 足够高）
awk '/MemTotal/{t=$2} /MemAvailable/{a=$2} END{printf "可用内存比例: %.1f%%\n", a/t*100}' /proc/meminfo

# 3. 观察目标进程的 VmSwap 变化（需要多次采样对比）
# 第一次采样
grep -E 'VmSwap|VmRSS' /proc/<PID>/status
# 等待一段时间后再次采样
sleep 60
grep -E 'VmSwap|VmRSS' /proc/<PID>/status
```

**如果 `VmSwap` 从 0 变为正数，且 `VmRSS` 相应减小，说明真实 swap 已经发生。**

> 📌 **总结：** 在没有 openEuler/FusionOS 内核和 etmem 内核模块的环境中，您可以完整验证控制面流程，但无法验证数据面。如果您需要验证真实的内存回收效果，请在支持 etmem 的目标环境中部署并按照本节指导进行验证。

## PID 作用域与跨 Pod 隔离

### 当前版本的进程匹配方式

从 v0.4.0-pid 起，etmem-operator Agent 使用 **`type=pid`** 而非 `type=name` 来定位目标进程：

| 版本 | 匹配方式 | 配置 | 作用域 |
|---|---|---|---|
| < v0.4.0-pid | `type=name` | `value=<进程名>` | 节点级（`pgrep -x`） |
| ≥ v0.4.0-pid | `type=pid` | `value=<PID>` | Pod 级（cgroup 限定） |

### PID 如何获取

Agent 通过 Pod 的 cgroup 路径精确获取进程 PID：

```
/sys/fs/cgroup/memory/kubepods/<qos>/<pod-uid>/<container-id>/cgroup.procs
```

该文件仅包含属于指定容器的进程 PID，天然实现了 Pod 级别的隔离。

### 在您的集群上验证隔离性

```bash
# === 1. 部署两个同名进程的 Pod（一个启用，一个不启用） ===
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: memhog-enabled
  namespace: etmem-test
  labels:
    etmem.openeuler.io/enable: "true"
  annotations:
    etmem.openeuler.io/process-name: "memhog"
    etmem.openeuler.io/profile: "default"
spec:
  containers:
  - name: memhog
    image: <your-memhog-image>
    command: ["memhog", "256M"]
---
apiVersion: v1
kind: Pod
metadata:
  name: memhog-noenable
  namespace: etmem-test
spec:
  containers:
  - name: memhog
    image: <your-memhog-image>
    command: ["memhog", "256M"]
EOF

# === 2. 等待换出生效 ===
sleep 60

# === 3. 检查 Agent 日志 ===
kubectl -n etmem-system logs ds/etmem-agent --tail=50 | grep -i "project\|pid\|type="
# 预期：项目名包含 -p{pid}，配置为 type=pid
# 预期：仅出现 memhog-enabled 的 PID

# === 4. 在节点上检查 VmSwap ===
# 获取两个 Pod 的 memhog PID，然后：
cat /proc/<enabled-pid>/status | grep VmSwap    # 应 >> 0
cat /proc/<noenable-pid>/status | grep VmSwap   # 应 = 0

# === 5. 清理 ===
kubectl delete pod -n etmem-test memhog-enabled memhog-noenable
```

### 常见问题排查

| 问题 | 可能原因 | 解决方法 |
|---|---|---|
| Agent 日志报 "PID not found" | 进程尚未在 cgroup 中就绪 | 等待进程完全启动后重新 reconcile；检查 `cgroup.procs` 是否非空 |
| Agent 日志仍显示 `type=name` | Agent 镜像版本过旧 | 确认镜像版本 ≥ v0.4.0-pid，重启 Agent DaemonSet |
| 未启用 Pod 的 VmSwap > 0 | 可能是系统 swap（非 etmem） | 检查 `/proc/<pid>/smaps` 确认换出来源；检查系统 swappiness 设置 |
| 项目名不含 `-p{pid}` 后缀 | Agent 逻辑未更新 | 检查 Agent 代码版本，确认使用了新的命名逻辑 |
| cgroup.procs 文件为空 | 容器尚未启动或已退出 | 检查 Pod 状态 (`kubectl get pod`)，确认容器处于 Running 状态 |

## 基于现有高负载工作负载验证 etmem

除了部署专用的测试 Pod，你也可以直接对集群中**已有的高负载工作负载**添加 etmem 标签来验证。这种方式更贴近生产场景，无需构建额外镜像。

### A. 识别现有工作负载

首先列出集群中所有 DaemonSet、Deployment、StatefulSet：

```bash
kubectl get ds,deploy,sts -A
```

查看内存消耗最多的 Pod，筛选验证候选对象：

```bash
kubectl top pods -A --sort-by=memory
```

**选择建议**：RSS 内存占用 >100MB 的工作负载是较好的验证候选对象。内存占用越大，冷页越多，etmem 换出效果越明显。

### B. 给现有工作负载添加 etmem 标签

etmem 的策略选择器为 `etmem.openeuler.io/enable: "true"`。需要将此标签添加到工作负载的 **Pod 模板** 中（而非直接添加到 Pod）。

> ⚠️ **注意**：修改 Pod 模板会触发滚动更新（Rolling Update），Pod 会被重建。请在合适的时间窗口进行操作。

**对 DaemonSet 添加标签：**

```bash
kubectl patch ds <名称> -n <命名空间> --type='json' \
  -p='[{"op":"add","path":"/spec/template/metadata/labels/etmem.openeuler.io~1enable","value":"true"}]'
```

**对 Deployment 添加标签：**

```bash
kubectl patch deploy <名称> -n <命名空间> --type='json' \
  -p='[{"op":"add","path":"/spec/template/metadata/labels/etmem.openeuler.io~1enable","value":"true"}]'
```

**对 StatefulSet 添加标签：**

```bash
kubectl patch sts <名称> -n <命名空间> --type='json' \
  -p='[{"op":"add","path":"/spec/template/metadata/labels/etmem.openeuler.io~1enable","value":"true"}]'
```

> 说明：路径中的 `~1` 是 JSON Patch 对 `/` 字符的转义，`etmem.openeuler.io~1enable` 实际对应标签 `etmem.openeuler.io/enable`。

Patch 成功后，Kubernetes 会触发滚动更新，新创建的 Pod 将自动携带 etmem 标签。

也可以使用自动化脚本：

```bash
bash scripts/patch-existing-workload-for-etmem.sh ds dbservice default
```

### C. 验证控制面

等待新 Pod 运行后（约 30-60 秒），检查 Agent 是否已匹配到目标 Pod。

**查看 Agent 日志：**

```bash
kubectl logs -n etmem-system -l app=etmem-agent --tail=100 | grep -i "matched\|project\|pid"
```

预期输出中应包含类似 `Matched pod <pod-name>` 的记录，以及为每个进程创建的项目信息。

**查看 NodeState：**

```bash
kubectl get etmemnodestate -o yaml
```

在 `status.tasks` 中可以看到：
- `podName`：被管理的 Pod 名称
- `projectName`：etmemd 项目名，格式通常为 `<namespace>-<pod>-<container>-p<PID>`
- `pids: N`：表示为该 Pod 发现了 N 个用户进程

如果 `pids` 数量与 Pod 内实际用户进程数一致，说明控制面已正确工作。

### D. 验证数据面

数据面验证需要在 Pod 所在节点上检查进程的内存换出情况。

**找到目标 Pod 的进程 PID：**

通过 Pod 的 cgroup 路径查找宿主机 PID：

```bash
# 获取 Pod UID
POD_UID=$(kubectl get pod <pod-name> -o jsonpath='{.metadata.uid}')
POD_UID_UNDER=$(echo "$POD_UID" | tr '-' '_')

# 在节点上查找 cgroup 下的进程（以 burstable QoS 为例）
CGROUP_BASE="/sys/fs/cgroup/memory/kubepods.slice/kubepods-burstable.slice/kubepods-burstable-pod${POD_UID_UNDER}.slice"
cat ${CGROUP_BASE}/cri-containerd-*.scope/cgroup.procs
```

**检查 VmSwap：**

```bash
grep VmSwap /proc/<PID>/status
```

- **VmSwap > 0**：表示数据面已生效，该进程的冷页已被 etmem 换出到 swap
- **VmSwap = 0**：数据面尚未生效

> 说明：etmem Agent 的 reconcile 周期约 30 秒，首次换出可能需要 1-2 个周期（30-60 秒）。如果使用了 `sysmem_threshold`，还需要系统可用内存低于阈值才会触发换出。

也可以使用自动化验证脚本：

```bash
sudo bash scripts/verify-existing-workload-etmem.sh <pod-name>
```

### E. 恢复工作负载

验证完成后，移除 etmem 标签以恢复工作负载到原始状态：

**对 DaemonSet 移除标签：**

```bash
kubectl patch ds <名称> -n <命名空间> --type='json' \
  -p='[{"op":"remove","path":"/spec/template/metadata/labels/etmem.openeuler.io~1enable"}]'
```

**对 Deployment / StatefulSet 类似操作**（将 `ds` 替换为 `deploy` 或 `sts`）。

移除标签后，Kubernetes 会再次触发滚动更新，新 Pod 不再携带 etmem 标签，Agent 会自动清理对应的 etmemd 项目和配置。

使用脚本移除：

```bash
bash scripts/patch-existing-workload-for-etmem.sh ds dbservice default --remove
```

### F. 常见问题

| 问题 | 原因 | 解决方法 |
|---|---|---|
| Pod 启动时间很长 | 数据库等大型应用初始化耗时较长 | 这是正常的，etmem 在进程稳定运行后才开始工作。耐心等待 Pod Ready |
| 熔断触发（Agent 停止管理） | Pod 反复重启超过阈值 | Agent 检测到 Pod 频繁重启后会主动退避，避免加剧不稳定。排查 Pod 本身的重启原因 |
| VmSwap 始终为 0 | sysmem_threshold 未达到触发条件 | 检查当前 profile 的 `sysmem_threshold` 值。大内存机器（如 1TB）需要使用 aggressive（95）或 extreme（99）profile 才能触发 |
| Agent 未匹配到 Pod | 标签未正确添加到 Pod 模板 | 确认使用了 `--type='json'` 的 patch 命令，检查新 Pod 的标签：`kubectl get pod <name> --show-labels` |
| 标签添加后 Pod 未重建 | 直接在 Pod 上添加标签而非 Pod 模板 | 必须通过 `kubectl patch ds/deploy/sts` 修改 Pod 模板，而非 `kubectl label pod` |

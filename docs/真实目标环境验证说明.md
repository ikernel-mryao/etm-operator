# 真实目标环境验证说明

> 本文档说明**为什么本地/最小化 K8s 验证无法替代真实目标环境验证**，
> 以及在真实环境中如何验证 etmem-operator 的完整功能。

---

## 1. etmem-operator 的两层架构

etmem-operator 的功能分为**控制面**和**数据面**两个层次。理解这两层的区别是理解验证边界的关键。

### 1.1 控制面（Control Plane）

| 组件 | 职责 |
|------|------|
| **Operator** | 监听带有 `etmem.openeuler.io/enable=true` 标签的 Pod，自动创建/更新/删除 EtmemPolicy 资源 |
| **Agent** | 监听 EtmemPolicy 资源，在本节点上匹配目标 Pod，通过 cgroup v1 路径解析容器 PID，生成 etmemd 配置文件 |

**控制面做的事情：**
- Pod 打标签 → Operator 创建 EtmemPolicy
- Pod 删除/移除标签 → Operator 删除 EtmemPolicy
- Agent 发现匹配的 Policy → 解析 PID → 生成配置 → 调用 etmem CLI

**控制面可以在哪里验证：**
- ✅ 任意 Kubernetes 集群（Kind、Minikube、kubeadm 单节点、云托管集群等）
- ✅ 任意 Linux 发行版（Ubuntu、CentOS、Debian 等）
- ✅ 不需要 etmem 内核模块
- ✅ 不需要 etmemd 守护进程
- ✅ 不需要 swap 设备

### 1.2 数据面（Data Plane）

| 组件 | 职责 |
|------|------|
| **etmemd** | 接收 Agent 的配置后，使用内核模块扫描进程的内存页访问频率，识别「冷页」 |
| **内核模块** | `etmem_scan` 提供内存页扫描能力，`etmem_swap` 提供页面交换能力 |
| **swap 设备** | 冷页的最终目的地 — 从物理内存交换到 swap 设备，释放物理内存 |

**数据面做的事情：**
- etmemd 通过内核模块扫描目标进程的内存页
- 识别一段时间内未被访问的「冷页」
- 将冷页交换到 swap 设备
- 物理内存被释放，系统可用内存增长

**数据面必须在真实目标环境中验证：**
- ❌ 无法在普通 Ubuntu/CentOS 上验证（没有 etmem 内核模块）
- ❌ 无法在没有 etmemd 的机器上验证
- ❌ 无法在没有 swap 的机器上验证
- ✅ 仅在 openEuler/FusionOS 内核 + etmemd + swap 环境中有效

## 2. 为什么本地验证不能替代真实环境

### 2.1 控制面验证的价值与局限

在本地 K8s 集群上，您可以验证：

| ✅ 可以验证的内容 | 说明 |
|-------------------|------|
| CRD 正确安装 | `kubectl get crd \| grep etmem` |
| Operator 正常运行 | Pod 状态为 Running，日志无报错 |
| Agent 正常运行 | Pod 状态为 Running，日志无报错 |
| Label 触发 Policy 创建 | 给 Pod 打标签后自动出现 EtmemPolicy |
| Policy 内容正确 | profile 参数、nodeName、podName 等字段正确 |
| 移除 Label 后 Policy 自动删除 | 去掉标签后 EtmemPolicy 自动消失 |
| Agent 匹配 Pod 并解析 PID | 日志中出现 "Matched pod" 和 PID 信息 |
| Profile 切换生效 | 修改 annotation 后 Policy 内容更新 |

但是：

| ❌ 无法验证的内容 | 原因 |
|-------------------|------|
| 冷页是否被正确扫描 | 需要 etmem_scan 内核模块 |
| 冷页是否被交换到 swap | 需要 etmem_swap 内核模块 + swap 设备 |
| 可用内存是否增长 | 需要真实的 swap 操作 |
| 进程 VmSwap 是否增加 | 需要真实的 swap 操作 |
| etmemd 配置是否正确执行 | 需要 etmemd 守护进程 |
| 不同 profile 的实际效果差异 | 需要真实的内存压力和 swap 操作 |

### 2.2 形象比喻

> 控制面验证就像「检查导航系统是否能正确规划路线」——地图加载了、路线计算了、指令发出了。
>
> 数据面验证就像「检查汽车是否真的能沿着路线行驶」——需要引擎（etmemd）、燃料（内核模块）、道路（swap 设备）。
>
> 两者都不可或缺，但只有数据面验证才能证明系统真正在工作。

## 3. 真实目标环境的要求

### 3.1 操作系统

| 要求 | 说明 |
|------|------|
| **openEuler 22.03 LTS SP1+** 或 **FusionOS 23** | etmem 内核模块仅存在于 openEuler 系列内核中 |

```bash
# 验证操作系统
cat /etc/os-release
uname -r  # 应包含 "oe" 字样
```

### 3.2 内核模块

```bash
# 加载 etmem 内核模块
sudo modprobe etmem_scan etmem_swap

# 验证加载成功
lsmod | grep etmem
```

**预期输出：**
```
etmem_swap            xxxxx  0
etmem_scan            xxxxx  0
```

> 如果 `modprobe` 失败，说明内核不包含 etmem 模块。请确认：
> 1. 操作系统为 openEuler/FusionOS
> 2. 内核版本包含 etmem 支持
> 3. 内核模块未被禁用

### 3.3 etmemd 守护进程

```bash
# 安装 etmemd（openEuler）
sudo yum install etmem

# 启动 etmemd
sudo etmemd -s etmemd_socket -l 0 &

# 验证运行状态
ps aux | grep etmemd
sudo /usr/bin/etmem project show -s etmemd_socket
```

> **关键：** etmemd 必须在 Agent Pod 启动**之前**运行。Agent 启动时会尝试连接名为 `etmemd_socket` 的**抽象 Unix socket**。
> 该 socket **不是文件系统路径**，所以不要用 `ls /var/run/etmemd_socket` 判断。

### 3.4 swap 设备

```bash
# 检查 swap 状态
swapon -s
cat /proc/swaps

# 如果无 swap，配置 swap 文件
sudo fallocate -l 8G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile

# 验证
free -h
```

### 3.5 内核参数

```bash
# 查看 swappiness（建议 ≥60）
cat /proc/sys/vm/swappiness

# 如需调整
sudo sysctl vm.swappiness=60

# openEuler 特有：检查 swap_extension
cat /proc/sys/vm/swap_extension 2>/dev/null
```

### 3.6 cgroup v1

```bash
# 确认 cgroup v1
stat -fc %T /sys/fs/cgroup/
# 应输出 tmpfs（v1），而非 cgroup2fs（v2）

# 确认 memory 子系统存在
ls /sys/fs/cgroup/memory/kubepods/
```

### 3.7 环境要求检查清单

```bash
echo "=== 操作系统 ==="
cat /etc/os-release | grep -E 'NAME|VERSION'
uname -r

echo "=== 内核模块 ==="
lsmod | grep etmem

echo "=== etmemd ==="
ps aux | grep etmemd | grep -v grep

echo "=== swap ==="
swapon -s
free -h | grep -i swap

echo "=== cgroup 版本 ==="
stat -fc %T /sys/fs/cgroup/

echo "=== swappiness ==="
cat /proc/sys/vm/swappiness
```

## 4. 数据面验证方法

当真实目标环境准备就绪后，按以下步骤验证 etmem 的**实际内存回收效果**。

### 4.1 部署并启用 etmem

```bash
# 1. 部署 etmem-operator（参考快速上手手册）
helm install etmem-operator ./deploy/helm -n etmem-system --create-namespace

# 2. 部署工作负载
kubectl apply -f your-workload.yaml

# 3. 启用 etmem
kubectl label pod <pod-name> -n <namespace> etmem.openeuler.io/enable=true
```

> 请在 etmem-operator 仓库根目录执行该命令；如果当前目录不是仓库根目录，请使用完整路径。

### 4.2 观察 Agent 日志

```bash
kubectl -n etmem-system logs ds/etmem-agent --tail=50 -f
```

**关注以下关键日志：**
```
Matched pod <pod-name> in namespace <namespace>
Found container PIDs: [12345, 67890]
Generated etmemd config for pod <pod-name>
Executing etmem add project ...
```

> 如果看到 "Matched pod" 和 PID 列表，说明 Agent 已成功发现并解析了目标容器的进程。

### 4.3 推荐使用自动化验证脚本

在真实目标环境中，推荐直接在 **目标 Pod 所在节点** 上以 root 身份运行：

```bash
sudo bash scripts/verify-existing-workload-etmem.sh <pod-name> [命名空间]
```

当前脚本会同时完成以下检查：

1. 只筛选**当前目标 Pod**的 Agent 日志，不再混入旧 Pod 记录
2. 检查 EtmemNodeState 中该 Pod 的任务记录
3. 展示目标节点的 `socketReachable` / `etmemdReady` / `totalManagedPods`
4. 自动兼容 **cgroup v1** 下的：
   - systemd 驱动路径
   - cgroupfs 驱动路径
   - Guaranteed / Burstable / BestEffort 三种 QoS
5. 输出进程级 `VmRSS` / `VmSwap`
6. 补充输出主机级 `free -m` 和 `MemAvailable`

> 该脚本的前提是：**当前节点使用 cgroup v1**，且您是在目标 Pod 所在节点上执行。

### 4.4 检查进程 VmSwap

**这是验证数据面最直接的指标。**

```bash
# 获取容器进程 PID（从 Agent 日志获取）
PID=12345

# 查看内存详情
cat /proc/$PID/status | grep -E 'VmRSS|VmSwap'
```

**未启用 etmem 时：**
```
VmRSS:    256000 kB
VmSwap:        0 kB
```

**etmem 生效后（一段时间后）：**
```
VmRSS:    180000 kB
VmSwap:    76000 kB
```

> `VmSwap > 0` 说明冷页已被交换到 swap，物理内存（VmRSS）相应减少。

### 4.5 检查系统内存变化

```bash
# 查看整体内存状况
free -h

# 详细 swap 信息
cat /proc/meminfo | grep -E 'MemTotal|MemFree|MemAvailable|SwapTotal|SwapFree|SwapCached'
```

**观察重点：**

| 指标 | etmem 生效前 | etmem 生效后 | 说明 |
|------|-------------|-------------|------|
| `MemAvailable` | 较低 | **增长** | 冷页被 swap 出去后释放的物理内存 |
| `SwapFree` | 接近 SwapTotal | **减少** | swap 设备中写入了冷页数据 |
| `SwapCached` | 接近 0 | **可能增长** | 被换出但可能被再次访问的页面缓存 |

### 4.6 检查 NodeState 健康字段

```bash
kubectl get etmemnodestate -o yaml
```

重点观察当前节点的：

- `status.socketReachable: true`
- `status.etmemdReady: true`
- `status.metrics.totalManagedPods`
- `status.tasks[]`

判定方式：

- `socketReachable=true`：Agent 当前可通过 `etmem project show -s etmemd_socket` 连到 etmemd
- `etmemdReady=true`：Agent 当前认为 etmemd 通路可用
- `tasks[]` 中出现目标 Pod：说明控制面已经把目标 Pod 纳入管理

### 4.7 理解 swapcache → swapfile → available 链路

etmem 的内存回收链路如下：

```
进程内存页（物理内存）
    ↓ etmem_scan 识别为冷页
    ↓ etmem_swap 标记为可交换
    ↓
SwapCache（交换缓存）
    ↓ 内核 swap 子系统写入
    ↓
Swap 设备（磁盘/文件）
    ↓
物理内存页被释放
    ↓
MemAvailable 增长（系统可用内存增加）
```

**详细过程：**
1. **etmem_scan** 内核模块扫描目标进程的内存页，按访问频率标记冷页
2. **etmem_swap** 内核模块将冷页移入 swap cache
3. 内核的 swap 子系统将 swap cache 中的页面写入 swap 设备（磁盘或文件）
4. 原来占用的物理内存页被释放，操作系统将其标记为可用
5. `MemAvailable` 增长，系统可以利用这些内存运行更多工作负载

### 4.8 检查 etmemd 状态

```bash
# etmemd 进程状态
ps aux | grep etmemd

# 如果有 systemd 管理
journalctl -u etmemd --tail=30

# 直接检查 etmem CLI 能否通过抽象 socket 访问 etmemd
sudo /usr/bin/etmem project show -s etmemd_socket
```

## 5. Swap 不生效的诊断清单

> 💡 **真实发现：** 在我们的验证过程中，遇到过 etmemd 能成功扫描进程冷页，但始终无法将其 swap 出去的情况。
> 这通常是**宿主机配置问题**，而非 etmem-operator 的 bug。

### 5.1 逐项排查

按以下顺序逐项检查：

#### ① etmemd 是否在运行？

```bash
ps aux | grep etmemd | grep -v grep
```
如果无输出：需要启动 etmemd（`etmemd -s etmemd_socket -l 0 &`）。

#### ② 内核模块是否加载？

```bash
lsmod | grep etmem
```
应看到 `etmem_scan` 和 `etmem_swap`。如果缺少，执行 `modprobe etmem_scan etmem_swap`。

#### ③ swap 是否启用？

```bash
swapon -s
```
应有 swap 设备列出。如果无输出，需要配置 swap。

#### ④ swap 空间是否充足？

```bash
free -h | grep -i swap
```
`Swap free` 应大于 0。如果 swap 已满，etmemd 无法交换更多页面。

#### ⑤ swappiness 是否足够？

```bash
cat /proc/sys/vm/swappiness
```
如果值过低（如 0 或 10），内核会非常不愿意使用 swap。建议设为 60：
```bash
sudo sysctl vm.swappiness=60
```

#### ⑥ 系统内存是否达到 profile 阈值？（**最常见根因**）

> ⚠️ **`sysmem_threshold=N` 的真实含义**：当 MemAvailable < N% 时才触发 swap。
> 也就是说，阈值越高越容易触发，不是越低越容易触发。

| Profile | sysmem_threshold | 含义 | 1TB/82%可用 机器 |
|---------|-----------------|------|-----------------|
| conservative | 85 | 可用 < 85% 才触发 | ✅ 会触发 |
| moderate | 90 | 可用 < 90% 才触发 | ✅ 会触发 |
| aggressive | 95 | 可用 < 95% 才触发 | ✅ 会触发 |
| extreme | 99 | 可用 < 99% 才触发 | ✅ 会触发 |

**验证当前是否被阈值挡住：**
```bash
# 计算当前可用内存百分比
MEM_TOTAL=$(grep MemTotal /proc/meminfo | awk '{print $2}')
MEM_AVAIL=$(grep MemAvailable /proc/meminfo | awk '{print $2}')
echo "scale=1; $MEM_AVAIL * 100 / $MEM_TOTAL" | bc
# 如果结果 > sysmem_threshold，etmem 不会触发
# 解决：使用更高 threshold 的 profile（如 aggressive=95 或 extreme=99）
```

#### ⑦ Agent 是否成功执行了 etmem CLI？

```bash
kubectl -n etmem-system logs ds/etmem-agent --tail=100
```
查找以下关键信息：
- `Executing etmem add project` — 是否成功添加了项目
- `Error` / `Failed` — 是否有报错

#### ⑧ etmemd 配置文件是否正确生成？

```bash
# 进入 Agent Pod 查看生成的配置
kubectl -n etmem-system exec -it ds/etmem-agent -- ls /etc/etmem/
kubectl -n etmem-system exec -it ds/etmem-agent -- cat /etc/etmem/<config-file>
```

#### ⑨ 等待时间是否足够？

etmem 不会立即交换内存。根据 profile 的 `loop`、`interval`、`sleep` 参数，可能需要多个扫描周期才能识别冷页并执行 swap。

- `conservative` profile：loop=3, interval=5s, sleep=3s → 一个完整周期约 18 秒
- `aggressive` profile：loop=1, interval=1s, sleep=1s → 一个完整周期约 2 秒

建议至少等待 **2-3 分钟**后再检查 VmSwap。

### 5.2 常见失败模式总结

| 症状 | 可能原因 | 解决方案 |
|------|---------|---------|
| Agent 日志无 "Matched pod" | EtmemPolicy 未创建或 Agent 未在目标节点 | 检查 Operator 日志和 Policy 状态 |
| Agent 报 PID 为空 | cgroup v2 环境或 cgroup 路径不匹配 | 确认 cgroup v1，并按 QoS + 驱动类型检查 kubepods 路径 |
| 验证脚本混入旧 Pod 日志 | grep 条件过宽 | 使用当前版本脚本；它只筛选 JSON 字段中 `"pod": "<目标 Pod>"` 的日志 |
| NodeState 健康字段不可信 | 旧版本未动态写入 `socketReachable/etmemdReady` | 升级到当前版本 Agent，字段已按每轮 socket 探测实时更新 |
| VmSwap 始终为 0 | **sysmem_threshold < 当前可用%**（最常见） | 使用 aggressive(95) 或 extreme(99) profile |
| VmSwap 始终为 0 | **配置有多个 [task] 段** | 升级 Agent 到 v0.2.0+（自动过滤 pause 容器） |
| VmSwap 始终为 0 | **宿主机有同名进程干扰** | 清理残留进程，确保 name 匹配唯一 |
| VmSwap 始终为 0 | **swap_flag=yes**（FusionOS 5.10） | 确保 profile 使用 swap_flag=no |
| VmSwap 始终为 0 | swappiness=0 | `sysctl vm.swappiness=60` |
| VmSwap 始终为 0 | etmemd 未运行 | 启动 etmemd |
| VmSwap 始终为 0 | 内核模块未加载 | `modprobe etmem_scan etmem_swap` |
| VmSwap 始终为 0 | swap 未启用 | 配置并启用 swap |
| VmSwap 始终为 0 | 配置文件权限非 600/400 | `chmod 600 /var/run/etmem/configs/*.conf` |
| swap 使用量不增长 | swap 设备已满 | 扩大 swap 或清理现有 swap |

## 6. 验证成功的判定标准

当您在真实目标环境中完成部署和启用后，满足以下**全部条件**即可判定验证成功：

### 6.1 控制面验证通过

- [x] `kubectl get crd | grep etmem` 显示两个 CRD
- [x] `kubectl -n etmem-system get pods` 所有 Pod 为 Running
- [x] 给 Pod 打标签后 `kubectl get etmempolicy -A` 有对应 Policy
- [x] Agent 日志中出现 "Matched pod" 和 PID 列表
- [x] `kubectl get etmemnodestate -o yaml` 中当前节点 `socketReachable=true` 且 `etmemdReady=true`
- [x] 移除标签后 Policy 自动删除

### 6.2 数据面验证通过

- [x] `cat /proc/<PID>/status | grep VmSwap` 显示 VmSwap > 0
- [x] `free -h` 显示 Swap used > 0（对比启用前后）
- [x] `cat /proc/meminfo | grep MemAvailable` 显示可用内存增长
- [x] Agent 日志显示成功执行 etmem CLI 命令

## 7. 总结

| 维度 | 控制面 | 数据面 |
|------|--------|--------|
| **验证内容** | Operator 创建 Policy、Agent 解析 PID、配置生成 | 冷页扫描、swap 执行、内存释放 |
| **环境要求** | 任意 K8s 集群 | openEuler/FusionOS + etmemd + swap + cgroup v1 |
| **验证方式** | kubectl 查看资源和日志、NodeState 健康字段 | /proc/PID/status、free -h、/proc/meminfo |
| **验证结果** | 证明 Operator 逻辑正确 | 证明系统真正回收了内存 |
| **不足之处** | 无法证明内存被实际回收 | 需要特定环境，不易复现 |

**核心结论：** 控制面验证证明的是「指令正确发出了」，数据面验证证明的是「动作真正执行了」。两者缺一不可，但数据面验证**必须在真实目标环境中完成**。

> 📖 部署和操作步骤请参阅 [其他机器快速上手手册](./其他机器快速上手手册.md)。

> 以下内容基于 2026-04-15/16 在 FusionOS 23 真实集群上的 E2E 验证结果。

### 8.1 验证环境

| 项目 | 值 |
|------|-----|
| 操作系统 | FusionOS 23（openEuler 衍生） |
| 内核 | 5.10.0-278.0.0.181.u183.fos23.x86_64 |
| 物理内存 | 1TB |
| Swap | 16GB |
| cgroup | v1 |
| etmemd | 手动编译安装，/usr/bin/etmemd |

| K8s | v1.26.15 + containerd 1.6.22 |

### 8.2 E2E 成功结果

| 指标 | 操作前 | 操作后 | 说明 |
|------|--------|--------|------|
| VmSwap | 0 kB | 272,156 kB (265MB) | 冷页全部换出 |
| VmRSS | 270,792 kB | 1,540 kB | 物理内存几乎完全释放 |
| 触发时间 | — | ~40 秒 | 从 Agent 配置下发到 swap 完成 |

完整流水线：
```
Pod(label=enable:true, profile=aggressive)
  → Operator 自动创建 EtmemPolicy
    → Agent 匹配 Pod、解析 PID、生成单 [task] 配置
      → etmemd 加载配置、扫描冷页、执行 swap
        → VmSwap 0→272MB ✅
```

### 8.3 发现并修复的三个根因

在本次验证过程中，发现了三个导致 VmSwap 始终为 0 的根因，全部已修复：

#### 根因 1：sysmem_threshold 语义在大内存机器上失效

- **问题**：`sysmem_threshold=N` 的含义是"当 MemAvailable < N% 时才触发 swap"
- **现象**：1TB 机器有 82% 可用内存，旧 profile 阈值 30/50/70 永远不会触发
- **修复**：调整为 conservative=85, moderate=90, aggressive=95, extreme=99
- **关键认知**：在大内存服务器上，这个值必须大于当前可用内存百分比

#### 根因 2：etmemd INI 解析器只加载最后一个 [task]

- **问题**：etmemd 的配置解析器遇到多个 `[task]` 段时，只保留最后一个
- **现象**：Agent 为每个容器生成一个 [task]，pause 容器排在最后 → etmemd 只对 pause 执行 swap
- **修复**：Agent 过滤 pause/sandbox 基础设施容器，确保每个配置只有单个 [task]
- **关键认知**：这是 etmemd 本身的限制，未在任何文档中记载

#### 根因 3：宿主机残留同名进程干扰

- **问题**：`type=name, value=memhog` 匹配宿主机上所有同名进程
- **现象**：旧 Pod 的孤儿进程或其他 Deployment 的容器干扰匹配
- **修复**：验证前清理残留进程；后续考虑改用 `type=pid` 精确匹配
- **关键认知**：name 匹配是全局的，不限于目标 Pod 的 cgroup

### 8.4 其他发现

| 发现 | 详情 |
|------|------|
| **配置文件权限** | etmemd 要求 600 或 400，644 被拒绝且无有用错误信息 |
| **swap_flag=yes 失效** | FusionOS 5.10 内核下，swap_flag=yes 导致 ioctl VMA_SCAN_ADD_FLAGS 返回 0 字节 |
| **抽象 Unix socket** | etmemd 使用抽象 socket（无文件系统条目），Agent 需 hostNetwork:true |
| **swapcache_high/low_wmark** | 当前由 profile 硬编码（aggressive: 5/3），正式版本应开放配置 |

### 8.5 在新环境复现数据面的最小条件

**9 个缺一不可的条件：**

1. ✅ etmem 内核模块（etmem_scan + etmem_swap）已加载
2. ✅ etmemd 守护进程运行中（`/usr/bin/etmemd -s etmemd_socket`）
3. ✅ swap 设备已启用且有空闲空间
4. ✅ `vm.swappiness >= 60`
5. ✅ `sysmem_threshold` > 当前 MemAvailable 百分比
6. ✅ 配置文件只有单个 `[task]` 段（Agent v0.2.0 已自动保证）
7. ✅ 宿主机上无同名进程干扰
8. ✅ `swap_flag=no`（FusionOS 5.10 必须）
9. ✅ 配置文件权限为 600 或 400

## 多进程 swap 的观测方式

> 从 Agent v0.3.0-mp 开始，etmem-operator 支持 Pod 内多进程管理。本节说明如何在真实环境中验证多进程 swap 是否正常工作。

### 1. 确认多进程项目已创建

```bash
sudo /usr/bin/etmem project show -s etmemd_socket
```

**观察要点**：

- 如果目标 Pod 有 N 个用户进程，应看到 N 个（或 N+1 个，含 sh wrapper）独立项目
- 项目命名格式为 `{namespace}-{podName}-{processName}`
- 例如 Pod `multiproc-test` 有 memeat1 和 memeat2 两个进程，应看到：

```
default-multiproc-test-memeat1
default-multiproc-test-memeat2
default-multiproc-test-sh        # 容器入口 wrapper，无害
```

### 2. 逐进程检查 VmSwap

**这是区分"只有一个进程被换出"和"所有进程都被换出"的关键步骤。**

```bash
# 获取 Pod 中各进程的 PID（方法 1：从 Agent 日志获取）
kubectl -n etmem-system logs ds/etmem-agent --tail=100 | grep "Found container PIDs"

# 获取 Pod 中各进程的 PID（方法 2：从 cgroup 获取）
cat /sys/fs/cgroup/memory/kubepods/<qos>/<pod-uid>/*/tasks

# 逐进程检查
for pid in <PID1> <PID2> <PID3>; do
  echo "=== PID=$pid ==="
  grep -E 'Name|VmRSS|VmSwap' /proc/$pid/status
done
```

**正常多进程输出示例**（所有进程都在被管理）：

```
=== PID=2338929 ===
Name:   memeat1
VmRSS:      1788 kB
VmSwap:   152424 kB     # ✅ > 0，冷页已换出

=== PID=2338930 ===
Name:   memeat2
VmRSS:      1784 kB
VmSwap:   152424 kB     # ✅ > 0，冷页已换出
```

**异常输出示例**（只有一个进程被换出）：

```
=== PID=2338929 ===
Name:   memeat1
VmRSS:      1788 kB
VmSwap:   152424 kB     # ✅ 已换出

=== PID=2338930 ===
Name:   memeat2
VmRSS:    153600 kB
VmSwap:        0 kB     # ❌ 未换出！该进程未被管理
```

### 3. 如何区分问题原因

| 现象 | 可能原因 | 排查方法 |
|------|---------|---------|
| 只看到一个项目（多进程 Pod） | Agent 使用旧版单进程代码（`processes[:1]` 截断） | 确认 Agent 版本 ≥ v0.3.0-mp；检查 Agent 镜像是否更新 |
| 看到多个项目但只有一个进程 VmSwap > 0 | 其他进程可能内存占用极小（如 sh wrapper），无冷页可换 | 检查 VmRSS 初始值——如果本身就很小（< 100 kB），则 VmSwap=0 是正常的 |
| 看到多个项目但所有 VmSwap = 0 | sysmem_threshold 未达到、swap 未启用等通用原因 | 参考 [第 5 节 Swap 不生效的诊断清单](#5-swap-不生效的诊断清单) 逐项排查 |
| 项目名中不含进程名（如 `default-pod-test`） | Agent 使用旧版命名（v0.2.0 按 Pod 命名而非按进程） | 升级 Agent 到 v0.3.0-mp |

### 4. 快速诊断脚本

```bash
#!/bin/bash
# 多进程 swap 快速诊断
echo "=== etmemd 项目列表 ==="
sudo /usr/bin/etmem project show -s etmemd_socket

echo ""
echo "=== 配置文件 [task] 段计数 ==="
for f in /var/run/etmem/configs/*.conf; do
  task_count=$(grep -c '^\[task\]' "$f")
  echo "  $f → [task] 数量: $task_count"
  if [ "$task_count" -gt 1 ]; then
    echo "    ⚠️ 警告：存在多 [task] 段，etmemd 只会加载最后一个！"
  fi
done

echo ""
echo "=== 请输入目标 Pod 的各进程 PID（空格分隔）==="
read -r pids
for pid in $pids; do
  if [ -f "/proc/$pid/status" ]; then
    echo "--- PID=$pid ---"
    grep -E 'Name|VmRSS|VmSwap' /proc/$pid/status
  else
    echo "--- PID=$pid: 进程不存在 ---"
  fi
done
```

### 5. 关键警告

> ⚠️ **如果您在一个多进程 Pod 上只看到一个 etmemd 项目**，说明 Agent 可能仍在使用旧版单进程逻辑。请检查：
>
> 1. Agent 镜像版本是否为 v0.3.0-mp 或更新版本
> 2. Agent 代码中是否仍有 `processes[:1]` 的截断逻辑
> 3. Agent Pod 是否已重启以加载新代码
>
> 升级后应能看到多个项目，每个进程的 VmSwap 均 > 0。

## PID 作用域隔离验证（生产环境）

### 为什么 type=pid 对多租户集群至关重要

在生产多租户集群中，同一节点上通常运行着多个租户的 Pod，其中可能存在同名进程（如多个 Java 应用、多个 nginx 实例等）。旧版 `type=name` 使用 `pgrep -x <name>` 匹配进程，会命中节点上所有同名进程，存在以下风险：

- **跨租户干扰**：租户 A 启用 etmem 后，可能导致租户 B 的同名进程被换出
- **不可预测行为**：etmemd 的 pgrep 只取第一个匹配结果，实际换出的进程取决于 PID 排序
- **安全隐患**：未经授权的内存操作可能影响其他租户的服务质量

`type=pid` 从 Pod 的 cgroup 路径获取 PID，天然将作用域限定在 Pod 内，彻底消除了上述风险。

### 如何在生产环境验证无跨 Pod 干扰

#### 步骤 1：确认 Agent 版本

```bash
# 确认 Agent 使用 type=pid
kubectl -n etmem-system logs ds/etmem-agent --tail=100 | grep "type="
# 预期输出应包含 type=pid，不包含 type=name
```

#### 步骤 2：选择验证目标

选取一个已启用 etmem 的 Pod，并确认同一节点上存在其他运行同名进程的 Pod：

```bash
# 查找启用了 etmem 的 Pod
kubectl get pods -A -l etmem.openeuler.io/enable=true -o wide

# 查看同一节点上的其他 Pod
NODE=<目标节点名>
kubectl get pods -A --field-selector spec.nodeName=${NODE} -o wide
```

#### 步骤 3：对比各进程的 VmSwap

```bash
#!/bin/bash
# 生产环境跨 Pod 隔离验证脚本
NODE="<目标节点>"
PROCESS_NAME="<进程名>"  # 如 java, nginx, memhog

echo "=== 节点 ${NODE} 上所有 ${PROCESS_NAME} 进程 ==="
# 在节点上执行（通过 SSH 或 Agent Pod）
pgrep -xa "${PROCESS_NAME}" | while read pid cmd; do
  # 获取进程所属的 cgroup（定位到 Pod）
  CGROUP=$(cat /proc/${pid}/cgroup 2>/dev/null | grep memory | cut -d: -f3)
  VMSWAP=$(grep VmSwap /proc/${pid}/status 2>/dev/null | awk '{print $2, $3}')
  VMRSS=$(grep VmRSS /proc/${pid}/status 2>/dev/null | awk '{print $2, $3}')
  echo "PID=${pid} | VmSwap=${VMSWAP} | VmRSS=${VMRSS} | cgroup=${CGROUP}"
done
```

#### 步骤 4：验证结论

预期结果：

| 进程 | 所属 Pod 启用 etmem | VmSwap | 判定 |
|---|---|---|---|
| 启用 Pod 中的进程 | ✅ 是 | > 0 | ✅ 正确换出 |
| 其他 Pod 中的同名进程 | ❌ 否 | = 0 | ✅ 未被干扰 |

如果发现未启用 Pod 的进程 VmSwap > 0，需要排查：

1. 是否为系统级 swap（非 etmem 触发）：检查 `swappiness` 和系统内存压力
2. Agent 是否仍使用旧版 `type=name`：检查 Agent 日志和镜像版本
3. 是否有其他 etmem 策略意外覆盖到该进程

### 推荐的验证步骤（同名进程场景）

针对生产环境中需要严格验证隔离性的场景，建议按以下步骤操作：

1. **部署测试 Pod**：在目标节点上部署两个运行相同进程名的 Pod，仅对一个启用 etmem
2. **等待 reconcile**：等待 Agent 完成配置下发（约 30-60 秒）
3. **检查 Agent 日志**：确认项目名包含 `-p{pid}` 后缀，且仅包含启用 Pod 的 PID
4. **对比 VmSwap**：启用 Pod 的 VmSwap > 0，未启用 Pod 的 VmSwap = 0
5. **检查 etmemd 配置文件**：确认 `/var/run/etmem/` 下的配置文件使用 `type=pid`

```bash
# 检查 etmemd 配置文件
ls /var/run/etmem/
cat /var/run/etmem/<project-name>.conf | grep -E "type|value"
# 预期：type=pid, value=<具体PID>
```

6. **清理测试 Pod**：验证完成后删除测试 Pod，确认 Agent 清理了对应的 etmemd 项目

## 优先复用现有高负载工作负载

在真实目标环境中进行验证时，**推荐优先使用集群中已有的高负载工作负载**，而不是从零创建测试 Pod。

### 优势

- **更贴近生产场景**：真实工作负载的内存访问模式、进程数量、cgroup 结构都更具代表性
- **无需构建镜像**：不需要额外准备 memhog 等测试镜像，降低验证门槛
- **操作简便**：只需通过 `kubectl patch` 添加标签，验证后移除标签即可恢复

### 推荐流程

1. **筛选候选工作负载**：选择 RSS >100MB 的 DaemonSet、Deployment 或 StatefulSet
2. **添加 etmem 标签**：通过 `kubectl patch` 修改 Pod 模板，触发滚动更新
3. **验证控制面和数据面**：检查 Agent 日志、NodeState 和进程 VmSwap
4. **移除标签恢复原状**：验证完成后移除标签，Pod 回滚到无 etmem 管理的状态

### 自动化工具

```bash
# 添加标签（触发滚动更新）
bash scripts/patch-existing-workload-for-etmem.sh ds <名称> [命名空间]

# 验证 etmem 管理效果
sudo bash scripts/verify-existing-workload-etmem.sh <pod-name>

# 移除标签（恢复原状）
bash scripts/patch-existing-workload-for-etmem.sh ds <名称> [命名空间] --remove
```
